# 章节转框架

使用此参考将章节顺序转化为真实知识框架。

## 原则

课程顺序常为演示优化，而非为理解优化。

你的工作是恢复依赖图：

- 什么必须先被理解
- 什么主要作为应用存在
- 什么可以推迟

## 转换过程

### 第1步：标记每个章节

将每个章节分类为：

- 方向导引
- 核心概念
- 工作流程
- 应用
- 高级扩展
- 回顾总结

方向导引和回顾总结很少属于知识主干。

### 第2步：提取章节 IO

对每个章节问：

- 输入是什么
- 输出是什么
- 依赖什么
- 为下一个解锁什么

这揭示章节是基础性的还是派生的。

### 第3步：合并等效章节

不同章节可能实际上教同一底层内容。

示例：

- 安装 + 工具 + 环境信心
- 上下文管理 + 代码理解
- 测试 + 调试 + 反馈循环
- 实现 + 重构 + 验证

当机制共享时合并。

### 第4步：分离主干与枝叶

主干模块通常：

- 支持多个后续模块
- 包含可复用机制
- 改变学习者思考方式，不只是做什么

枝叶模块通常：

- 将主干应用于特定上下文
- 引入工具特定变体
- 可以在不破坏主地图的情况下稍后学习

### 第5步：重建教学顺序

优先顺序：

1. 动机和地图
2. 第一个主干模块
3. 下一个主干模块
4. 最小实践
5. 当骨架稳定时的枝叶模块

## 合理性检查

好的重建框架有以下特性：

- 顶级模块少到可以记住
- 清晰的依赖边
- 无章节标题堆砌
- 至少有一个明显的起始模块
- 枝叶材料明确标记为可选或稍后
