# MD 课程包解析

当学习来源是从课程提取的 Markdown 文件文件夹时使用此参考。

## 目标

将一堆 Markdown 课程文件转化为：

1. 候选全局框架
2. 依赖感知的学习顺序
3. 一小组主干模块
4. 枝叶或推迟模块列表
5. 交互式教学的起点

不要把文件夹布局当作最终框架。把它当作证据。

## 解析优先级

如果有，按此顺序读取：

1. 首页、总览或着陆页文件
2. 章节索引或模块索引
3. 框架总结或知识地图文件
4. 习题库
5. 资源清单
6. 模块文件

如果时间有限，起初不要深度阅读每个文件。构建候选结构，然后只钻取下一个教学模块。

## 从每种文件类型提取什么

### 首页或总览文件

提取：

- 主题名称
- 总模块数
- 官方章节名称
- 课程声称帮助学习者做什么

用它来识别课程表面结构，不是最终学习结构。

### 章节索引

提取：

- 章节序列
- 章节分组
- 重复动词如：安装、理解、构建、调试、重构、部署

这些动词常揭示隐藏的主线。

### 框架总结

提取：

- 候选一级模块
- 明确的主干概念
- 声称的概念间关系

不要盲目信任。检查实际章节内容是否支持它。

### 习题库

提取：

- 被反复测试的能力
- 被假定一起出现的概念
- 问题暴露了什么样的误解

习题揭示课程实际重视什么，不只是说重视什么。

### 资源清单

提取：

- 官方文档
- 支持项目
- 与特定模块相关的示例

只有当它们改善当前教学步骤时才拉取。

### 模块文件

提取：

- 解决的问题
- 教授的机制
- 假定的前置知识
- 提到的最小演示或工作流
- 常见失败模式

## 压缩通道

读取高优先级文件后，将课程压缩成 3 层：

1. 课程解决的一句话问题
2. 理解课程所需的最少主干模块集合
3. 延伸、专门化或应用主干的枝叶模块

对每个候选模块问：

- 它是真正的概念、工作流步骤、示例还是只是包装？
- 其他模块依赖它吗？
- 学习者没有它能进步吗？
- 它解释机制还是只是展示？

保留主干模块。把包装和展示材料往下推。

## 噪声模式

课程 Markdown 包中的常见低信号模式：

- 重复的动机章节
- 章节引言和回顾
- 教同一机制的多个示例
- 无机制的吹工具
- 少有结构价值的长字幕
- 概念骨架存在前的实现细节

压缩或忽略这些直到需要时。

## 输出形式

产出：

1. 一句话课程问题
2. 候选框架
3. 主干模块
4. 枝叶模块
5. 推迟模块
6. 下一个最佳教学模块

如果课程顺序不是最佳学习顺序，指出并重排。
