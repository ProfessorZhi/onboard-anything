# P6 6-同时添加多个功能 知识框架

## 本P位置

- **课程章节**：第六章 · Claude Code 进阶应用
- **分P序列**：P6（第六章第6节）
- **课程定位**：P5（权限管理）→ **P6** → P7（GitHub集成/拉取请求审查）
- **知识梯度**：从单实例操作进阶到多实例并行管理

## 核心问题

**如何让多个Claude Code实例在同一项目中并行工作而不互相覆盖文件？**

- 问题本质：多个AI助手同时操作同一代码库时，写入同一文件会导致冲突
- 解决思路：利用Git工作树创建代码库隔离副本，各自独立开发后合并
- 本P附加收获：自定义命令的创建与参数传递机制

## 最小知识骨架

1. **自定义命令创建**
   - 位置：Claude/.claude/commands/文件夹
   - 格式：Markdown文件，第一行为描述，正文为命令内容
   - 参数：$ARGUMENTS变量引用用户输入

2. **自定义命令与Claude.md的作用域区别**
   - commands：不自动加入上下文，需手动触发
   - Claude.md：所有实例自动加载全局配置

3. **Git工作树概念**
   - 定义：代码库的独立工作副本
   - 特性：共享同一仓库历史，独立文件系统操作
   - 命令：git worktree add / list / remove

4. **Git工作树工作流**
   - 创建工作树 → 独立开发 → 提交更改 → 合并回主分支 → 删除工作树

5. **多实例并行开发环境**
   - 每个工作树启动独立终端
   - 每个终端运行独立Claude Code会话

6. **合并与冲突处理**
   - 主分支接收各工作树的合并
   - Claude Code可辅助分析冲突并解决

7. **命令预授权机制**
   - .settings.json配置常用命令自动批准
   - 避免每次执行时的确认提示

## 输入 / 输出 / 前置 / 支撑

**输入**：

- 自定义命令文件（implement feature.md）
- 主分支代码库
- 工作树命名（ui功能、测试功能、质量功能）

**输出**：

- 多个功能分支的代码更改
- 合并后的主分支更新
- 变更记录文件（front end changes.md等）

**前置对象**：

- .settings.json（已配置权限）
- Git仓库（主分支）
- Claude Code运行环境

**支撑对象**：

- .trees/文件夹（工作树存储目录）
- 独立终端窗口（每个工作树一个）
- 测试框架（playwright示例）
- 代码质量工具（格式化工具、检查脚本）

## 易错点

**1. 混淆commands和Claude.md的作用域**
- 以为创建在commands文件夹的命令会自动被所有实例加载
- 实际上自定义命令需要手动触发调用
- Claude.md才是全局生效的配置文件

**2. 忽略工作树隔离保护**
- 直接在代码库开多个终端窗口操作
- 导致同一文件被多个实例覆盖
- 必须通过git worktree创建隔离环境

**3. 合并前未规范提交**
- 工作树内随意commit，消息不清晰
- 合并时无法追溯各工作树的变更意图
- 应为每个工作树维护规范的提交历史

**4. 忽略.settings.json预授权**
- 每次执行常用命令都要手动确认
- 降低并行开发效率
- 应提前配置playwright等常用工具的权限
