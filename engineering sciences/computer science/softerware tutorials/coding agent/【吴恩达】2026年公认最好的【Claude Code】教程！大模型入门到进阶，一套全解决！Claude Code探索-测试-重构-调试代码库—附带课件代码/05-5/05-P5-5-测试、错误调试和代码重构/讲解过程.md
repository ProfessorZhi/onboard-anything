## 本P定位
本P是第五课的核心实践环节，承接P4对代码库的基础探索，进入真实的开发工作流。学员将在此学习如何用Claude Code进行系统化的错误调试，而非盲目复制错误信息。具体而言，本P将展示从识别问题、编写测试、定位根因到修复验证的完整闭环，并在后半部分演示一个实用的代码重构场景。

1. **问题发现**：回到应用发现查询失败，但代码库缺少测试

4. **验证修复**：运行测试确认修复有效，浏览器验证
5. **代码重构**：将AI生成器从单轮工具调用改为支持多轮迭代调用

## 本P结论

**测试驱动的错误调试方法**：
1. 发现应用问题后，先编写测试而非直接复制错误给AI
2. 测试能精确定位问题根因（本P中发现max_results为0）
3. 测试同时验证修复的正确性，确保后续修改不破坏已有功能

**AI生成器的多轮工具调用重构**：
1. 将单轮搜索改为支持多轮迭代调用
2. 通过子代理并行探索多种重构方案
3. 重构后能处理复杂查询（如跨课程比较）
4. 保持向后兼容，不影响现有RAG系统和API端点

**核心工程原则**：
- 测试应验证外部行为而非内部状态
- 提示应提供当前行为和期望行为的具体描述
- 复杂任务可以让Claude探索多个方案再选择

## 易错点

1. **直接复制错误信息给AI**：看似高效，实则缺乏系统性，容易被无意义的追逐带偏。正确做法是先编写测试建立验证基础。

2. **max_results参数意外变为0**：配置问题可能导致向量搜索无结果返回，但不会被明确报错，需要通过测试才能发现。

3. **单轮工具调用无法处理复杂查询**：简单搜索设定每查询一次工具调用足够，但比较课程、获取大纲等场景需要多轮迭代。

4. **重构时忽略向后兼容**：修改核心逻辑时应确保不破坏现有RAG系统和API端点。

5. **测试关注内部状态而非外部行为**：测试应验证查询→工具调用→响应的完整流程，而非过度关注内部实现细节。

6. **复杂重构缺少方案对比**：不确定最佳方案时，可以让AI探索多个选项而非强制它随机选择一个。
