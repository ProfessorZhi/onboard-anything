# P06 讲解过程 - 同时添加多个功能

- 所属章节：`6-6-同时添加多个功能`
- 分P标题：`P06 同时添加多个功能`
- 分P主文档：`P06-同时添加多个功能.md`
- 字幕：`字幕.md`

## 本P定位

本P是《Claude Code 教程》的第6课，在前几P已介绍过 Claude Code 的基础命令、内置功能后，本P进入**进阶用法**：如何同时处理多个功能而不产生文件覆盖冲突。这是大模型辅助开发中"规模化使用"的关键节点，也为下一课（终端外使用 Claude Code、与 GitHub 集成审查 PR）做铺垫。

## 本P主线

本P围绕一个核心目标展开：**如何安全地让多个 Claude Code 实例并行工作**。主线分三个阶段：

1. **铺垫**：多会话概念 → 自定义命令机制
2. **核心工具引入**：Git 工作树（git worktree）
3. **实战演示**：创建三个工作树（UI功能、测试功能、质量功能），并行运行 Claude Code，修改不同文件，最终合并并解决冲突

## 讲解过程

### 00:00 - 02:00 引入多会话场景

**讲了什么：** 开门见山，指出可以用 Claude Code 打开多个会话同时处理多个功能，并预告本P的核心问题——管理这些会话并确保不覆盖同一文件。

**为什么这里讲：** 上一P（或前几P）已经建立了"单次对话内与代码库交互"的基础，自然引出"如果我要同时做三件事呢？"的需求。这是功能的自然延伸。

**和前后内容的连接：** 提出了"多会话"这个需求，但还没讲解决方案，为接下来引入 Git 工作树埋下伏笔。

### 02:00 - 06:30 自定义命令机制铺垫

**讲了什么：** 老师说明 Claude Code 内置了许多命令，但你也可以创建自己的命令来自定义功能。具体操作路径是：在 `Claude` 文件夹内新建一个名为 `commands` 的文件夹，在此文件夹内创建一个 Markdown 文件，使用我们想要的命令名称作为文件名。

老师演示创建了名为 `implement feature.md` 的自定义命令文件。

**关键讲解点：**
- 自定义命令的核心价值：传递参数。使用 `$arguments` 变量引用参数
- 自定义命令的示例内容：描述正在实现新功能、用户指定具体功能内容、要求 Claude Code 明确知晓仅针对前端功能执行操作并将修改写入 `front_end_changes.md`
- 老师强调：自定义命令的内容不会自动添加到上下文（与 `CLAUDE.md` 不同）
- `CLAUDE.md` 适用于全局配置；自定义命令适合存放可能跨对话使用的特定命令

**为什么这里讲：** 自定义命令是本P实战演示中会用到的工具。先铺垫好机制，后面直接调用时不至于突兀。

**和前后内容的连接：** 后续在三个工作树中，老师会实际使用这个 `implement feature` 命令。先讲机制后见应用。

**资源与对象：**
- `Claude` 文件夹（字幕直接提到）
- `commands` 文件夹（字幕直接提到，需在 Claude 文件夹内新建）
- `implement feature.md` 文件（字幕直接提到，自定义命令文件名）
- `$arguments` 变量（字幕直接提到，用于传递参数）
- `front_end_changes.md`（字幕直接提到，用于记录前端修改）
- `CLAUDE.md` 文件（字幕直接提到，对比说明其与自定义命令的作用域差异）

### 09:30 - 10:30 切换话题：Git 工作树解决方案

**讲了什么：** 老师正式引出 Git 工作树的概念。解释为什么需要它：当有多个 Claude Code 实例操作同一文件时，会产生覆盖错误并引发混乱。Git 提供了名为**工作树（worktree）**的优秀选项，允许创建代码库的独立副本，操作后合并。

**为什么这里讲：** 这是本P的**核心解决方案**。从"多会话需求"到"冲突问题"再到"Git 工作树"这个答案，逻辑链条完整。

**为什么这里讲：** 动手演示 git worktree 的具体用法，从创建文件夹到执行命令再到验证，逐层展开。

## 本P结论

1. **多会话并行工作可行但需管理**：通过 Claude Code 可以打开多个会话同时处理多个功能。

2. **Git 工作树是并行开发的核心工具**：使用 `git worktree add` 创建独立工作目录，避免多个 Claude Code 实例修改同一文件时产生覆盖冲突。

3. **自定义命令提升效率**：在 `Claude/commands/` 目录下创建 Markdown 文件可定义自定义命令，使用 `$arguments` 变量传递参数；`CLAUDE.md` 适用于全局配置，自定义命令适合跨对话使用的特定任务。

4. **Claude Code 可以辅助 Git 工作流**：生成规范提交信息、执行添加和提交操作、解决合并冲突（尤其对小型冲突有效）。

5. **完整工作流**：`git worktree add` → 并行开发 → 分别提交 → `git merge` → 解决冲突 → `git worktree remove`（可选）。

6. **Claude Code 的协作能力边界**：可以在整个技术栈各部分并行编辑，在代码检查和 DevOps 环节操作，不覆盖原有内容。

## 易错点

1. **混淆自定义命令与 CLAUDE.md 的作用域**：自定义命令文件中的内容不会自动添加到上下文（"这里的内容不会自动添加到上下文"），而 CLAUDE.md 是全局生效的。
2. **忽视 `$arguments` 变量的使用**：创建需要参数的自定义命令时，必须使用 `$arguments` 变量来引用传入的参数，否则参数无法传递。
3. **多会话时不使用工作树直接并行**：直接在代码库中打开多个终端窗口操作同一文件会产生覆盖错误（"将会产生覆盖错误并引发大量混乱"）。
4. **忘记分别提交各工作树的更改**：在合并前必须先在各工作树中分别提交，否则无法正确合并。
5. **合并冲突未验证就提交**：即使 Claude Code 能辅助解决冲突，仍应运行测试确保合并后代码库正常（"确保完成后我们可以运行测试并验证代码库正常"）。
6. **权限配置的遗漏**：频繁使用的命令（如 Playwright）每次都需确认，可通过 `.claude/settings/local.json` 配置允许的命令来简化。
