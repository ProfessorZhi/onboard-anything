## P02 什么是Claude Code？

**分P定位**：智能代理架构与核心概念

**前置模块**：P01

**核心内容**：
- **智能代理架构**：模型+工具链+运行环境
- **两种操作入口**：CLI（命令行）vs Sonar
- **内置工具集**：文件读写、编辑、搜索、子代理、bash命令
- **MCP可扩展性**：Model Context Protocol连接外部服务
- **记忆系统**：CLAUDE.md配置文件+本地对话存储

**后续模块**：P03

**能力边界**：
- 不仅是代码生成工具，更是“发现、解释、设计”能力的载体
- 数据留在本地，不索引代码库

# 第二部分：核心开发模块

## P04 添加功能

**分P定位**：功能开发工作流

**前置模块**：P03（代码库理解能力）

**核心内容**：
- **@标签文件引用**：直接指定目标文件，避免猜测
- **计划模式（Shift+Tab）**：先规划后执行，用户审批后再修改
- **自动接受修改**：逐步建立信任后启用，减少重复授权
- **上下文管理**：新功能开发时清空对话历史，避免污染
- **截图反馈法**：截屏展示给Claude，比文字描述更精确
- **MCP扩展协议**：Model Context Protocol连接外部工具（演示Playwright MCP）

**后续模块**：P05

**工作流依赖**：
```
计划模式 → 用户审批 → 自动接受修改
文件引用 → 高质量上下文 → 正确代码输出
新对话 → 清洁上下文 → 专注当前任务
```

**分P定位**：测试驱动调试与架构重构

## P07 GitHub集成与Hooks探索

**分P定位**：团队协作与生命周期控制

**前置模块**：P06

**核心内容**：
- **GitHub集成架构**：Claude Code SDK → GitHub App → 平台协作能力
- **GitHub操作工作流**：安装配置 → PR自动审查 → Issue响应 → 自我复核
- **Hooks生命周期钩子类型**：
  - pre-tool-use（工具执行前）
  - post-tool-use（工具执行后）
  - on-notify（通知触发）
  - on-user-input（用户输入时）
  - on-task-stop（任务停止时）
  - on-subagent-response（子代理响应时）
- **Hook配置结构**：匹配器规则（matcher）+ 执行动作（shell命令）
- **resume恢复机制**：--resume标志恢复历史对话上下文

**后续模块**：P08、P09

**能力边界**：
- 突破终端边界，进入GitHub平台协作
- Hooks实现工具执行前后的自定义逻辑注入

## P09 基于Figma原型创建Web应用

**分P定位**：MCP协议集成实战

**前置模块**：P07（GitHub集成与Hooks基础）

**核心内容**：
- **MCP协议**：Claude Code与外部工具通信的中间层标准
- **Figma MCP Server**：将Figma设计稿转为可读代码/图像/设计规则
- **Playwright MCP Server**：自动化浏览器测试与截图验证
- **Next.js App Router**：目标应用框架，代码组织在app文件夹
- **FRED API**：联邦储备经济数据库，获取真实数据
- **设计到代码闭环**：Figma原型 → Claude Code → Next.js应用 → Playwright验证

**后续模块**：P10

**关键依赖**：
- Figma开发者模式必须启用，否则MCP Server无法工作
- MCP配置不跨项目，每个独立项目需单独配置

**MCP工作流**：
```
Figma设计稿 → Figma MCP Server → Claude Code（读取设计）
                                        ↓
                                  Next.js应用
                                        ↓
                              Playwright MCP Server
                                        ↓
                                  自动化验证
```

# 第五部分：总结内化模块

## P10 Conclusion结论

**分P定位**：核心技能闭环与工程方法论

**前置模块**：P02-P09（全部技术内容）

**核心内容**：

### 四大核心技能

| 技能 | 能力描述 | 适用场景 |
|------|---------|---------|
| Explore | 理解代码库结构，快速定位关键代码 | 接手陌生项目、理解架构 |
| Test | 编写和运行测试，验证功能正确性 | 质量保障、回归测试 |
| Refactor | 改进代码质量，提升可维护性 | 技术债务清理、架构优化 |
| Debug | 定位和修复问题，确保系统稳定 | Bug修复、问题诊断 |

### 三条使用原则

```
明确上下文 → 给出清晰指令 → 指向相关文件 → 执行操作（E/T/R/D）
```

1. **清晰指令原则**：明确告诉AI要完成什么任务，而非模糊描述
2. **明确上下文原则**：提供项目背景、约束条件和已有的实现逻辑
3. **指向相关文件原则**：在对话中明确引用需要操作的具体文件路径

### 能力边界扩展

- **CLAUDE.md配置**：将代码库规范持久化为Claude Code启动时读取的配置
- **MCP服务扩展**：通过Model Context Protocol连接Playwright、Figma等外部工具

# 易错点汇总

| 分P | 易错点 |
|-----|-------|
| P02 | 把Claude Code当成纯代码生成工具；误以为代码库会被索引上传 |
| P03 | 混淆/clear与/compact；忽视Escape键中断能力；混淆三种CLAUDE.md作用域 |
| P04 | 跳过计划直接编码；新功能不清空历史导致上下文污染；不使用@标签 |
| P05 | Bug定位方法不当；未先写测试就修复 |
| P06 | 并行工作树文件覆盖冲突；自定义命令参数使用错误 |
| P07 | Hooks匹配器规则配置错误；GitHub App权限设置不当 |
| P08 | 职责分离不彻底；提示工程不够具体 |
| P09 | Figma开发者模式未启用；MCP配置未针对项目单独设置 |
