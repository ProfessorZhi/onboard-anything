# P4 4-添加功能 知识框架

## 本P位置

**课程进度**：P4/总课程数（推测12-15P）

**课程阶段划分**：

- P1-P3：基础认知阶段——代码库结构、Claude Code基础操作、配置方法
- **P4：实操入门阶段——开始实际功能开发，展示完整工作流**
- P5：问题解决阶段——调试与测试（预告）
- P6+：进阶能力阶段

**承上启下**：
- 前置：P1-P3建立的基础认知
- 后续：P5的调试能力，形成"开发→验证→修复"闭环

---

## 核心问题

**如何运用Claude Code完成一个功能的完整开发？**

具体分解为：

- 如何正确提供上下文让Claude理解需求
- 如何通过计划模式降低大修改的风险
- 如何高效迭代视觉需求
- 如何实现程序化的自动化测试
- 如何开发后端工具并集成到系统

---

## 最小知识骨架

### 骨架1：工作方法论三件套

**@标签引用文件**

- 作用：指定Claude需要阅读的文件/文件夹
- 操作：输入@后选择文件，支持Tab补全
- 原理：Claude代码质量取决于提供的上下文质量
- 适用场景：任何需要Claude理解项目结构或修改代码时

**计划模式**

- 激活：连续按Shift+Tab两次
- 作用：先让Claude制定修改计划，批准后再执行
- 适用场景：较大修改、涉及多个文件、需要前后端配合
- 输出：待办清单式计划

**自动接受修改**

- 激活：Shift+Tab一次
- 作用：避免重复授权信任的操作
- 配合：可写入CLAUDE.md记录偏好

### 骨架2：前端功能开发流程

```
需求分析 → 引用文件 → 激活计划模式 → 审查计划 → 自动执行 → 验证
```

- 关键：先明确做什么，再动手coding
- 工具：计划模式生成待办清单
- 偏好：可写入CLAUDE.md（如"不要启动服务器"）

### 骨架3：视觉迭代方法——截图分析

**传统方式**：描述问题→手动截图上传→等待分析
**截图分析方法**：直接粘贴截图→描述期望效果→Claude自主分析并修改

- 原理：利用Claude的视觉理解能力
- 优势：比语言描述更准确，减少来回沟通
- 适用：样式调整、布局问题、颜色对比等视觉需求

### 骨架4：对话上下文管理

**何时清空**：

- 构建新功能时
- 需要更长上下文窗口时
- 之前对话内容与当前任务无关时

**如何清空**：点击"新对话"按钮（可让Claude添加此功能）

### 骨架5：MCP服务器与自动化测试

**MCP（Model Context Protocol）**：

- 定义：允许Claude Code连接外部数据源和系统的协议
- 用途：扩展Claude的工具能力

**Playwright MCP服务器**：

- 添加命令：claude mcp add playwright npx playwright
- 提供能力：执行JS、上传文件、按键、导航、截图、浏览器自动化
- 核心价值：程序化的自动化测试，Claude可自主完成"截图→分析→修改→验证"闭环

### 骨架6：后端工具开发流程

```
理解现有架构 → 制定计划 → 创建工具代码 → 更新系统提示 → 注册到RAG系统 → 测试
```

- 关键文件：search.pi（现有工具文件）
- 关键步骤：理解架构后才开始写代码

### 骨架7：功能验证

**前端验证**：

- 浏览器手动测试
- 或使用Playwright MCP自动化测试

**后端验证**：

- 检查工具输出格式
- 确认数据来源正确显示

## 关系与依赖

```
工作方法论(1)
    │
    ├─→ 前端开发(2)
    │       │
    │       └─→ 视觉迭代(3) ←─ 截图粘贴
    │
    ├─→ 上下文管理(4)
    │
    └─→ 后端开发(6)
            │
            └─→ 功能验证(7)

MCP服务器(5) ──→ 支撑自动化测试 ──→ 赋能所有开发环节(2)(3)(7)
```

**依赖方向**：

- 工作方法论是所有开发活动的基础
- MCP服务器不是必选项，但在复杂场景下大幅提升效率
- 前后端开发流程相对独立，但需要协调（特别是工具注册）

## 易错点

**1. 不提供上下文直接让Claude修改**
- 表现：用模糊描述让Claude猜测
- 后果：修改错误文件，浪费时间
- 正确：先用@标签引用相关文件

**2. 大改动跳过计划模式**
- 表现：直接让Claude开始写代码
- 后果：遗漏细节、破坏现有功能
- 正确：激活计划模式审查后再执行

**3. 对话上下文污染**
- 表现：继续使用旧对话处理新功能
- 后果：Claude被无关内容干扰
- 正确：新功能前清空对话

**4. 视觉需求用语言描述**
- 表现：写一大段文字描述样式问题
- 后果：描述不准确，反复修改
- 正确：直接粘贴截图

**5. 忽略CLAUDE.md的价值**
- 表现：每次重复说明偏好
- 后果：Claude反复犯同一错误
- 正确：重要偏好写入CLAUDE.md

**6. 不验证功能就继续**
- 表现：写完代码直接开始下一个任务
- 后果：问题累积，难以定位原因
- 正确：每个功能开发后立即测试

**7. 前后端修改不同步**
- 表现：前端调用了后端未注册的接口
- 后果：功能无法正常工作
- 正确：确保工具注册到系统后再使用

**8. 过度依赖手动操作**
- 表现：每次都手动截图、手动授权
- 后果：效率低下
- 正确：启用自动接受、MCP自动化
