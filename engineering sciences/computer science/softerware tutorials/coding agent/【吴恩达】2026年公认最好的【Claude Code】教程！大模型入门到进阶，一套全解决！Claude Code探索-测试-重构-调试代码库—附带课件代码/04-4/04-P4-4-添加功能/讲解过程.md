# P4 4-添加功能

## 本P定位

本P是Claude Code教程的核心实操演示环节。在P1-P3介绍了基础操作后，本P开始**实际为聊天机器人项目添加功能**，展示完整的功能开发工作流：从需求分析→技术方案→代码实现→测试验证的全流程。

## 本P主线

1. **明确需求**：为界面添加来源链接显示功能
2. **建立方法论**：正确引用文件 + 计划模式
3. **前端开发**：添加带来源引用的界面 → 视觉优化
4. **引入新工具**：MCP服务器 + Playwright自动化测试
5. **后端开发**：添加获取课程详情的工具
6. **功能验证**：测试前后端功能是否正常

### 00:48 - 01:45 工作方法论：引用文件与计划模式

讲了什么：介绍两个重要工作组件：
- **正确引用文件**：使用@标签指定文件/文件夹，可Tab补全。强调Claude代码质量取决于提供的上下文。
- **计划模式**：使用Shift+Tab激活，先让Claude制定修改计划，批准后再执行。适合较大修改。

为什么这里讲：在动手之前先建立规范的工作方法论，确保后续实操高效有序。

关键演示：
- 用@标签引用文件，Claude会阅读必要的文件
- Shift+Tab激活计划模式，Claude生成待办清单
- 启用自动接受修改（--approve-all）避免重复授权

为什么这里讲：介绍了一种**快速迭代视觉需求**的强大方法——利用Claude的视觉能力分析截图，而不是用语言描述问题。

### 06:12 - 07:01 MCP服务器与Playwright

关键演示：

## 本P结论

1. **正确引用文件**（@标签）是提高Claude代码质量的关键，务必提供正确的上下文
2. **计划模式**适合较大修改，先规划后执行，降低出错风险
3. **截图分析方法**是快速迭代视觉需求的强大工具
4. **清空上下文**是构建新功能时的推荐做法，避免无关内容干扰
5. **MCP + Playwright**实现了程序化的自动化测试与修改，Claude可以自主完成整个迭代
6. **CLAUDE.md文件**可以记录项目偏好（如不启动服务器），帮助Claude理解团队习惯
7. 后端工具开发遵循：理解架构→制定计划→创建工具→更新系统提示→注册到RAG的流程

## 易错点

1. **不提供上下文直接让Claude修改代码**：会导致Claude猜测错误文件，效率低下。正确做法是先用@标签引用相关文件。

2. **大改动用claude直接编写**：没有计划的修改容易遗漏细节。正确做法是使用计划模式，先让Claude分析需求和架构。

3. **不清理无关对话上下文**：构建新功能时，之前对话可能干扰当前任务。正确做法是在开始新功能前清空对话历史。

4. **手动截图上传迭代视觉需求**：效率低且描述可能不准确。正确做法是粘贴截图让Claude自主分析。

5. **每次都手动授权工具执行**：繁琐且影响效率。正确做法是对于信任的操作启用自动接受，或在CLAUDE.md中记录偏好。

6. **忽略项目内存文件**：每个项目的偏好和规则应该记录在CLAUDE.md中，避免Claude每次都重复错误操作。

7. **不验证功能就继续**：开发后必须测试确认功能正常，否则问题会累积到后续排查。

8. **前端后端修改没有协调**：新功能往往需要前后端配合，需要确保两边的修改一致（如注册工具到系统）。
