# P8 8-重构Jupyter笔记本并创建仪表盘
## 本P定位
本P是课程第8章的第8个分P，属于“Claude Code多场景应用”章节的进阶内容。在此之前（P5-P7）已讲解了Web应用开发、API调用、测试与调试等场景，本P将场景从Web应用转向数据科学领域，展示如何用Claude Code处理Jupyter笔记本的重构工作，并最终将其转换为可部署的Streamlit仪表盘。

## 本P主线
从一个问题重重的Jupyter笔记本出发，展示如何通过精心设计的提示词引导Claude Code完成代码重构、职责分离、可视化改进，最终生成一个可部署的Streamlit仪表盘，实现从混乱笔记本到整洁仪表盘的完整工作流。

## 讲解过程

### 00:02 - 00:15 课程目标导入
讲了什么：开篇点明本节主题——使用Claude Code重构Jupyter笔记本并创建仪表盘。
为什么这里讲：作为本P的开场，需要让观众明确这节课要解决的问题和达到的目标。
前后连接：承接前几P对Web应用的讲解，自然过渡到数据科学场景。

### 00:15 - 00:48 问题背景展示
讲了什么：教师展示一个包含虚构电商数据的Jupyter笔记本，这些数据从几个CSV文件中读取，使用Pandas读取数据并显示统计信息。教师指出虽然能获取所需结果，但数据组织不够理想，格式和显示不符合预期。
为什么这里讲：建立问题意识，让观众理解为什么要重构——现有代码虽然能运行，但缺乏可维护性和可展示性。
前后连接：为后续的重构需求做铺垫，具体化问题场景。

### 00:48 - 01:00 年同比分析痛点
讲了什么：教师希望从数据中提取年收入增长情况，进行2023年对比2022年的同比分析。教师演示需要逐步合并两个CSV和数据框，操作耗时较长且出现警告信息。
为什么这里讲：展示具体业务需求，揭示当前笔记本在处理复杂业务逻辑时的局限性。
前后连接：引出重构的核心目标之一——更好地整合业务逻辑。

### 01:00 - 02:08 可视化改进需求
讲了什么：教师继续分析现有笔记本，指出需要计算核心指标如平均订单价值（目前只有2023年数据），并在接近底部时仍有改进空间。笔记本计算各州销售额并生成可视化图表。教师总结：“随着我们开始使用像Claude Code这样的工具，我们可以做得更好”。
为什么这里讲：明确重构的具体需求点，包括数据处理、指标计算、可视化展示三个层面。
前后连接：从问题描述转向解决方案预告，自然过渡到使用Claude Code进行重构。

### 02:08 - 02:19 重构方案预告
讲了什么：教师正式宣布将展示如何使用Claude Code重构笔记本，更好地整合业务逻辑到演示中。
为什么这里讲：作为讲解的分界点，宣告从问题分析阶段进入实际操作阶段。
前后连接：引入Claude Code作为解决方案工具。

### 02:19 - 04:09 详细提示词讲解
讲了什么：教师打开Claude，准备输入提示词。在正式粘贴前，教师新建Markdown文件用于逐步讲解。首先强调使用符号引用文件名和文件夹名称的技巧（这里指的应该是@符号引用）。然后教师分享了一个重要经验：可以随时询问Claude“这就是我想要实现的目标，请给出完成此任务的最佳提示”。接下来教师详细解释提示词的具体内容：重构代码结构以满足特定要求、代码质量改进、对Python文件有特定需求。具体包括创建单独的Python文件用于加载和处理数据、另一个Python文件用于计算指标、关注职责分离以便更好地测试和理解代码、改进可视化效果、增加数据处理配置选项。最后教师建议告知Claude预期，而非简单说“重构这个”。
为什么这里讲：这是本P的核心教学点之一——如何撰写高质量的提示词。教师的讲解涵盖技巧（符号引用）、策略（询问Claude获取最佳提示）、具体需求描述（职责分离、可测试性）、表达方式（明确预期）。
前后连接：为后续Claude Code的实际执行提供输入材料。

### 04:09 - 04:49 Claude Code执行过程
讲了什么：教师复制粘贴提示词，Claude Code开始工作。教师解释Claude Code会使用不同于其他工具的工具集，包括Read notebook来探索单元格、分析结构、创建重构笔记本。执行后会生成数据加载辅助函数Python文件、处理数据的Python文件、计算业务指标的Python文件、requirements.txt、重构后的笔记本，以及README文件。教师接受Python文件的更改，看到Python代码被正确编写，第二个包含指标计算的文件也被创建。
为什么这里讲：展示Claude Code处理笔记本任务的完整流程，让观众了解AI辅助重构的实际工作方式。
前后连接：生成的文件为后续验证和测试做准备。

### 04:49 - 05:40 重构结果验证
讲了什么：教师查看重构后的笔记本，指出代码更加整洁，采用面向对象方法比直接嵌入所有内容在笔记本内部更整洁。教师运行所有单元格，一开始一切正常，但向下滚动时出现键错误（KeyError）。教师让Claude Code帮忙修复，提供出错的代码以便Claude分析，然后Claude进行了修复。
为什么这里讲：展示重构过程中的调试环节，体现Claude Code在代码审查和修复方面的能力。
前后连接：修复完成后进入效果展示阶段。

### 05:40 - 06:46 重构效果展示
讲了什么：教师展示重构后的可视化效果。首先查看业务指标，得到更友好的摘要显示，客户满意度信息比之前更容易处理。收入分析显示更清晰的数值和更多显示信息，可视化效果更美观。类别表现有了更好的工具来分析类别和相关收入。地理表现方面，各州可视化效果不错，悬停可获取各州详细信息。初始配送分析一切如预期般良好。教师总结重构笔记本达到了更直观展示信息和分离关注点的目标。
为什么这里讲：充分展示重构的成果，让观众看到从混乱到整洁、从难懂到易读的转变。
前后连接：从Jupyter笔记本场景自然过渡到Streamlit仪表盘场景。

### 06:46 - 09:00 Streamlit仪表盘创建
讲了什么：教师宣布将笔记本转为网页界面，使用Streamlit驱动的美观仪表盘。教师解释Streamlit是用于展示界面特别是仪表盘的Python库，且非常容易无缝过渡到Python文件。新建Markdown文件命名为“转换为仪表盘.md”，教师讲解具体的需求内容：标题和筛选器的头部、包含收入增长平均订单值和高订单的卡片KPI、收入类别图表、按州划分的收入图表、底部满意度条形图。教师强调需要明确数据展示和信息渲染方式。教师粘贴提示词，Claude开始创建仪表盘。
为什么这里讲：这是本P的第二个核心场景——从笔记本迁移到Web仪表盘。教师先讲解需求再执行演示，符合教学逻辑。
前后连接：将数据科学场景扩展到Web展示场景。

### 09:00 - 09:32 仪表盘实现与依赖处理
讲了什么：教师展示Claude创建的仪表盘代码结构：导入Streamlit、Pandas、可视化模块（包括Plotly）、自定义CSS样式、页面配置（使用st对象）。Claude更新了requirements.txt并创建README文件展示仪表盘内容。教师打断Claude，告诉它不要新建README只需修改现有文件。
为什么这里讲：展示Streamlit仪表盘的典型代码结构，同时体现与Claude Code交互时的灵活调整能力。
前后连接：进入运行和调试阶段。

### 09:32 - 10:40 启动服务器与问题发现
讲了什么：教师打开新终端窗口，使用streamlit run dashboard.py启动服务器。但命令不可用，需要安装streamlit依赖。教师执行pip install安装所有必要依赖。安装完成后启动服务器，在浏览器中查看效果。发现默认显示2023年收入和增长数据，但向下滚动时发现部分内容缺失，卡片为空且冗余。教师解释2024年作为起点有困难，因为2024年并非每月都有数据，月度增长情况有些混乱。
为什么这里讲：展示完整的部署流程，同时揭示初始生成代码的问题，为后续优化做铺垫。
前后连接：进入迭代优化阶段。

### 10:40 - 11:09 仪表盘优化
讲了什么：教师决定让Claude清理这些冗余部分，设置默认年份为2023年（因为2024年数据不完整），并添加月份筛选功能。教师新建提示词：确保默认年份设为2023年、添加月份筛选功能、移除空卡片。Claude更新仪表盘代码，添加了月份筛选处理功能，CSS也在调整，图表容器看起来更整洁。
为什么这里讲：展示通过迭代提示不断完善代码的实践方法。
前后连接：进入最终效果验证。

讲了什么：教师返回浏览器查看效果，确认默认年份是2023年，空卡片已经消失。教师总结本P成果：从一个有点混乱、阅读理解困难的笔记本，迁移到了可以部署的仪表盘，可以与团队成员共享并快速迭代。教师预告下一课内容：构建Next.js应用，使用Claude Code连接MCP服务器用于Figma和Playwright来处理线框图，快速构建强大的网络应用。

前后连接：结束本P并预告下一P内容。

## 本P结论
Claude Code具备完整的Jupyter笔记本处理能力，可以将混乱的笔记本重构为结构清晰、职责分离的代码库。通过精心设计的提示词，可以实现：数据加载逻辑与业务逻辑分离、更美观的可视化效果、更好的可配置性。最终可以将重构后的笔记本无缝转换为Streamlit仪表盘，实现从数据分析到Web展示的完整工作流。

## 易错点
1. 直接说“重构这个”而不给出具体需求和预期效果，Claude可能无法准确理解意图
2. 提示词中缺少职责分离的明确要求，可能导致所有代码仍然混在一起
3. 忽略数据完整性和年份配置问题，导致生成的仪表盘出现空值或不合理的数据展示
4. 一次性要求太多功能而不进行迭代，可能导致生成代码质量不高
5. 没有意识到可以询问Claude获取最佳提示词的建议，可能错失优化提示的机会
