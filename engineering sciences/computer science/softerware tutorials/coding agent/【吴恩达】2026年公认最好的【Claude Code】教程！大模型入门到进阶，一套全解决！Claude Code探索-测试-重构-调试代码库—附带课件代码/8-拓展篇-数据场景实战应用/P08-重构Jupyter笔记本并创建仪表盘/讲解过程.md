# P08 讲解过程 - 重构Jupyter笔记本并创建仪表盘

- 所属章节：`8-8-重构Jupyter笔记本并创建仪表盘`

## 本P定位

本P是课程第二部分（第5-8课）的最后一集，承接第7课对Web应用的端到端探索。在前几课建立的基础上，本P将应用场景从Web后端扩展到数据分析和可视化领域，展示Claude Code处理Jupyter笔记本的能力。

## 本P主线

使用Claude Code重构一个组织混乱的电商数据分析笔记本，通过职责分离（创建独立的Python模块）和迭代优化，最终将其转换为可部署的Streamlit仪表盘，实现从混乱笔记本到专业数据展示工具的完整旅程。

## 讲解过程

### 00:02 - 00:15 课程预告与目标设定

老师首先指出Claude Code具备读取和编辑Jupyter笔记本的功能，这是本课的技术基础。随后明确本课的两个核心任务：一是使用Claude Code重构现有的Jupyter笔记本，二是将重构后的笔记本转换为仪表盘。

老师简要回顾了前几课的内容——已经学习过如何使用Claude Code处理Web应用，从前端到后端。现在转换思路，展示如何用同样的工具处理数据分析和Jupyter笔记本场景。

**连接前后**：这里为后续的操作设定预期——用Claude Code处理数据工作流，与处理Web应用遵循相似的模式（read→analyze→modify→write）。

### 00:15 - 00:50 展示原始笔记本的问题

老师打开一个包含虚构电商数据的Jupyter笔记本开始演示。数据从几个CSV文件中读取，存放在“电商数据”文件夹中。老师使用Pandas读取这些CSV数据，获取了一些统计信息，但很快指出问题所在：

- 数据组织方式不够理想
- 统计信息的格式和显示可能不符合预期
- 想要展示的内容和实际效果之间存在差距

**讲解意图**：通过展示原始笔记本的问题，建立“为什么需要重构”的动机。这些问题包括代码凌乱、缺乏结构、难以维护等，是后续重构方向的铺垫。

### 00:50 - 01:00 分析业务需求缺口

老师继续查看笔记本，发现更多需要改进的地方。笔记本试图进行年收入增长的同比分析（2023年对比2022年），但过程中出现了一些警告，且仅展示所需内容就耗时较长。虽然最终得到了所需的数字，但过程显得繁琐凌乱。

老师指出笔记本中还在计算平均订单价值等核心指标，但这些指标的计算存在问题——只处理了2023年的数据，如果想要2024年及以后的数据怎么办？此外还有一些可视化图表，但接近底部时仍有改进空间。

**关键观察**：老师指出了核心问题——数据不完整（缺少2024年数据）、缺乏灵活性（硬编码年份）、可视化效果需要提升。

### 01:00 - 01:05 按各州分析销售额

老师继续查看笔记本，看到按各州划分的销售额分析，以及更多生成可视化图表的操作。此时老师明确表达了改进的方向：

> “随着我们开始使用像Claude Code这样的工具，我们可以做得更好。”

**转折点**：这是从问题陈述转向解决方案的节点。老师明确表示，Claude Code可以帮助改进这种混乱的状态，实现更好的代码组织和数据展示。

### 01:05 - 01:41 介绍重构方案与提示词策略

老师打开Claude Code，准备输入重构提示词。在粘贴提示词之前，老师新建一个Markdown文件来逐步讲解这些步骤，说明为什么要采取这些措施。

老师首先解释提示词中引用的用法：使用符号引用文件名，同时引用文件夹名称。这种符号引用的技巧是让Claude Code理解上下文的关键。老师强调：

> “如果你不确定如何最佳提示，或如何向Claude提出需求，随时可以询问Claude。”

**提示词策略**：

1. **需求描述**：需要重构代码结构以满足特定要求，需要代码质量改进，对某些Python文件有特定需求
2. **职责分离**：创建一个单独的Python文件用于加载和处理数据，另一个Python文件用于计算指标
3. **代码质量**：关注职责分离，这有助于更好地测试和理解代码
4. **可视化改进**：要求改进可视化效果
5. **配置灵活性**：希望对处理的数据有更多配置

老师建议的提示习惯：

> “告知Claude你的预期是个好习惯，与其只说'重构这个'。”

更具体的请求包括：请为重构后的笔记本创建底层资产、请提供函数文档和业务指标计算方法、并生成其他重要依赖文件、以及说明如何使用的README文件。

**连接前后**：这个部分建立了一个重要的提示工程模式——主动向AI询问最佳提示方式，而不是盲目尝试。

### 01:41 - 01:54 执行重构任务

老师将提示词粘贴到Claude Code中。这里有一个值得注意的细节：

> “你会注意到粘贴一定行数后，Claude Code不会显示所有内容。为了在终端中保持内容更简洁。”

老师解释接下来将使用的工具集与之前不同：不再是read（读取文件），而是read notebook（读取笔记本），但之前看到的任务清单仍将被执行。Claude Code会探索笔记本的单元格、分析结构，并创建预期的重构笔记本。

**工具切换说明**：老师明确指出read notebook是专门处理Jupyter笔记本的工具，这是本P的核心技术点之一。

### 01:54 - 02:02 接受重构成果

Claude Code开始处理任务，老师依次接受Python文件的更改。首先创建了数据加载辅助函数的Python文件，用于处理数据。然后创建了另一个包含指标计算的Python文件，用于最终的数据处理和展示。

在Claude Code处理这些任务的过程中，老师随时检查已创建的文件，确认面向对象的方法比直接嵌入所有内容在笔记本内部更整洁。老师看到业务指标计算的代码能够传入数据框并返回可视化和数据验证。

最终成果包括：

- 业务指标Python文件
- 重构后的笔记本
- requirements.txt
- README文件

**讲解意图**：通过实时查看生成的文件，帮助学生理解重构后的结构是如何实现职责分离的。

### 02:02 - 02:19 测试重构后的笔记本

老师查看重构后的笔记本，确认这些单元格能按预期运行。从视觉上看，笔记本整洁了很多。

初始单元格显示数据读取的提示，老师告知Claude无需读取提示，只需完成README。查看这个笔记本时，老师指出：

> “这整洁多了。”

重构后的笔记本包含：

- 数据分析和相关配置
- 可配置的比较年份设置
- 所需的数据加载
- 从自定义模块导入的功能

**关键改进**：老师特别提到现在可以指定比较年份，甚至可以按月筛选。这种灵活性是原始笔记本所缺乏的。

### 02:19 - 02:27 发现并修复错误

老师运行所有单元格后，发现了一个键错误（KeyError）。老师指出这个问题很快就能看出来，然后请Claude帮忙修复：

> “我运行某个单元格时出现键错误。我可以让Claude代码帮我解决。”

老师将出错的代码提供给Claude Code，以便提供更多上下文。Claude代码分析需要处理的内容，查看问题所在，并做出修改。

修复完成后，老师重新运行这些单元格，确认问题已解决，不再看到之前的错误。

**迭代优化**：这是用Claude Code处理实际问题的典型流程——执行→发现错误→反馈给AI→修复→验证。这个循环是后续操作的基础模式。

### 02:27 - 02:35 展示改进后的业务指标

老师向下滑动，查看改进后的业务指标展示。重构后的笔记本返回更友好的摘要显示，包括客户满意度信息，比之前更容易处理。

在收入分析部分，开始看到更清晰的数值、更多信息显示，以及更轻松的数据展示。可视化效果看起来更美观。类别表现分析有了更好的工具来分析类别和相关收入。地理表现分析中，各州的可视化效果不错，悬停可获取各州详细信息。

**阶段性总结**：从混乱的笔记本到结构化的代码，从不清晰的展示到美观易读的可视化，重构的价值在这里得到了验证。

### 02:35 - 02:43 引入Streamlit仪表盘

老师总结重构笔记本的过程：

> “重构笔记本以更直观展示信息并分离关注点。”

现在要进入下一步——从笔记本转为网页界面，使用Streamlit驱动的美观仪表盘。老师介绍Streamlit：

> “如果你不熟悉Streamlit，这是一个用于展示界面的Python库，特别是仪表盘，而且非常容易无缝过渡到这些Python文件。”

**场景转换**：从数据处理笔记本转向可交互的数据仪表盘，这是数据分析项目的常见需求。

### 02:43 - 02:54 创建仪表盘转换提示

老师创建一个新的Markdown文件，命名为“转换为仪表盘.md”，用于存放仪表盘创建的提示词。

老师再次展示提示策略：直接告诉Claude需求，而不是让Claude猜测。老师描述了仪表盘的具体要求：

**提示工程**：老师强调提前规划布局的好处——避免反复沟通，直接给出精确的需求。

**迭代调整**：这是用Claude Code的常见模式——AI生成的内容不总是完美的，需要人工介入调整方向。

### 03:02 - 03:15 启动Streamlit服务器

## 本P结论

1. **Claude Code的read notebook功能**：专门用于处理Jupyter笔记本的工具，支持探索单元格、分析结构、创建修改
2. **职责分离的重构价值**：将数据加载、指标计算、笔记本展示分离为独立模块，提高可测试性和可维护性
3. **Streamlit是数据分析的桥梁**：无缝衔接Python数据处理代码和可交互Web界面
4. **迭代优化是核心工作流**：生成→检查→发现问题→提出具体需求→修改→验证，这种循环比一次性完成更高效
5. **提示工程的关键习惯**：主动询问AI最佳提示方式、提前规划布局、给出具体而非模糊的需求

## 易错点

1. **长提示的显示限制**：粘贴较长的提示词时，Claude Code不会显示所有内容，这是终端显示特性而非问题
2. **数据年份配置**：2024年数据不完整，不适合作为同比分析的起点，默认年份应设为2023年
3. **AI生成内容需验证**：Claude Code生成的文件（如README）可能需要手动调整，不必完全接受所有输出
4. **依赖安装时机**：生成代码后需要手动安装依赖，Claude Code不自动处理运行环境配置
5. **README创建策略**：有时让AI修改现有文件而非创建新文件更高效，需要根据情况调整指令
