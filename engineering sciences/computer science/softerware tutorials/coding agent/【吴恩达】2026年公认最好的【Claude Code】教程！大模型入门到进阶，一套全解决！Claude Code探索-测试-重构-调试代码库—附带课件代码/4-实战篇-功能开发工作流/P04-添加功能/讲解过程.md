# P04 讲解过程 - 添加功能

- 所属章节：`4-4-添加功能`

- 分P主文档：`P04-添加功能.md`

## 本P定位

本P是课程第4章节“Claude Code探索-测试-重构-调试代码库”的第4个分P，承接P3对代码库基础结构的讲解，进入**实战开发阶段**——将两个具体功能需求落地到项目中。通过本P，学习者将掌握使用Claude Code进行功能开发的标准工作流程，包括文件引用策略、计划模式使用、前端界面构建、后端工具开发，以及MCP服务器集成等核心能力。

## 本P主线

本P围绕“为一个课程大纲生成应用添加功能”这一真实开发任务展开，包含两条并行主线：
- **前端主线**：构建带来源引用的界面显示 + 新增“开始新对话”按钮
- **后端主线**：开发“获取课程详情”工具，让用户能查看特定课程的课时列表

核心教学目标是展示Claude Code在**完整功能开发周期**中的使用方法，而非碎片化的命令演示。

## 讲解过程

### 00:00 - 00:52 开场引入与任务背景

老师首先回顾了学习路径：前几P已经讲解了聊天机器人代码库的基础知识。明确本P的目标——为界面添加功能并实现新工具。

**讲解意图**：建立前后关联，让学习者清楚本P在课程体系中的位置。老师强调接下来的任务不再是单纯学习命令，而是实际动手开发。

### 00:52 - 02:29 需求分析与功能规划

老师从用户场景出发讨论需要添加的功能。回溯之前应用中展示过的功能——生成课程大纲时返回的信息非常详细，包含引用来源。老师指出用户看到这些被引用的资料时，能够直接点击链接跳转到权威来源会非常方便。

**关键洞察**：老师将这个需求提炼为具体的开发目标——构建一个前后端正确配合的界面，能够渲染链接并显示数据来源，而不仅仅是纯文本展示。

**与后续内容的连接**：这个需求将指导后续所有的开发工作，最终验证标准就是用户能否看到可点击的链接并正确跳转。

### 02:29 - 04:30 Claude Code使用策略：文件引用与计划模式

老师开始讲解使用Claude Code进行功能开发时最重要的两个组件。

**组件一：正确引用文件**

老师指出Claude的代码质量取决于你提供的上下文。当你要求Claude修改代码时，必须确保它知道需要修改的正确文件。老师演示了两种方式：
- 使用`claude`命令（字幕中写为"clog代码"，疑似字幕识别错误，推断应为"claude"命令）让Claude自行分析
- 如果已知需要修改的文件，立即提供给Claude上下文

老师强调后者能让工具更高效——不要让Claude猜测，直接用@标签指定文件或文件夹，甚至可以使用Tab键补全文件名。

**组件二：计划模式**

老师介绍当需要构建功能或进行较大修改时，不应该直接使用Claude尝试分析需求并立即编写代码。正确做法是先进入规划模式，让Claude创建需要修改的具体计划，如果批准了计划，再让Claude修改多个文件。老师解释这个策略的价值：让Claude有机会在执行修改前分析需求，制定完整方案。

**操作演示**：老师按下了Shift+Tab两次（字幕显示"Shift+Tab两次"，推断为连续按两次），开启计划模式。

**讲解意图**：这是本P最核心的方法论输出。老师反复强调“先规划后执行”的工作流，这是使用Claude Code进行复杂开发的关键策略。

### 04:30 - 05:52 实际操作：构建带来源引用的界面

老师开始实际操作构建第一个功能——在界面中显示带来源的引用链接。

老师调用了一个提示词，要求Claude构建带来源引用的界面。在对话中老师演示了正确引用文件的操作——使用@标签引用正确文件，同时引用需要修改的其他文件，给Claude提供足够的上下文。

**操作演示**：
- 调用计划模式
- 提供完整的任务描述和文件上下文
- 让Claude制定解决方案
- Claude将阅读认为最必要的文件，从前端追踪到后端，理解需要实现的内容

**讲解意图**：让学习者看到完整的“人机协作”过程——不是简单的命令执行，而是需求表达、上下文提供、方案审批、修改执行的完整循环。

### 05:52 - 07:17 计划审批与自动接受模式

老师演示了计划制定完成后的审批流程。Claude完成计划制定后，老师可以批准或提供修改反馈。老师特别指出一个重要观察：Claude并未直接编写代码，而是生成了计划等待审批。

老师介绍可以使用**自动接受修改**功能（Enable auto-accept edits），这样Claude在执行计划时无需反复确认权限。老师解释这个功能的适用场景：用户通常开始时每次都要授权，随着对Claude代码的信任增加，自动接受修改会逐渐启用。

**操作演示**：
- 查看Claude生成的计划
- 判断计划是否符合预期
- 如果满意，按Shift+Tab启用自动接受
- 让Claude直接执行修改

**讲解意图**：展示Claude Code的权限管理机制，帮助学习者理解如何在“谨慎控制”和“效率优先”之间找到平衡。

### 07:17 - 09:04 文件修改与运行测试

老师演示Claude直接在必要文件中进行编码。代码库发生更改，由于启用了自动接受，无需重复授权。

**工作流程说明**：老师指出这是常见工作流——先让Claude生成计划，审批后让它自动执行修改。老师还特别处理了一个细节：当Claude准备启动服务器时，老师告知“服务器正在运行，无需自行启动”。老师解释这也可以记录在Claude.md文件中，告知Claude这一偏好。

**讲解意图**：展示如何与Claude进行有效的沟通和指令调整，同时介绍项目内存（Claude.md）作为持久化偏好的机制。

### 09:04 - 11:12 测试与视觉问题发现

老师回到浏览器测试第一个功能。请求课程大纲的概要，如果构建正确，应该能看到每个功能下方有相关链接。老师确实看到了链接，但发现界面存在问题——蓝色链接的颜色与背景对比度不够，视觉上难以阅读。

**问题识别**：老师敏锐地发现问题所在——链接使用默认蓝色，在当前界面中不够醒目。

**讲解意图**：展示真实的开发迭代过程——不是一次成型，而是需要不断发现问题并改进。

### 11:12 - 12:53 视觉优化：截图反馈法

老师决定继续让Claude Code进行视觉修改。这里老师展示了两种方式：
- 直接告诉Claude“我想要的”效果
- 使用Claude的视觉能力，查看截图并分析需要修改的部分

老师选择了第二种方式——截取当前界面的截图，粘贴到Claude Code中，告诉Claude“这些链接难以阅读，能否让视觉效果更吸引人”。

**操作演示**：
- 在浏览器中截取界面截图
- 切换回Claude Code窗口
- 粘贴截图
- 用自然语言描述问题和期望

**讲解意图**：展示Claude Code的视觉理解能力——不仅能处理文字，还能分析图片内容，这是快速迭代界面的强大方法。老师特别指出“截屏并展示给Claude Code看”是做视觉调整时的常用技巧。

### 12:53 - 14:07 验证修改效果

Claude分析截图后识别出问题（链接使用默认蓝色），并进行了修改。老师让Claude执行修改后，回到浏览器再次请求课程大纲概要，验证链接是否更易阅读。老师查看源文件确认改进效果，评价“太好了”（字幕"好多了"为口语化表达）。

**讲解意图**：展示“截图-分析-修改-验证”的快速迭代循环，让学习者看到这个工作流的实际效率。

### 14:07 - 16:20 新对话：上下文管理

老师做了一个重要的操作——从头开始，清空对话历史。老师解释这样做的原因：让上下文窗口更长，与Claude协作而不受其干扰，因为之前的对话可能被Claude见过但与当前构建无关的内容混淆。

**方法论输出**：老师明确指出“当我们想要构建新功能时”，应该考虑开启新的对话会话来保持上下文清洁。这是一个实用的工作流建议。

**讲解意图**：帮助学习者理解Claude Code的上下文管理策略——不是一直延续一个会话，而是根据任务需要合理创建新对话。

### 16:20 - 19:07 新对话按钮：完整功能开发演示

老师提出第二个功能需求——在界面中添加“开始新对话”按钮，这样用户就不需要每次刷新页面来开始新对话。老师希望在这里添加一个按钮，点击时能重置对话并重新开始。

**完整工作流演示**：

1. **启用计划模式**：调用计划模式并引入提示，描述需要构建的功能——在界面中添加新对话按钮，点击时清空聊天窗口中的对话内容，启动新会话并处理必要的清理工作。

2. **计划生成与审批**：Claude分析需要构建的内容，生成全面的计划。老师快速浏览计划，判断“这看起来不错”，批准计划。

3. **文件修改**：由于启用了自动接受编辑，Claude开始更新文件。需要修改的内容包括：
   - 为现有部分添加HTML结构
   - 添加处理点击事件的JavaScript代码
   - 处理新对话启动、清空输入并聚焦等逻辑
   - 添加必要的样式

4. **项目管理**：老师还把“不要用./run.sh启动服务器”写入Claude.md文件，作为项目记忆的一部分，供其他开发者参考。老师说明这是本地文件，会被git忽略，但对自己和团队有用。

**讲解意图**：完整展示了一个功能的开发周期——从需求到计划、从审批到执行、从代码到文档。这是本P最详细的实操演示。

**讲解意图**：自然过渡到下一个话题——MCP服务器的强大功能，同时展示真实迭代过程中的“不满意-改进”循环。

### 20:47 - 25:27 MCP服务器介绍与配置

老师介绍了一个强大的扩展机制——MCP（Model Context Protocol，模型上下文协议）。MCP允许像Claude Code这样的工具获取额外功能，连接外部数据源和系统。

老师决定添加一个Playwright MCP服务器，用于自动化浏览器操作——打开浏览器、截取截图、分析截图，而不需要手动截图并与Claude对话。让Claude全部自动完成这些操作。

**配置步骤**：
1. 退出当前Claude Code会话
2. 输入命令`claude mcp`添加MCP服务器
3. 指定服务器名称为`playwright`
4. 使用`npx playwright`启动服务器

老师提示对于任何MCP服务器，最新版本的命令可参考文档，所有MCP服务器都会指定连接命令。

**操作演示**：
- 重新打开Claude Code
- 使用`/mcp`命令查看已连接的MCP服务器
- 确认已成功连接到playwright MCP服务器
- 查看playwright服务器提供的可用工具列表

**可用工具示例**（基于上下文推断为Playwright MCP的标准工具集）：
- 执行JavaScript
- 上传文件
- 按下按键
- 进行导航
- 返回上一页
- 截取屏幕截图
- 页面分析

**讲解意图**：介绍MCP作为Claude Code的扩展生态，展示如何通过MCP让Claude Code获得更强的自动化能力。老师强调这些原本需要手动测试的操作，Playwright可以程序化执行。

## 本P结论

1. **Claude Code核心工作流**：文件引用（@标签）+ 计划模式（Shift+Tab）+ 自动接受修改，这三者的组合是高效使用Claude Code进行功能开发的关键。

2. **上下文管理策略**：构建新功能时建议清空对话历史，避免历史上下文干扰；使用Claude.md持久化项目偏好。

3. **视觉优化技巧**：截屏并展示给Claude Code是快速迭代界面的有效方法，比纯文字描述更直观。

4. **MCP扩展能力**：MCP（Model Context Protocol）允许Claude Code连接外部工具和数据源，Playwright MCP可以实现自动化的浏览器操作和UI测试。

5. **完整功能开发周期**：需求分析 → 计划制定 → 审批 → 执行 → 测试 → 迭代 → 文档记录，Claude Code可以辅助每个环节。

6. **后端工具开发**：工具不仅包含Python代码，还需要更新系统提示词、在RAG系统中注册，才能真正被LLM使用。

## 易错点

1. **跳过计划直接编写代码**：老师反复强调在进行较大修改时应先使用计划模式，让Claude分析需求并制定方案，而不是直接让Claude写代码。

2. **上下文不清晰**：当构建新功能时，不清空历史对话可能导致Claude被无关内容混淆，降低代码质量。

3. **文件引用缺失**：不提供必要的文件上下文，让Claude猜测需要修改的文件，既低效又容易出错。应该使用@标签主动引用相关文件。

4. **权限管理不当**：初期应谨慎使用自动接受修改功能，待建立信任后再启用。老师建议“用户通常开始时每次都要授权，随着信任增加逐渐启用”。

5. **服务器重复启动**：Claude可能不知道服务器已在运行，应在Claude.md中记录偏好或直接告知，避免重复启动。

6. **视觉描述不精确**：做界面调整时，纯文字描述可能不够精确。使用截屏并展示给Claude Code是更有效的方式。
