## 核心问题

如何用Claude Code完成一个功能的完整开发周期——从需求到计划、从审批到执行、从代码到系统集成。

## 最小知识骨架

1. **文件引用策略**：使用@标签直接指定要修改的文件/文件夹，不让Claude猜测
2. **计划模式**：Shift+Tab开启，先规划后执行，审批通过再修改
3. **自动接受修改**：逐步建立信任后启用，减少重复授权
4. **上下文管理**：构建新功能时清空对话历史，避免无关内容干扰
5. **项目记忆机制**：Claude.md持久化项目偏好（如服务器运行状态）
6. **截图反馈法**：截屏展示给Claude Code，比文字描述更精确
7. **MCP扩展协议**：Model Context Protocol连接外部工具（演示Playwright MCP）
8. **工具开发三步**：Python代码实现 → 系统提示更新 → RAG系统注册

## 关系与依赖

```
文件引用 → 高质量上下文
计划模式 → 审批 → 自动接受修改
新对话 → 清洁上下文
Claude.md → 持久化偏好

截图反馈 → 视觉迭代

MCP服务器 → 扩展Claude能力
后端工具 → 代码+系统提示+RAG注册
```

- 计划模式是自动接受修改的前置条件
- 文件引用质量决定上下文质量，进而影响代码输出
- MCP是能力扩展层，可叠加在任意工作流之上

## 输入 / 输出 / 前置 / 支撑

- **输入**：功能需求描述、目标文件上下文、截图（如需视觉调整）
- **输出**：功能代码、界面修改、系统配置更新
- **前置**：P3代码库基础结构认知、Claude Code基本操作
- **支撑**：Claude.md项目记忆、Shift+Tab计划模式、@文件引用、Playwright MCP

## 易错点

1. **跳过计划直接编码**：大修改应先规划，批准后再执行
2. **上下文污染**：新功能不清空历史，Claude被无关内容干扰
3. **文件引用缺失**：不使用@标签，让Claude猜测目标文件
4. **权限管理不当**：信任未建立时过早启用自动接受，或全程手动授权
5. **服务器重复启动**：Claude可能忽略服务器已运行状态，应在Claude.md记录
6. **视觉描述模糊**：纯文字描述界面问题，不如直接截图展示
