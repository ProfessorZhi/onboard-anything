# P9 9-基于Figma原型创建Web应用

## 本P定位

本P是整个Claude Code教程中展示AI与设计工具深度集成的核心实操环节。在此之前，学生已学习完Claude Code的基础使用（第1-8P），现在要进入高级应用场景——如何利用MCP（Model Context Protocol）服务器实现设计到代码的无缝转化。本P起到承前启后的作用，为后续可能的数据可视化、真实API集成等进阶功能奠定基础。

## 本P主线

从零开始，通过MCP服务器连接Figma设计与Claude Code开发环境，先用Mock数据快速生成可运行的Web应用界面，再用真实联邦储备经济数据替换静态数据，最终展示一条完整的“设计→开发→数据填充”工作流。

## 讲解过程

### 00:02 - 00:14 本P目标与定位

**讲了什么**：老师开门见山说明本节课的任务——连接Claude到Figma MCP服务器，将设计原型导入Claude Code并开发成可运行的应用。

**为什么这里讲**：作为本系列教程第9P，需要展示Claude Code的进阶能力，而Figma是业界最流行的设计工具，两者的结合能大幅提升开发效率，这是学生之前没有接触过的能力边界。

**和前后内容的连接**：前面P1-P8讲解了Claude Code基础操作，本P开始展示AI与外部工具的协同能力，为后续可能的数据API集成做铺垫。

### 00:14 - 00:26 为什么要用MCP服务器

**讲了什么**：老师强调使用MCP服务器的价值——快速创建应用、在多环境中协作、与Figma等工具协同工作，将原型快速转化为网页应用。

**为什么这里讲**：在动手操作前先讲清楚“所以然”，帮助学生理解这个工作流的效率优势，而不是机械地跟着敲命令。

**关键结论**：MCP服务器是Claude Code扩展能力的关键，让AI可以调用外部工具的API和功能。

### 00:26 - 00:66 工具准备与前置条件

**讲了什么**：详细说明本P需要的三个核心工具——Figma MCP服务器（用于读取设计）、Playwright MCP服务器（用于自动化测试和截图）、Figma账号及开发者模式。同时提到即使不想订阅Figma也有替代方案，以及开发者模式无需订阅Figma付费版也能启用。

**为什么这里讲**：动手前必须先确保环境就绪，避免学生在实际操作时因缺少前置条件而卡壳。

**关键结论**：开发者模式可以在不订阅Figma Pro的情况下启用，Figma MCP服务器提供5个工具用于获取底层代码和图像。

### 00:66 - 00:93 创建Next.js项目

**讲了什么**：老师演示使用命令行创建Next.js应用，采用默认配置选择，让npm自动安装依赖。强调即使之前没用过Next.js也没关系，只需几条命令即可完成。

**为什么这里讲**：Next.js是老师选择的现代技术栈（相比纯HTML更符合实际项目需求），需要先搭建好开发环境。

### 00:93 - 00:126 查看Figma原型

**讲了什么**：老师切换到Figma界面展示一个仪表盘原型——使用联邦储备经济数据的可视化界面。老师说明这个原型设计并不复杂，图层和组件较少，但足够作为开发起点。重点操作是：进入设置→启用开发者模式→选中MCP服务器→复制特定图层（因为后续配置需要用到）。

**为什么这里讲**：让学生先理解输入（Figma设计）的质量会影响输出（生成的代码），设计越详细，后续工作量越少。但同时也说明即使设计简陋，AI也能从有限信息中推断并生成可用的代码。

**关键结论**：原型的质量很大程度上决定了生成效果的下限，但即使简陋的设计也能作为良好起点。

### 00:126 - 00:175 MCP服务器配置

**讲了什么**：这是本P的核心操作环节。老师在Claude Code项目中配置两个MCP服务器。首先在项目配置文件中添加Figma MCP服务器命令，然后添加Playwright MCP服务器命令。使用`mcp__server__`相关命令在本地启动。配置完成后使用MCP命令验证两个服务器是否连接成功，看到Figma MCP服务器显示5个可用工具。

**为什么这里讲**：这是实现“设计→代码”转化的技术基础，必须确保MCP服务器正确配置才能让AI调用这些工具。

**操作要点**：强调每个项目需要单独配置MCP服务器，这不会自动应用到其他Claude Code项目，属于项目级别的独立配置。

**关键结论**：MCP服务器连接成功后，可以看到Figma MCP提供的工具——获取图像、获取代码、分析设计规范等。

## 本P结论

1. **MCP服务器是扩展AI能力的关键**：通过Figma MCP和Playwright MCP，Claude Code可以读取设计文件、自动生成代码、进行自动化测试，形成完整的“设计→开发→验证”工作流。

2. **Next.js + Recharts是现代数据可视化开发的常用技术栈**：老师选择这个组合具有代表性，学生可以举一反三到其他技术栈。

3. **设计的质量影响生成效果的下限**：Figma原型越详细、图层结构越规范，生成的代码质量越高，但这不是绝对门槛——简陋的设计也能生成可用的基础代码。

4. **从Mock到真实数据的路径已打通**：Claude可以帮助完成API文档研究、凭证配置、数据获取服务编写等全流程，数据可视化应用可以快速从原型进化到真实产品。

5. **每个项目需要单独配置MCP服务器**：这是Claude Code的项目级配置特性，不会全局生效。

6. **模型选择建议**：复杂任务可考虑Opus模型（如果使用专业计划），Sonnet也能良好完成大多数任务。

## 易错点

1. **开发者模式vs订阅**：学生可能误以为需要Figma Pro订阅才能使用开发者模式，实际上开发者模式无需订阅即可启用（来源：字幕直接提到）

2. **MCP服务器是项目级配置**：每次新建Claude Code项目都需要重新配置MCP服务器，这个配置不会自动应用到其他项目（来源：字幕直接提到）

3. **设计质量决定下限**：如果Figma原型过于简陋，生成的代码可能需要更多手动调整，但仍是良好起点（来源：字幕直接提到）

4. **模型选择影响效果**：Sonnet可以完成大多数任务，但Opus在复杂任务上表现更优，学生应根据实际需求选择（来源：字幕直接提到）

5. **API密钥的必要性**：使用联邦储备API需要API密钥，学生需要知道如何获取和配置（来源：字幕直接提到）

6. **依赖安装时机**：Claude会自动提示并安装必要的npm依赖（如recharts），学生不要忽略这些提示（来源：字幕直接提到）

7. **服务器重复启动**：如果开发服务器已在运行，不要再次启动，避免端口冲突（来源：字幕直接提到）
