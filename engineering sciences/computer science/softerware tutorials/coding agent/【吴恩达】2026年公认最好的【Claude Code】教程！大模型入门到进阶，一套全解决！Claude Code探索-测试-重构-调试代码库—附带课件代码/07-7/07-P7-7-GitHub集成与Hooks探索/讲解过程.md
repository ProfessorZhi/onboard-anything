# P7 7-GitHub集成与Hooks探索

## 本P定位

本P是Claude Code进阶应用的第三部分，承接P6的终端内操作扩展到**GitHub平台上的集成能力**。课程从清理上节遗留问题开始，逐步引导至GitHub App安装、工作流配置、拉取请求审查、Issue驱动修复，最后引入Claude Code Hooks这一新发布的功能特性。起到从“终端操作”向“平台化工作流”过渡的桥梁作用。

## 本P主线

1. 清理上节遗留的worktrees文件夹并推送到GitHub
2. 安装并配置Claude Code的GitHub集成（App + 工作流）

### 00:466.5 - 00:537.0 Claude Code Hooks概念介绍

**为什么讲这个**：这是最近发布的新功能，需要专门讲解其概念和使用场景。

**Hooks的核心思想**：
- 在Claude的不同操作节点注入自定义代码
- 可在工具执行前（pre-tool）触发
- 可在工具执行后（post-tool）触发
- 甚至可以阻止工具执行

**可接入的事件类型**：
- pre_tool_use：工具执行前
- post_tool_use：工具执行后（可发送通知、执行操作）
- on_tool_use：工具使用时
- on_user_message：用户提交提示时
- on_task_stop：任务停止时
- on_sub_agent_response：子代理完成响应前

老师强调安全注意事项：如果你在运行任意shell命令，必须非常注意你的操作。

**配置位置**：
- 在项目的本地设置JSON文件中配置
- 可以在权限配置附近找到Hooks配置入口

### 00:542.3 - 00:631.7 Hooks实战演示

**为什么讲这个**：用具体例子说明Hooks如何配置和使用。

**示例场景**：创建一个post-tool-use hook，当执行读取或grep操作时，触发一个shell命令让计算机朗读"完成"。

**hook配置结构**：
- hook名称：host_tool_use
- matcher部分：指定触发条件（读取操作或grep操作）
- 执行命令：say命令（调用计算机音频朗读文本）

**配置步骤**：
1. 在Claude Code中输入/hooks命令
2. 进入管理工具事件的配置界面
3. 在matcher中添加规则：匹配ReadTool或grep工具
4. 配置执行命令：使用say命令朗读"完成"
5. 配置写入本地settings.json文件

**测试Hooks**：
1. 退出Claude Code
2. 重新打开
3. 执行读取文件操作（如运行.h文件）
4. 当读取完成后，计算机会朗读"完成"

**应用场景扩展**：
- 执行代码后运行测试（run tests）
- 运行代码检查器（run linters/checkers）
- 在不需要时阻止使用工具
- 在特定情况下让Claude进行自我审查
- 发送通知等

老师提到还有更多功能即将推出，务必查看文档了解所有可用功能。也可以让Claude Code为你编写钩子并相应修改。

## 易错点

1. **忽略身份验证步骤**：安装GitHub集成时可能需要额外的CLI身份验证，如看到设置说明必须遵循
2. **Hooks安全风险**：运行shell命令的Hooks必须非常注意操作，避免执行危险命令
3. **Hook配置位置混淆**：需要在项目的本地settings.json中配置，而非全局或用户级配置
4. **自我审查的必要性**：尽管信任Claude，让另一个Claude实例复核工作也是有价值的实践
5. **PR审查配置**：默认设置可能需要根据团队需求调整，包括审查深度、提示内容等
6. **推送前清理**：从上节延续的习惯——合并代码后记得推送到GitHub，避免遗留问题到下一节
