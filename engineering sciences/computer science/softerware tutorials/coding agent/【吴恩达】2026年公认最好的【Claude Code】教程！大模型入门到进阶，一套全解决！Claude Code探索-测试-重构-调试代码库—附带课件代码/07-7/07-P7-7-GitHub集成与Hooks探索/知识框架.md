# P7 7-GitHub集成与Hooks探索 知识框架

## 本P位置

课程第7P，Claude Code进阶应用的第三部分。前接P6（终端内工作树和分支管理），后接P8（Jupyter笔记本可视化）。处于从终端操作向平台化工作流过渡的桥梁位置，是技术栈递进的关键节点。

## 核心问题

如何将Claude Code能力从终端扩展到GitHub平台实现协作化工作流，以及如何通过Hooks机制在工具执行过程中注入自定义自动化逻辑。

## 最小知识骨架

1. **GitHub App安装与身份验证** — Claude Code接入GitHub的入口，需要完成App安装和CLI身份验证两步
2. **GitHub Actions工作流配置** — 通过YAML配置文件定义自动审查等行为的触发条件和执行逻辑
3. **PR自动代码审查机制** — GitHub Actions驱动的Claude分析PR变更、检查质量、识别安全问题
4. **Issue驱动开发流程** — 在Issue中@提到Claude，触发其分析问题、生成PR的完整闭环
5. **Claude Code Hooks事件模型** — pre_tool_use / post_tool_use / on_tool_use / on_user_message / on_task_stop / on_sub_agent_response
6. **Hooks配置与matcher规则** — 在settings.json中定义触发条件和执行命令的对应关系

## 关系与依赖

```
GitHub App安装
       ↓
身份验证完成 ← → CLI身份验证（可能需要）
       ↓
工作流安装（code_review.yml + Actions YAML）
       ↓
仓库级访问令牌创建
       ↓
      ┌───────────────────────────────┐
      ↓                               ↓
PR自动审查                     Issue驱动开发
（需GitHub Actions）           （需App + @提及）
      ↓                               ↓
审查结果 → 合并决策                分析 → 计划 → PR生成 → 自我审查
```

Hooks独立于GitHub集成存在，两者是并行扩展路径。Hooks可作用于GitHub集成中的工具执行，形成叠加效果。

## 输入 / 输出 / 前置 / 支撑

**前置对象**：GitHub账号、可访问的仓库、P6遗留的代码上下文、终端Git状态

**输入对象**：GitHub Issue内容、PR创建请求、用户触发的工具调用、Hook触发事件

**输出对象**：推送到origin main的代码变更、PR描述与审查结果、代码审查报告、Hook执行后的side-effect（如TTS朗读）

**支撑对象**：本地settings.json（Hooks配置）、code_review.yml（审查配置）、GitHub Actions运行时环境、Claude Code SDK（集成底层）

## 资源与外部对象

**平台与工具**：Claude Code GitHub App（集成入口）、GitHub Actions（自动化引擎）、Claude Code SDK（集成底层）、VS Code（前端预览）

**配置文件**：本地settings.json（项目级Hooks配置）、code_review.yml（审查行为配置）、Actions YAML（CI/CD流程定义）

**系统命令**：git push origin main（推送代码）、say命令（macOS TTS朗读）、shell命令（Hook中可执行）

**外部资源**：Claude Code官方文档（Hooks最新功能）、后续P8的Jupyter笔记本环境

## 易错点

身份验证步骤遗漏导致GitHub集成失败，安装App后还需检查CLI端是否需要额外授权。Hooks配置位置必须在项目本地settings.json而非全局或用户级配置。运行shell命令的Hooks存在安全风险，必须确认命令来源可靠。Hook matcher规则配置错误会导致触发条件失效，需精确指定工具名称。PR审查深度由提示内容决定，默认设置可能不符合团队需求。Claude生成的PR需自我审查，不可盲目信任。合并代码后未推送会导致遗留问题延续到下一节。
