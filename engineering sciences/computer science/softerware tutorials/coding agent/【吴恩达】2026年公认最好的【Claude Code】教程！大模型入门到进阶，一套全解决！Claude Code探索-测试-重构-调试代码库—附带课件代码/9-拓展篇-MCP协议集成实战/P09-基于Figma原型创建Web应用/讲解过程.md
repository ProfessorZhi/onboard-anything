## 本P定位

本P是课程从基础操作（Claude Code 核心功能）转向**高级集成与实际应用**的过渡环节。前面若干分P已覆盖 Claude Code 的安装、启动、代码编辑、Git 操作、测试等独立技能；本P则首次引入 **MCP（Model Context Protocol）服务器** 作为连接外部工具的核心机制，展示如何将 Claude Code 与 Figma、Playwright 两个外部服务打通，形成"设计 → 生成 → 验证"的完整闭环工作流。同时，本P也通过 Next.js 技术栈的引入，为后续分P（若有）演示了真实项目的起点搭建方式。

## 本P主线

1. **引入问题**：手动从设计稿到代码耗时，能否让 AI 直接"看"设计稿并生成应用？
2. **工具组合**：Figma MCP Server（读取设计） + Playwright MCP Server（自动化验证） + Claude Code（代码生成） + Next.js（目标框架）
3. **操作流程**：创建 Next.js 项目 → 配置两个 MCP 服务器 → 让 Claude 分析 Figma 原型 → Claude 生成代码 → Playwright 截图验证 → 用真实经济数据（联邦储备数据）替换模拟数据 → 最终效果确认
4. **核心观点**：Figma 原型的质量决定生成效果的下限，但即使原型简陋，AI 也能快速迭代出一个可用起点

## 本P结论

1. **Figma MCP Server + Playwright MCP Server + Claude Code** 构成了一套从设计稿到可运行应用的自动化流水线
2. Next.js 是本 P 演示的目标框架，App Router 架构下的代码主要在 `app` 文件夹中组织
3. MCP 服务器配置需要在每个独立项目中单独添加，不会跨项目自动生效
4. Figma 原型的质量直接决定 AI 生成效果的下限；但即使原型粗糙，AI 也能快速迭代出一个可用起点
5. Figma 的开发者模式（Developer Mode）必须启用才能让 MCP 服务器正常工作
6. FRED（Federal Reserve Economic Data）API 可用于获取真实经济数据，但需要 API 密钥
7. 在极短时间内（原本可能需要几天完成的任务），借助 MCP 服务器可以从 mock 版本推进到包含真实数据的可用应用

## 易错点

1. **MCP 配置不跨项目**：MCP 服务器配置在独立项目/独立环境中，需要为每个 Claude Code 项目单独配置，不能期望全局生效（老师特别强调）
2. **Figma 开发者模式未开启**：如果 Figma 未启用开发者模式，MCP 服务器无法正常工作（老师提前提示此步骤）
3. **Figma 原型质量决定下限**：原型过于简陋（如图层很少、结构不清晰），AI 生成的效果也会有明显局限；需要事先花时间完善 Figma 设计
4. **API 密钥缺失**：获取真实联邦储备数据需要 FRED API 密钥，如果没有密钥则只能停留在模拟数据阶段
5. **模型选择影响复杂任务效果**：Sonnet 可以完成大多数任务，但 Opus 在复杂多步骤任务中表现更优；在没有 Pro Plan 时可考虑切换模型
6. **开发服务器重复启动**：如果服务器已经在运行，不要重复启动；需要主动告知 Claude 当前状态（老师实际操作中的干预点）
