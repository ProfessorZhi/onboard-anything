## 本P定位

本P是课程第五课的核心实践环节，承接上一P快速上手代码库实现新功能的经验，进入**系统化调试与重构**阶段。课程目标从"会用工具"升级为"建立工程化开发流程"，涵盖测试驱动调试（TDD式调试）和多轮工具调用的架构重构。

## 本P主线

```
发现问题 → 编写测试 → 运行测试定位Bug → 修复Bug → 重构AI生成器支持多轮对话 → 验证重构效果
```

本P有两条并行线索：
- **明线**：修复聊天机器人查询失败的问题

**关键概念：**

3. 需要升级架构支持多轮对话

## 本P结论

1. **测试驱动调试优于无序尝试**：面对错误时，先编写测试定位问题，再修复并验证，比直接复制错误到AI更可靠

2. **Bug根因**：向量搜索的`max_results`参数被错误设置为0，导致返回0个结果块

3. **架构重构**：AI生成器从单次工具调用升级为支持多轮迭代调用，可处理复杂查询

4. **Claude Code高级用法**：
   - 扩展思考模式处理复杂任务
   - 计划模式确保修改前确认
   - 子代理并行探索多种方案
   - 测试编写→运行→修复的完整闭环

5. **工程化开发**：通过测试建立坚实基础，确保后续修改不导致回归

## 易错点

1. **直接复制错误到AI**：这是老师明确反对的做法，会导致无意义的追逐而非真正理解问题

2. **跳过测试直接修复**：不编写测试就修复，无法确保修复正确，也无法防止回归

3. **忽视测试的基础设施价值**：测试不仅是"验证这次修复"，更是"保障未来修改不破坏现有功能"

4. **单次调用思维**：对复杂查询（如跨课程比较）需要多轮工具调用，单次调用架构无法支持

5. **重构不验证向后兼容**：修改AI生成器时可能破坏现有API端点或RAG系统，需要测试保障

6. **自动接受所有更改**：在复杂重构时，关闭自动接受，先看计划再执行，避免破坏性修改

7. **单一方案思维**：复杂重构任务可以让Claude探索多种方案，由人类选择而非AI单方面决定
