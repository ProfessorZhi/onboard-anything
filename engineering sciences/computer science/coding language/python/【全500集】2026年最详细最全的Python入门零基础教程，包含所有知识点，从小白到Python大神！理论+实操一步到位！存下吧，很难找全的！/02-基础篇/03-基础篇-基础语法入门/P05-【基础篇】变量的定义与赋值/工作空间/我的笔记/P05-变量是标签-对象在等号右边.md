# P05 核心理解：变量是标签，对象在等号右边

## 一句话结论

`变量不是盒子，变量是名字；赋值不是改名字本身，而是让名字绑定到某个对象。`

---

## 这一节真正要解决什么问题

P05 真正要解决的不是“记住几个命名规则”，而是：

1. 变量到底是什么
2. `=` 在 Python 里到底是什么意思
3. 为什么有时“改了 a，b 也会变”，有时不会

---

## 最小骨架

### 1. 变量是名字，不是对象本身

```python
x = 10
```

这里更接近真实的理解是：

```text
名字 x  ─────→  对象 10
```

`x` 只是一个名字，真正的数据在右边的对象里。

### 2. 赋值是绑定

`x = 10` 的含义不是“x 永远等于 10”，而是：

`把名字 x 绑定到对象 10`

### 3. 多个名字可以绑定同一个对象

```python
a = 10
b = a
```

此时可以理解成：

```text
a ───→ 10
b ───→ 10
```

### 4. 重新赋值通常是重新绑定

```python
a = 10
b = a
a = 20
```

最后：

- `a` 是 `20`
- `b` 还是 `10`

因为发生的是：

- `a` 原来绑定到 `10`
- 后来 `a` 改为绑定 `20`
- `b` 仍然绑定原来的 `10`

---

## 最重要的分界：重新绑定 vs 原地修改

这是 P05 最值钱的部分。

### 情况一：重新绑定

```python
a = [1, 2]
b = a
a = [3, 4]
```

结果：

- `a == [3, 4]`
- `b == [1, 2]`

原因：

这里不是改原来的列表，而是 `a` 重新绑定到了一个新列表对象。

### 情况二：原地修改

```python
a = [1, 2]
b = a
b.append(5)
```

结果：

- `a == [1, 2, 5]`
- `b == [1, 2, 5]`

原因：

`a` 和 `b` 本来就绑定同一个列表对象，`append()` 是对原对象做修改。

---

## 可变对象 vs 不可变对象

### 不可变对象

常见例子：

- `int`
- `str`
- `tuple`

对它们来说，很多看起来像“修改”的操作，本质上是创建新对象再重新绑定。

```python
a = "hi"
b = a
a = a + "!"
```

最后：

- `a == "hi!"`
- `b == "hi"`

因为字符串不可变，`a = a + "!"` 不是改原字符串，而是让 `a` 绑定到一个新字符串。

### 可变对象

常见例子：

- `list`
- `dict`
- `set`

对它们来说，很多操作会直接改原对象本身。

```python
a = [1, 2]
b = a
a.append(3)
```

最后 `a` 和 `b` 都看到同一个被改过的列表。

## 命名规则：现在先记最关键的

变量名：

- 只能用字母、数字、下划线
- 不能以数字开头
- 不能用关键字
- 区分大小写

例如：

- 合法：`user_name`
- 不合法：`2nd_place`

---

## 一组必须会解释的例子

### 例 1

```python
x = 10
y = x
x = 20
```

解释：

`x` 重新绑定到 `20`，`y` 还绑定在原来的 `10`。

### 例 2

```python
x = [10]
y = x
x.append(11)
```

解释：

`x` 和 `y` 指向同一个列表对象，`append` 是原地修改，所以两个名字都看到变化。

## 当前阶段的教学近似

为了学懂 P05，你可以先这样理解：

`变量名像标签，贴在对象上。`

这个说法在入门阶段非常有用，因为它能解释：

- 多个变量指向同一个对象
- 重新赋值为什么不会自动连带另一个变量
- 可变对象和不可变对象为什么表现不同

但它是教学近似，不是底层完整实现。

更严格一点说：

- 变量名存在于命名空间中
- 命名空间维护“名字 → 对象”的绑定关系
- 在 CPython 里，对象底层确实有内存结构

## 本节最终总结

1. 变量是名字，不是盒子。
2. 赋值的本质是绑定对象。
3. 多个名字可以绑定同一个对象。
4. 重新绑定和原地修改不是一回事。
5. 可变对象和不可变对象会让“多个名字绑定同一个对象”的后果完全不同。

### `PyLong_FromLong(10)` 返回的是什么？

```c
PyLongObject *p = PyLong_FromLong(10);   // 返回 PyLongObject*，指向新创建的整数对象
```

```
p 指向的内存地址 = X（比如 0x1000）

地址 X 处开始的完整结构：
X ───→ ┌──────────────────────────────────────┐
        │ ob_refcnt: 1      ← p->base.ob_refcnt
        │ ob_type: 指向int   ← p->base.ob_type
        │ ob_digit: [10]    ← p->ob_digit[0]
        └──────────────────────────────────────┘

p 本身是一个指针变量（存储在栈上），存的值是地址 X。

p->base.ob_refcnt 读到的就是地址 X 处的第一块内容。
```

**为什么返回 PyLongObject\* 而不是 PyObject\*？**
因为 `PyLong_FromLong` 需要返回完整的整数对象（包含 ob_digit）。返回 `PyObject*` 也可以，只是调用方需要自己强转。

```c
PyObject *obj = (PyObject *)PyLong_FromLong(10);   // 强转也可以，因为PyLongObject开头就是PyObject
```

**关键理解：PyLong_FromLong 返回一个指针，指向堆上新分配的 PyLongObject 结构体。p 存的是这个结构体在内存中的地址。**

### ob_type 指针指向的是什么？

`ob_type` 指向一个 `PyTypeObject` 结构体，里面存着这个类型的**说明书**——告诉 Python 这个类型能做什么操作。

```c
PyTypeObject *ob_type;   // 指向类型对象
```

```
PyTypeObject 里存的是什么？（简化版）

┌─────────────────────────────────────┐
│ tp_name     = "int"                 │  ← 类型名字
│ tp_add     = long_add               │  ← 加法函数
│ tp_mult    = long_mul               │  ← 乘法函数
│ tp_str     = long_to_str            │  ← 转字符串函数
│ tp_hash    = long_hash              │  ← 哈希函数
│ tp_as_number = &number_methods      │  ← 数学运算表
│ ...还有几十个字段                    │
└─────────────────────────────────────┘
```

### 为什么需要 ob_type 指针？

**场景：Python 的加法 `a + b`**

```python
a = 10
b = 20
c = a + b    # Python 怎么知道用整数加法而不是字符串加法？
```

Python 运行时查表：

```c
// 伪代码：Python 执行 a + b 的过程

// 1. 从 a 拿到类型对象
PyTypeObject *type_a = a->ob_type;      // 指向 int 的 PyTypeObject

// 2. 从类型对象里拿到"加法操作"的函数地址
binaryfunc add_func = type_a->tp_as_number->nb_add;

// 3. 调用这个函数
PyObject *c = add_func(a, b);           // 执行 long_add
```

```
a + b 的查找链：

a->ob_type ──────────→ [ PyTypeObject: "int" ]
                              │
                              ├── tp_as_number->nb_add = long_add    ← 加法
                              ├── tp_str = long_to_str               ← 转字符串
                              └── ...
```

**这就是多态的底层实现**：同一行代码 `a + b`，通过 `ob_type` 找到当前类型对应的加法实现。

### ob_type 为什么是指针而不是整数ID？

如果用整数存类型：

```c
// 假设用 int 存类型
int ob_type_id;   // 0=int, 1=str, 2=list...
```

那每次做运算时都要写 if-else：

```c
if (ob_type_id == 0) {
    // int 加法
} else if (ob_type_id == 1) {
    // str 加法
} else if (ob_type_id == 2) {
    // list 加法
} // ...几十种类型，每种都要写
```

用指针指向 PyTypeObject 的话：

```c
// 不管什么类型，都是同一行代码
binaryfunc add_func = a->ob_type->tp_as_number->nb_add;
add_func(a, b);   // 任何类型都能用同一行调用
```

### 结构体嵌入技巧：PyObject base 放最前面

每个具体类型（int、float、str...）的结构体，把 PyObject 作为**第一个成员**嵌入：

```c
// int 对象的 C 结构
typedef struct {
    PyObject base;              // ★ 嵌入基结构体（放最前面！）
    digit ob_digit[1];          // int 自己的值
} PyLongObject;
```

**为什么放最前面？** 因为 C 语言允许把结构体 A 的指针强转成结构体 B 的指针，只要 B 的前面和 A 一样。

```c
PyLongObject *p = PyLong_FromLong(10);   // p 指向地址 X

// 假设 X = 0x1000，内存布局：
// 0x1000: [ob_refcnt][ob_type][ob_digit]

// 强转成 PyObject*
PyObject *obj = (PyObject *)p;
// obj 存的值 = X = 0x1000
// obj 指向的内存起始位置 = 0x1000
```

```
obj 指向 0x1000：

地址 0x1000 开始：
┌──────────────────────────────┐
│  ob_refcnt  │  ob_type       │  ← obj 只能"看到"这16字节
│    4字节     │     8字节       │
└──────────────────────────────┘
         ↑                    ↑
         obj->ob_refcnt      obj->ob_type
```

**核心：obj 只是换了个"看法"，实际内存完全没变。** 因为 PyLongObject 的开头刚好和 PyObject 一样，所以 `PyObject*` 能正确读取前16字节。

### 为什么必须放第一个？

如果 PyLongObject 改成这样：

```c
typedef struct {
    digit ob_digit[1];       // 放前面了！
    PyObject base;           // 放后面
} PyLongObjectBad;
```

内存布局：

```
┌─────────────────────────────────────────────────────────────┐
│  ob_digit[1]  │  base.ob_refcnt  │  base.ob_type         │
└─────────────────────────────────────────────────────────────┘
↑
PyLongObjectBad 起始地址 Y
```

强转后：

```c
PyObject *obj = (PyObject *)p;  // p 指向 Y
// obj->ob_refcnt 读到的是 ob_digit 的值！完全错了！
```

**所以必须 PyObject base 放结构体第一个，这样任何对象的起始地址都能当 PyObject* 用。**

### digit 是什么？

`digit` 是 C 语言的 `typedef`，不是对象，是 **30 位无符号整数**：

```c
// CPython 源码里：
typedef uint32_t digit;   // digit = 32位无符号整数（实际只用低30位）
```

Python 的 int 是**变长整数**，用多个 digit 数组拼接表示任意大的数：

```
x = 10：

┌────────────┬─────────────┬──────────┐
│ ob_refcnt  │  ob_type   │ digit[0] │
│    1       │  指向long   │   [10]   │
└────────────┴─────────────┴──────────┘
```

```
x = 2^100（需要4个digit）：

┌─────────────────────────────────────────────────────────────┐
│ ob_refcnt │ ob_type │digit[0]│digit[1]│digit[2]│digit[3]│
│     1     │  long   │    0   │757867775│5421010862│  1  │
└─────────────────────────────────────────────────────────────┘
```

**小数字用 1 个 digit，大数字用多个 digit 拼接。每个 digit 最大约 10 亿（2^30），Python 用类似"大整数乘法"的思路把多个 digit 拼起来表示超大整数。**

x = 10 的完整内存：

变量 x ──────→ PyLongObject
                  │
                  ├── ob_refcnt = 1
                  ├── ob_type ──────────────────┐

                  └── ob_digit = [10]    [ PyTypeObject: "int"说明书 ]

## Python 为什么比 C 慢？

### 核心原因：间接寻址 + 元数据开销

C 语言直接操作内存：
```c
int x = 10;
printf("%d", x);        // 直接从 x 的内存地址取值
```

Python 每次访问都要多跳转一次：
```python
x = 10
print(x)                # x 是 PyObject* 指针
                        # 先取 x 的值（地址）
                        # 再通过地址访问对象
                        # 取对象的 ob_fval 或 ob_digit
```

**C 是一次寻址，Python 是两次寻址。**

### 开销分解

| 开销来源 | 说明 | C 有吗？ |
|---------|------|---------|
| **间接寻址** | 变量 → 指针 → 对象，每次多跳转一次 | 无 |
| **对象头** | 每个对象额外 12-16 字节（refcnt + type + 元数据） | 无（C 的 int 直接就是 4 字节） |
| **类型指针** | 动态类型，运行时查表确认类型 | 无（C 编译时类型已知） |
| **引用计数维护** | 每次赋值 ±1，高并发时原子操作开销 | 无 |
| **内存分配** | `PyLong_FromLong(10)` 调用 malloc，碎片化严重 | 手动 `malloc`，但更直接 |
| **垃圾回收** | 循环引用需要 GC 扫描（虽然有分代优化） | 无 |

### 具体数字对比

| 操作 | C 耗时 | Python 耗时 |
|------|--------|------------|
| `int x = 10` | 1 次写内存 | 1 次 malloc + 对象构造 + refcnt 初始化 |
| `x + y` | 1 次 CPU 指令 | 2 次指针解引用 + 类型检查 + dispatch + 数值计算 |
| `a[i]` | 1 次基址 + 偏移 | 1 次指针解引用 + bounds check + 类型检查 + 偏移计算 |

### 慢 5-100 倍是正常的

Python 比 C 慢 **5-100 倍**，原因就是上面这些开销的叠加：
- 基础运算（加减乘除）→ 约 10-50 倍慢
- 函数调用 → 约 5-20 倍慢（因为每个对象都要构造/销毁）
- 面向对象特性 → 约 5-10 倍慢（方法查找、继承）

### Python 快的地方

Python 也不是全面落后：

| 场景 | Python 优势 | 原因 |
|------|------------|------|
| 开发速度 | 快 5-10 倍 | 无需声明类型、自动内存管理 |
| 字符串操作 | 接近 C | Python 字符串内部就是 C 实现 |
| 列表切片 | 接近 C | 内部是 C 数组 |
| 字典查找 | 比 C 快（对小数据） | 用了精巧的哈希算法 |
| 运行时反射 | 独特能力 | C 没有的动态特性 |

### 为什么不用 C 那套直接内存？

C 那套**没有对象概念**，所以快：

```c
// C 的做法
int x = 10;           // 内存: [10]
int *p = &x;          // p 存的是 x 的地址
*p = 20;              // 直接改 x 的内存

// Python 的做法
x = 10                 // x 是指针，指向堆上的 PyLongObject{10}
x = 20                 // 重新分配一个新 PyLongObject{20}，x 指向它
```

**Python 的"对象"是必须的代价**：没有对象头就没有引用计数、没有类型信息、没有自动 GC、没有多态。

Python 选择的是**开发速度 + 安全性**换取**运行时性能**。

---

## C语言底层对比

| | C | Python |
|--|---|---|
| `p` 是什么 | 存地址的整数 | `PyObject*` 指针变量 |
| `*p = 10` | 直接改内存 | 调用 `PyLong_FromLong` 申请堆内存 |
| 内存释放 | 手动 `free` | 引用计数自动归零即释放 |
| 类型信息 | 编译时确定 | `ob_type` 指针运行时记录 |
| 数据存储 | 直接存值 | 通过指针间接访问 |

**Python多出来的东西（代价）：**
1. **引用计数字段** — 自动内存管理
2. **类型指针** — 实现动态类型
3. **垃圾回收器** — 处理循环引用

**Python选择用运行时开销换开发效率。**

---

## 一句话总结

> **Python的"变量是标签"只是对C指针的高级包装。底层x就是一个`PyObject*`指针，指向堆上的对象结构体，对象里有引用计数+类型+数据。每次访问都要多跳转一次，加上对象头的元数据开销和引用计数维护，这就是Python比C慢5-100倍的原因。**

## 变量赋值的三步曲

```python
x = 10
```

1. **创建对象**：在内存中创建 int 对象 10
2. **绑定标签**：将 x 这个名字标签贴在对象 10 上
3. **后续赋值**：撕下 x 标签，贴到新对象上（旧对象失去标签）

**变量赋值操作的是标签，不是对象本身。**
