# P31 知识框架 - 【基础篇】包
## 本P位置

Python基础教程 → 模块系统 → 从模块升级到包

前置：P30 模块。包是模块的上一级组织单元，模块知识直接迁移。

---

## 核心问题

- 多个模块如何有序组织？
- 包如何被Python识别？
- 如何控制包内模块的导出范围？

---

## 知识骨架

1. **包 = 含 __init__.py 的文件夹**，用于收纳多个相关模块
2. **__init__.py** 三作用：标志包身份、控制导入范围（__all__）、执行轻量初始化
3. **三种导入路径**：精确路径（import 包.模块）、简化路径（from 包 import 模块）、批量导入（from 包 import *）
4. **__all__ 列表**：控制 from...import * 的可见范围，可写在包的 __init__.py 或模块内

---

## 关系与依赖

```
模块 ──→ 组成 ──→ 包（包含 __init__.py）
                    │
                    ├─── __all__ 控制导出 ──→ 导入方式配合
                    │
                    └─── 可嵌套 → 包.子包.模块

模块（add/sub函数） ← 被导入 ← 导入语句
         ↑  __all__ 控制可见性
```

---

## 输入 / 输出 / 前置 / 支撑

- **前置**：模块概念（模块名、导入语句基础）
- **输入**：.py 模块文件
- **输出**：可调用的模块或模块内的功能（函数、变量、类）
- **支撑**：__init__.py 是包被识别为包的关键文件

---

## 易错点

- 包名与内置模块或第三方库同名 → 冲突，覆盖原模块
- __all__ 只控制 from...import *，不控制其他导入方式
- 忘记 __init__.py → Python 不识别该目录为包
- __init__.py 写复杂逻辑 → 每次导入都执行，影响性能
- 嵌套层级过深（超过3层）→ 路径长、维护难

---

## 外部资源与对象

| 资源 | 角色 |
|------|------|
| random（标准库） | 示例：random 模块内部使用 __all__ 控制 from random import * 的导出内容 |
| requests（第三方库） | 反面示例：自定义包名不能叫 requests，否则覆盖同名第三方库 |

---

## 命名规范速查

| 元素 | 规则 |
|------|------|
| 包名/模块名 | 字母、数字、下划线；不以数字开头；不与内置/第三方库同名 |
| __init__.py | 包内必须存在（Python 3.3+ 可省略但建议保留） |
| __all__ | 列表类型，写在 __init__.py 或模块文件顶部 |
