# P29 讲解过程 - 【基础篇】抛出自定义异常
## 本P定位

本P是异常处理章节的第二小节，承接上一节对 Python 内置异常（ValueError、TypeError、ZeroDivisionError 等）的介绍，引入自定义异常的创建与抛出机制。本P的核心目标是让学习者理解为什么需要自定义异常，掌握自定义异常的基本实现步骤，并通过银行取款场景完整演示异常定义→抛出→捕获的全流程。

---

## 讲解过程

### 一、引入：内置异常的局限性

课程开头先回顾上一节内容，指出我们已经学会了处理 Python 自带的一些内置异常。随后话锋一转，引出核心问题：在实际编程中，有很多跟业务场景相关的错误，用内置异常来描述可能不够精准。

老师列举了两个典型业务系统的例子来说明：

- **银行系统**：余额不足、账户冻结
- **电商系统**：库存不足、订单已取消

这些都是特定业务相关的错误，而内置异常只能给出模糊的通用提示，无法直接说明具体问题。因此需要学习抛出自定义异常，让错误处理更加贴合实际需求，错误提示能够精准直达问题核心。

### 二、为什么需要自定义异常

老师提出三个核心原因：

**第一，能够精准描述业务错误。** 老师以 ValueError 为例进行对比说明——同样是 ValueError，可能是因为密码格式错误，也可能是金额输入错误，单看内置异常根本无法区分具体是哪种业务问题。而自定义异常可以直接说出具体的业务问题，比如"密码长度不足六位"、"余额不足，当前余额是多少"、"库存不足，仅剩多少件"，更加清晰直观。

**第二，方便针对性地处理不同的业务错误。** 可以为不同的业务场景定义不同的自定义异常，后续捕获时能够精准区分处理。老师举例说明：余额不足和账户冻结是两种不同的错误，前者需要提示用户充值，后者需要提示用户去解冻，分开处理逻辑会更清晰。

**第三，能够附加额外的信息。** 在自定义异常中可以封装错误码、用户 ID 等信息，方便调试和问题定位。老师举例提到：错误码 1001、用户是谁、取款金额多少、超过当前余额多少，通过这些信息能够快速找到问题根源。

### 三、自定义异常的实现步骤

老师强调实现自定义异常其实只需要两步，非常简单。

**第一步：创建异常对象。** 语法是 `异常类型(异常的具体描述信息)`。

**第二步：使用 raise 语句抛出这个自定义的异常对象。** 语法是 `raise 异常对象`。

随后老师进行了演示操作：

```python
# 第一步：创建异常对象
my_error = Exception("余额不足")
```

老师解释：如果这个异常对象需要在多个地方使用，可以定义变量保存；如果只使用一次，可以直接使用。

```python
# 第二步：抛出异常
raise my_error
```

老师指出运行后会报错，因为通过 raise 语句抛出了这个自定义异常，所以程序会中断。

接着老师进行了补充说明：入门阶段可以先用 Python 自带的 Exception 类去创建异常对象；后续学了类的知识之后，还能够定义更专属的一些异常类。也就是说学完面向对象之后，可以自己定义一个异常类（而不是直接用 Exception）。

### 四、实战演示：模拟银行取款场景

老师决定用一个实际的例子来演示，选择了模拟银行取款场景。

#### 4.1 准备初始数据

```python
# 账户余额
balance = 1000
# 账户是否冻结（布尔值比较合适，因为只有两种状态）
is_frozen = False
```

老师解释了为什么使用布尔值表示冻结状态：因为账户是否冻结只有两种状态，true 或 false，用布尔值非常合适。

#### 4.2 定义取款函数

在定义函数之前，老师先引导学生思考定义函数的三大要素：

1. **函数的作用是什么？** 实现取款功能，如果金额超出余额则抛出余额不足的错误，如果账户没有被冻结且余额充足则正常取款。
2. **是否需要参数？** 每次取款金额会不一样，实际取款时才知道要取多少，所以需要将取款金额设定为参数。
3. **是否需要返回值？** 老师认为这里重点是执行取款操作，可以不设置返回值。但也提到如果想要设置返回值，可以返回取款成功/失败（True/False），或者返回取完款之后的余额，这根据个人需求决定。这里老师选择不设置返回值。

```python
def withdraw(amount):
    # 函数体
```

#### 4.3 实现取款逻辑

**正常取款逻辑：** 余额减去取款金额，再重新赋值给余额变量。

```python
global balance
balance = balance - amount
print("当前余额是", balance)
```

老师指出这里会报错，因为 balance 是全局变量，而在函数（局部作用域）中修改全局变量的值，需要使用 global 声明。所以要先写 `global balance`。

**加入异常判断逻辑：**

首先判断账户是否冻结：

```python
if is_frozen:
    raise Exception("账户已冻结")
```

老师解释了为什么可以省略 `== True`：因为如果变量本身为 True，可以直接作为条件执行，不需要写 `if is_frozen == True`。

然后判断余额是否充足：

```python
if balance < amount:
    raise Exception("余额不足，你的当前余额是" + str(balance) + "元")
```

**关于条件判断方式的选择：** 老师特意强调这两个判断使用的是两个独立的 if 语句，而不是 if-elif。因为账户冻结和余额不足是两个独立的判断，不具备关联性。虽然代码执行时遇到 raise 会直接报错，后面代码都不会执行，但用独立的 if 语句从逻辑上看会更清晰。

#### 4.4 测试验证

老师进行了多次测试验证：

- 正常取款 200 元：成功，当前余额变为 800 元
- 账户冻结状态时取款：抛出"账户已冻结"异常
- 余额不足时取款（如取 2000 元，余额只有 800 元）：抛出"余额不足，你的当前余额是 XXX 元"异常

### 五、结合 try-except 捕获异常

老师指出代码报错后，后面的代码都无法执行，所以可以结合之前学习的 try-except 来捕获异常：

```python
try:
    withdraw(2000)
except Exception as e:
    print(e)
```

这样即使遇到异常也不会因为报错而终止运行，能够继续执行后续代码，输出友好的提示信息。

老师也进行了提醒：这种用 Exception 捕获的方式无法区分不同的业务异常，只适合简单的场景。如果是复杂的业务，后续学了类的相关知识后，推荐定义专属异常类的方式，能够更精准地分别处理不同的自定义异常。

## 本P结论

1. **自定义异常的本质**：自定义异常是专门用于描述业务相关错误的异常类型，是自己定义的错误类型，专门应对业务场景中的特定错误。相比内置异常，自定义异常能够直接提示具体的业务问题（如"余额不足"），而不是模糊的通用提示。

2. **为什么需要自定义异常**：第一，能够精准描述业务错误；第二，方便针对性处理不同的业务错误；第三，可以附加错误码、用户 ID 等额外信息，方便调试和问题定位。

3. **实现步骤**：第一步使用 `Exception("错误描述")` 创建异常对象；第二步使用 `raise 异常对象` 在合适的位置抛出。

4. **使用场景**：取款（余额不足、账户冻结）、用户注册（密码长度不足、手机号格式有误）、电商下单（库存不足、订单已取消）等特定业务场景都能用自定义异常精准描述错误。

5. **后续延伸**：当前入门阶段直接使用内置的 Exception 类；后续学了面向对象之后，可以定义更专属的异常类，实现更精细的异常区分与处理。

6. **课后练习建议**：可以尝试模拟一个用户注册功能，定义密码长度不足、手机号已被注册等自定义异常，用 raise 语句在合适位置抛出，然后结合 try-except 进行捕获处理。
