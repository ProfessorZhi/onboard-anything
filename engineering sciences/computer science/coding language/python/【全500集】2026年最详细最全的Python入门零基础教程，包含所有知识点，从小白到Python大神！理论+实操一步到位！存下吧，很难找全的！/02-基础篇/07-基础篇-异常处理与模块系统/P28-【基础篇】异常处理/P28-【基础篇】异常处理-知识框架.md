- 如何写出既能容错又能定位问题的异常处理代码？

## 知识骨架

异常处理是一套完整的「监控→捕获→处理→善后」机制

```
异常处理语法
├── try 块（监控区）
│   └── 放置可能出错的代码
├── except 块（捕获区）
│   ├── 捕获所有异常：except Exception
│   ├── 捕获指定异常：except ValueError
│   └── 获取异常信息：except ValueError as e
├── else 块（无异常分支）
│   └── 仅在try块无异常时执行
└── finally 块（善后区）
    └── 无论是否异常都执行，用于资源释放
```

核心概念
- 异常（Exception）：程序运行中的意外事件，导致执行流程中断
- 异常类型（Exception Type）：不同错误对应不同类型，如ValueError、ZeroDivisionError
- 异常对象：包含异常类型和描述信息的实例
- 异常回溯（Traceback）：Python输出的错误堆栈信息，用于定位错误

异常处理三范式
1. 捕获所有：except Exception（不推荐盲目使用）
2. 捕获指定：except ValueError（推荐优先使用）
3. 捕获并记录：except ValueError as e（便于调试）

## 关系与依赖

```
前置知识
├── 变量与数据类型（了解异常类型对应哪种错误）
├── 流程控制（if/循环的缩进规则是IndentationError的前提）
└── input()函数（异常处理的典型触发场景）

├── 异常处理语法（可保护后续文件操作、数据库操作、网络请求）
└── 异常定位能力（阅读Traceback定位问题）

后续依赖
├── P29 自定义异常（基于except语法扩展）
├── 文件操作（需要finally保证文件关闭）
└── 面向对象编程（异常是类的实例）
```

## 输入 / 输出 / 前置 / 支撑

输入
- 用户不可控数据：input()接收的任意字符串
- 文件操作：文件可能不存在或权限不足
- 类型转换：int()、float()等对非数字字符串的转换

输出
- 程序不崩溃，继续执行后续代码
- 用户友好的错误提示信息
- 异常描述信息用于调试

前置依赖
- Python基础语法（变量、函数、缩进）
- 常见异常类型的认知

支撑工具
- Python解释器（用于运行和验证异常处理代码）
- IDE/编辑器（显示行号，便于根据Traceback定位错误）

## 易错点

1. except后不写异常类型：虽然可以捕获所有异常，但会掩盖未知错误，不利于调试

2. 用异常处理替代代码修正：SyntaxError、IndentationError等语法错误必须通过修正代码解决，不能用try包裹

3. except Exception的误解：它不能捕获SyntaxError和IndentationError

4. else与except的混淆：else只在try块无异常时执行，except只在有异常时执行，两者互斥

5. 忘记finally的作用：如果代码报错且没有except，finally仍会执行，但后续代码不会继续

6. 异常类型覆盖顺序：多个except时，具体异常类型要写在前面，Exception父类要写在后面

7. as后的变量名误解：as是关键字，e是变量名（可自定义），但异常类型名不能改
