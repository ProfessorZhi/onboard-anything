## 讲解过程

### 1. 课程整体介绍

  - **第二节：抛出自定义异常**——掌握自定义专属异常的方法，让错误提示更贴合业务逻辑，精准定位问题。
- 明确本 P 主要聚焦**第一节异常处理**。

### 2. 异常的概念引入

- 先解释什么是异常：程序运行过程中发生的意外事件，会打断原本顺畅的执行流程。
- 用生活比喻说明：就像走路踩到水坑、吃饭咬到石头，原本的节奏被打乱，程序遇到异常时会直接停止运行并弹出错误信息。

### 3. 常见异常类型逐一介绍

老师列举了零基础小白最常遇到的异常类型，每个都配有简要说明和触发场景：

| 异常类型 | 含义 | 典型触发场景 |
|---|---|---|
| **NameError** | 名称错误（使用了未定义的变量） | 直接打印 `print(name)` 但未赋值 |
| **SyntaxError** | 语法错误 | if 语句少了冒号，不符合 Python 语法规则 |
| **IndexError** | 索引错误（下标越界） | 列表只有 3 个元素，却访问索引 3 |
| **ZeroDivisionError** | 除零错误 | 用零作为除数进行除法 |
| **KeyError** | 字典中键不存在 | 访问字典中不存在的键 |
| **IOError** | 文件读写错误 | 尝试打开一个不存在的 .txt 文件 |
| **AttributeError** | 属性/方法错误 | 字符串调用列表的 `append` 方法 |
| **ValueError** | 值错误 | 将非数字字符串通过 `int()` 转整数 |
| **TypeError** | 类型不匹配 | 整数与字符串直接相加 |
| **ImportError** | 模块导入错误 | 导入一个不存在的模块 |
| **IndentationError** | 缩进错误 | if/循环/函数没有正确缩进 |

老师特别提醒：遇到错误不要慌，解决后要自己总结错误原因和处理方法，避免同一坑踩多次。

### 4. 异常信息的结构解析

- 演示一个 `print(name)` 的报错现场，说明如何阅读 Python 的异常回溯（Traceback）信息。
- 指出回溯信息包含四个要素：
  1. **most recent call last**（最近一次调用）
  2. **file**（文件路径，点击可定位到具体文件）
  3. **LN2**（行号 2）
  4. **in module**（所属模块，即 .py 文件）
  5. 报错的**具体代码**
  6. **异常类型**（如 NameError）
  7. **异常描述信息**（如 name 'name' is not defined）
- 说明这套流程是之前的解决方案：通过回溯信息定位出错点，分析并改正错误。

### 5. 为什么需要异常处理

- 先演示直接修正错误（为 `name` 赋值 `"冰冰"`）的传统方式。
- 指出问题：有些异常**无法避免**，典型场景就是用户输入。
- 演示代码：

```python
nm = int(input("请输入一个数字:"))
```

- 如果用户输入 `12345` → 正常；如果输入 `abc` → 报错 `ValueError`。
- 说明原因：用户输入的内容不可控，`input()` 本身可以输入任意内容，无法限制。
- 引出核心需求：捕获异常并妥善处理，让程序遇问题不崩溃、继续执行，并给出友好提示。

### 6. 异常处理语法结构

老师展示了 try-except 的基本结构：

```python
try:
    # 可能出错的代码（可能报错，也可能不报错）
except:
    # 捕获到异常后要执行的代码（处理逻辑）
```

- 用走路预判水坑做比喻：提前准备好应对方案，踩到了就换条路走，而不是直接摔倒（程序崩溃）。
- 配合流程图说明：程序执行 try 块 → 判断是否有异常 → 无异常则继续执行后续代码；有异常则执行 except 块的处理代码，然后继续执行后续代码。
- 提到还有 `else` 和 `finally` 两个附加关键字，之后会逐一讲解。

### 7. 第一种方式：捕获所有异常

```python
try:
    # 可能出错的代码
    num = int(input("请输入一个数字:"))
    print("123")
except Exception:
    print("输入错误，请输入一个正确的数字")
```

- 说明 `except Exception` 可以省略写成 `except`，但如果后续需要获取异常描述信息，就要保留 `Exception`。
- 实际演示：输入 `123` → 正常执行；输入 `abc` → 打印提示信息，程序不崩溃，后续代码继续执行。
- 指出 `Exception` 是所有**非语法错误**异常的父类，能捕获除语法错误以外的所有异常。
- 强调**不推荐盲目使用**这种方式，因为无法区分异常的具体类型，不利于定位问题。
- 补充说明：`Exception` 捕获不到语法错误（如缺少冒号、缩进错误）。

### 8. 第二种方式：捕获指定异常

#### 8.1 只捕获特定类型

```python
try:
    num = int(input("请输入:"))
except ValueError:
    print("捕获到了值错误，请输入一个纯数字")
```

- 先演示不包裹 try，直接运行 `int(input())` 输入 `abc`，报错类型是 `ValueError`。
- 说明优势：只针对特定异常进行处理，代码逻辑更清晰。

#### 8.2 用元组捕获多种指定异常（统一处理）

```python
try:
    num = int(input("请输入:"))
    result = 1 / 0
except (ValueError, ZeroDivisionError):
    print("发生了错误，可能是输入有误或者除数为零")
```

- 说明可以用元组指定多种异常类型，触发其中任意一种都执行相同的处理逻辑。
- 演示：输入 `abc` → 捕获到 `ValueError`；输入 `123` → 触发 `1/0` 的 `ZeroDivisionError`。

#### 8.3 多个 except 分开处理

```python
try:
    num = int(input("请输入:"))
    result = 1 / 0
except ValueError:
    print("输入有误，请输入纯数字")
except ZeroDivisionError:
    print("除数不能为零")
except TypeError:
    print("类型错误")
```

- 说明可以对不同异常采用不同的处理方式，写多个 `except` 块即可。
- 老师依次演示三种错误场景，分别输出对应的友好提示，验证逻辑正确。
- 强调这是推荐优先使用的方式，逻辑清晰，用户能清楚知道自己错在哪里。

### 9. 第三种方式：捕获异常描述信息

```python
try:
    result = 1 / 0
except ZeroDivisionError as e:
    print("异常具体描述信息:", e)
```

- 说明 `as e` 中的 `e` 是一个普通变量名（可自定义），用于保存异常对象。
- 打印 `e` 会输出异常的具体描述信息（如 `division by zero`）。
- 扩展到 `except Exception as e:`，演示 `NameError` 和 `TypeError` 也能获取各自的具体描述信息。
- 强调这种方式的优势：不仅让程序不崩溃，还能看到具体错误信息，便于调试。
- 指出 `as` 是关键字不可改，但 `e` 是普通变量可以换成其他名称。

### 10. else 和 finally 两个附加关键字

#### 10.1 else——无异常时执行

```python
try:
    num = int(input("请输入一个数字:"))
except ValueError as e:
    print("异常的具体描述信息:", e)
else:
    print(num, "的数据类型是:", type(num))
```

- 说明 `else` 在 try 块中的代码没有出错时执行。
- 演示：输入 `abc` → 走 except 分支，打印异常描述信息；输入 `123` → 走 else 分支，输出变量的值和类型。
- 指出这样可以把正常逻辑和异常逻辑分离，代码更清晰。

#### 10.2 finally——无论是否异常都执行

- 说明 `finally` 经常用于释放资源（如关闭文件），避免资源泄漏和影响下次访问。
- 用文件操作为例讲解：正常流程是打开文件 → 文件操作 → 关闭文件。
- 指出如果中间的代码报错了，后面的关闭文件代码就不会执行，导致文件一直占用系统资源。
- 演示代码结构：

```python
try:
    # 打开文件
    # 文件操作（可能报错）
except Exception as e:
    print("异常描述信息:", e)
finally:
    print("关闭文件")  # 无论是否报错都会执行
```

- 说明 `finally` 可以单独和 `try` 搭配使用（`try...finally`），不需要 `except`。
- 演示：无论报不报错，`finally` 中的打印都会执行，确保文件最终能关闭。
- 补充：如果想让程序在报错后继续执行后续代码，需要同时写上 `except`。
- 最后用一句话总结四个关键字的职责：
  - **try**：放可能报错的代码
  - **except**：放如果报错怎么处理的代码
  - **else**：放如果不报错怎么处理的代码
  - **finally**：放不管是否报错都要执行的代码

### 11. 异常处理的核心规则

老师给出了两条重要规则：

1. **优先捕获具体的异常**：如果明确知道会触发 `ValueError`，就只捕获它，不要用 `Exception` 这种万能异常盲目捕获所有异常，否则会掩盖未知的错误，不利于调试。
2. **不要滥用异常处理**：语法错误（少冒号、缩进错误等）应该通过修正代码来解决，不要用异常处理来包裹这些本可以避免的问题。

## 本P结论

1. **异常**是程序运行中发生的意外事件，不处理会导致程序直接崩溃并输出错误堆栈。
2. Python 提供了完整的异常处理语法结构：**try**（监控可能出错的代码）、**except**（捕获并处理异常）、**else**（无异常时执行）、**finally**（无论是否异常都执行）。
3. 捕获异常有三种方式：① 用 `except Exception` 捕获所有非语法错误（不推荐盲目使用）；② 捕获指定类型的异常，推荐优先使用；③ 用 `except ExceptionType as e` 捕获异常并获取其描述信息，便于调试。
4. 可以用元组在同一个 `except` 中指定多种异常类型用统一逻辑处理，也可以写多个 `except` 块分别处理不同类型的异常。
5. `finally` 块常用于释放资源（如关闭文件），确保无论代码是否报错，相关资源都能被正确释放。
6. 使用异常处理时应优先捕获具体异常类型，不要滥用异常处理来包裹本可避免的语法错误。
7. 本 P 内容覆盖异常处理的基本用法，自定义异常的抛出方法将在下一节（第二小节）中讲解。
