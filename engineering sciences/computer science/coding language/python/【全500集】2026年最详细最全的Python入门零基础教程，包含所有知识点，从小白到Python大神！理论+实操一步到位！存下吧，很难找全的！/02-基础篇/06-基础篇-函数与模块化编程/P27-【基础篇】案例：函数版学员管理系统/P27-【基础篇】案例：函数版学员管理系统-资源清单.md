# P27 资源清单 - 【基础篇】案例：函数版学员管理系统

> **说明**：本 P 为纯 Python 标准库实战课，未引用任何第三方库、外部平台或工具链。资源全部来自 Python 内置语言特性与内置函数。来源标注以"字幕直接提到"或"基于上下文推断"区分。

---

## 一、语言内置函数

### 1. `input()`

| 字段 | 内容 |
|------|------|
| **名称** | `input()` |
| **类型** | Python 内置函数 |
| **本P作用** | 接收用户的所有交互输入：功能序号、学员姓名、学号、手机号、退出确认（yes/no）；返回值类型始终为字符串，直接与字符串字面量比较分支 |
| **来源** | 字幕直接提到 |
| **建议方向** | 练习在不同场景下对 `input()` 返回值做类型转换（`int()`、`float()`），并对空输入或格式错误设计防御性处理逻辑 |

---

### 2. `print()`

| 字段 | 内容 |
|------|------|
| **名称** | `print()` |
| **类型** | Python 内置函数 |
| **本P作用** | 输出菜单界面（分隔线、标题、选项）、操作反馈（添加成功/失败、删除成功/未找到、修改结果）、调试验证信息 |
| **来源** | 字幕直接提到 |
| **建议方向** | 研究 `print()` 的 `sep`、`end` 参数，尝试用 `print` 格式化对齐表格式输出（为后续"显示所有"功能的美化做准备） |

---

### 3. `len()`

| 字段 | 内容 |
|------|------|
| **名称** | `len()` |
| **类型** | Python 内置函数 |
| **本P作用** | 本 P 未显式使用，但"显示所有"功能判断列表是否为空时隐式依赖此逻辑（`if not info` 等价于 `len(info) == 0`）；知识框架中空列表提示属于潜在需求 |
| **来源** | 基于上下文推断 |
| **建议方向** | 在"显示所有"功能中加入空列表判断：`if not info: print("当前无学员信息")`，体会 `len()` 与布尔值判断的等价关系 |

---

## 二、内置数据结构与相关方法

### 4. `list`（列表）

| 字段 | 内容 |
|------|------|
| **名称** | `list` / 列表 `[]` |
| **类型** | Python 内置数据类型 |
| **本P作用** | 定义全局变量 `info = []` 作为所有学员数据的容器；所有函数通过访问同一个全局列表共享状态，避免参数传递 |
| **来源** | 字幕直接提到 |
| **建议方向** | 对比全局列表与参数传递两种方式的优劣；尝试将 `info` 作为参数传入各函数，观察函数签名的变化，理解可变对象传参的行为 |

---

### 5. `list.append()`

| 字段 | 内容 |
|------|------|
| **名称** | `list.append()` |
| **类型** | 列表实例方法 |
| **本P作用** | 在 `add_info()` 中将新构建的学员字典追加到全局列表 `info` 末尾，完成添加操作 |
| **来源** | 字幕直接提到 |
| **建议方向** | 对比 `append()` 与 `insert()` 的区别；尝试实现"按学号升序插入"功能，体会插入位置控制 |

---

### 6. `list.remove()`

| 字段 | 内容 |
|------|------|
| **名称** | `list.remove()` |
| **类型** | 列表实例方法 |
| **本P作用** | 在 `del_info()` 中从 `info` 列表中移除匹配到的学员字典对象；传入的是字典对象引用而非索引 |
| **来源** | 字幕直接提到 |
| **建议方向** | 研究 `remove()` 与 `pop()` 的区别（按值 vs 按索引）；探究在遍历列表过程中直接调用 `remove()` 的安全性问题，尝试复现"遍历时删除"导致跳过元素的经典 bug |

---

### 7. `dict`（字典）

| 字段 | 内容 |
|------|------|
| **名称** | `dict` / 字典 `{}` |
| **类型** | Python 内置数据类型 |
| **本P作用** | 每个学员信息以字典形式存储，固定三个键：`"id"`（学号）、`"name"`（姓名）、`"tel"`（手机号）；字典嵌套在列表中构成核心数据结构 |
| **来源** | 字幕直接提到 |
| **建议方向** | 尝试为字典新增字段（如 `"score"` 成绩），同步修改添加、显示、修改函数，体验扩展字段对各模块的影响范围 |

---

### 8. 字典键名直接赋值

| 字段 | 内容 |
|------|------|
| **名称** | `dict[key] = value`（键名赋值） |
| **类型** | Python 字典操作语法 |
| **本P作用** | 在 `modify_info()` 中直接将 `input()` 返回值赋给 `student["id"]` 和 `student["tel"]`，无需中转变量，原地修改字典内容 |
| **来源** | 字幕直接提到 |
| **建议方向** | 验证字典是可变对象（mutable）：在函数内修改通过遍历取得的字典对象，确认修改会直接反映到全局列表中，深化对 Python 引用语义的理解 |

---

## 三、控制流语法

### 9. `while True` + `break`

| 字段 | 内容 |
|------|------|
| **名称** | `while True` 死循环 + `break` |
| **类型** | Python 流程控制语法 |
| **本P作用** | 作为整个系统的主循环骨架：每轮循环显示菜单、接收输入、执行操作，循环次数不确定，唯一退出点是用户选择 6 并二次确认后执行 `break` |
| **来源** | 字幕直接提到 |
| **建议方向** | 对比 `while True + break` 与 `while 条件变量` 两种写法，改写本项目主循环为 `while running:` 的形式，比较可读性与维护性差异 |

---

### 10. `for` 循环（遍历列表）

| 字段 | 内容 |
|------|------|
| **名称** | `for ... in ...`（列表遍历） |
| **类型** | Python 流程控制语法 |
| **本P作用** | 在添加、删除、修改、查询功能中遍历 `info` 列表，逐个取出学员字典进行姓名匹配；是所有增删改查逻辑的查找基础 |
| **来源** | 字幕直接提到 |
| **建议方向** | 研究 `for...else` 语法（循环正常结束后执行 else）——本 P 未直接使用但知识框架已提及，可将"未找到"的 print 语句改写为 `for...else` 结构以消除标志变量 |

---

### 11. `if-elif-else` 多分支结构

| 字段 | 内容 |
|------|------|
| **名称** | `if-elif-else` |
| **类型** | Python 流程控制语法 |
| **本P作用** | 根据用户输入的序号（`"1"`～`"6"` 及非法输入）分发到对应功能函数；老师强调 `elif` 的短路特性可提高判断效率 |
| **来源** | 字幕直接提到 |
| **建议方向** | 尝试用字典映射（`dispatch dict`）重构本 P 的 `if-elif` 分支：`actions = {"1": add_info, "2": del_info, ...}`，体会两种分发方式的可扩展性差异 |

---

### 12. `return`（无返回值）

| 字段 | 内容 |
|------|------|
| **名称** | `return`（裸 return） |
| **类型** | Python 函数控制语法 |
| **本P作用** | 在 `add_info()` 检测到姓名重复时、`del_info()` 删除成功后、`modify_info()` 修改成功后，立即终止当前函数运行，防止后续代码继续执行；默认返回 `None` |
| **来源** | 字幕直接提到（老师重点辨析 return vs break） |
| **建议方向** | 故意将功能函数中的 `return` 替换为 `break`，运行程序复现 bug，通过实验深化对两者作用域差异的理解 |

---

## 四、字符串相关

### 13. f-string（格式化字符串）

| 字段 | 内容 |
|------|------|
| **名称** | f-string（`f"...{变量}..."`） |
| **类型** | Python 字符串格式化语法（Python 3.6+） |
| **本P作用** | 在所有操作反馈信息中嵌入变量值：`f"添加成功，学员信息为{student_dict}"`、`f"错误：姓名为 {new_name} 的学员已存在"` 等，使输出信息动态可读 |
| **来源** | 字幕直接提到 |
| **建议方向** | 研究 f-string 的格式化修饰符（如 `{value:<10}` 左对齐补空格），用于美化"显示所有"功能的表格输出 |

---

| 字段 | 内容 |
|------|------|
| **名称** | 字符串重复运算符 `*` |
| **类型** | Python 字符串操作语法 |
| **本P作用** | 在 `info_print()` 中生成菜单分隔线：`"=" * 30`，简洁生成固定宽度的视觉分隔符 |

## 五、设计模式与架构概念

### 15. 全局变量共享状态模式

| 字段 | 内容 |
|------|------|
| **名称** | 全局变量共享状态（Global State） |
| **类型** | 程序设计模式（概念） |
| **本P作用** | `info = []` 定义在模块顶层，所有功能函数直接读写该变量，无需传参；老师将其作为"简单场景下多函数共享数据"的解决方案 |
| **来源** | 字幕直接提到 |
| **建议方向** | 理解全局状态在多函数协作中的便利性与隐患（难以测试、难以复用）；学习面向对象后，用类属性替代全局变量重构本项目，对比两种方案 |

---

### 16. 函数封装 / 模块化设计

| 字段 | 内容 |
|------|------|
| **名称** | 函数封装（Function Encapsulation） |
| **类型** | 程序设计原则（概念） |
| **本P作用** | 将菜单显示、添加、删除、修改各自封装为独立函数（`info_print`、`add_info`、`del_info`、`modify_info`），主循环通过函数名调用，降低耦合、提高可读性 |
| **来源** | 字幕直接提到 |
| **建议方向** | 为每个功能函数补充函数文档字符串（docstring），并尝试为函数编写简单的单元测试，体验模块化带来的可测试性提升 |
