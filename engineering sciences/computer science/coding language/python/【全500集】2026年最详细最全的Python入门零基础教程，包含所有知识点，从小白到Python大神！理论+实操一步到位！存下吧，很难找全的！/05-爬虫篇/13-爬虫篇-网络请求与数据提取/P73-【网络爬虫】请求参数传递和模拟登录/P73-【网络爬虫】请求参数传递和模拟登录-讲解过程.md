# P73 讲解过程 - 【网络爬虫】请求参数传递和模拟登录
## 本P定位

本P位于Python网络爬虫教程的第73集，是爬虫进阶内容的前置章节。前半段承接第72P的豆瓣Top250案例，引出URL查询参数的概念与处理方式；后半段从GET请求参数传递过渡到POST请求，讲解POST请求在模拟登录场景中的应用，并系统介绍两种主流的鉴权机制——Cookie + Session 与 Token。整个P为后续真正的模拟登录实战打牢理论和编码基础。

## 讲解过程

### 一、问题的引出：豆瓣Top250 URL中的参数

课程从回顾第72P的豆瓣Top250数据抓取入手。讲师在浏览器地址栏中打开豆瓣Top250页面，指出URL中存在问号及后续的键值对（问号后面什么等于什么）。他明确告知学生，这种出现在URL地址后面的键值对在HTTP请求中叫做参数，其中以GET方式传递的这类参数特别称为查询字符串参数（Query String Parameters）。

讲师随后演示了如何通过浏览器开发者工具查看该参数。他右键选择“检查”，切换到Network（网络）面板，点击刷新页面，找到第一个请求条目，查看请求载荷（Payload）部分。演示中，他将翻页参数从0改成50，页面随之切换到第3页数据，以此说明参数的作用。页面上还有另一个参数filter，用于过滤，但具体用途未知。讲师强调，通过request发送GET请求时，查询参数的有无和具体值在浏览器Network面板中都可以直接观察。

在理论区分环节，讲师对比了两种不同类型的参数位置：路径参数出现在URL路径部分（斜杠后面），而查询参数必须以问号为标志出现在路径之后，格式为“键=值&键=值”。以百度搜索为例，在百度输入“长城”并回车后，Network面板中显示的请求地址末尾携带了“长城”这一查询参数。

### 二、GET请求参数传递的两种方式

进入代码演示阶段。讲师新建文件，命名为“03-request请求参数的传递方式”，导入Python的requests库，开始讲解GET请求传递查询参数的两种写法。

**方式一：直接拼接URL。** 讲师直接在目标URL后面手动追加参数部分，例如在请求百度搜索时将“?wd=长城&ie=UTF-8”拼接到URL末尾。他指出这种方式的弊端：当需要修改参数值时不够灵活，且参数若包含中文会被URL编码转义，可读性差。演示代码中使用headers添加请求头以模拟浏览器访问，response.text输出页面内容，运行后成功获取了包含“长城汽车官网”“长城皮卡”等信息的页面内容，与浏览器直接搜索的结果一致。

**方式二：使用requests的params参数。** 讲师注释掉方式一的代码，将URL恢复为不带参数的干净地址，演示如何通过params字典传递参数。他要求学生先通过抓包确认目标请求携带了哪些参数。演示中，从百度搜索的请求中提取出两个关键参数：ie（值为UTF-8）和wd（值为要搜索的关键词）。params字典以键值对形式声明这些参数，调用requests.get时通过params=params的方式传递。运行后同样成功获取了包含目标内容的页面。讲师通过对比验证，确认方式二与方式一效果相同，但代码可读性更高，参数管理更方便。

### 三、POST请求的引入与使用场景

讲师正式引入POST请求的概念。他说明编写爬虫大多数时候使用GET请求，但某些场景必须使用POST，主要包括三种情况：一是登录时提交账号密码数据，二是部分数据提交表单使用POST，三是某些需要发送数据的接口（例如手机版百度翻译）。

为说明POST请求的实际应用，讲师切换到手机版百度翻译页面演示抓包。他点击检查打开开发者工具，切换到Network面板，将页面切换为手机视图，刷新页面后在搜索框输入“python”并回车。他逐一查看抓到的请求列表，最终定位到两个请求——一个是搜索请求（GET），返回搜索结果列表；另一个是翻译请求，使用了POST方法，请求地址为翻译接口URL，传递的参数是“python3”。返回的数据是翻译结果。

### 四、POST请求参数传递的两种类型

讲师首先从理论层面区分POST请求中两类不同的参数。

第一类是表单参数（Form Data），参数以键值对形式编码在请求体中，通过data字段传递。这种参数对应浏览器Network面板中Payload下方的Form Data部分。GET请求的参数叫Query String Parameters，而POST表单参数在请求体中存放。

第二类是JSON参数，参数以JSON格式编码，同样放在请求体中，通过json字段传递。讲师预告后续会专门讲解这种参数格式。

### 五、百度翻译手机版实战：表单参数传递

进入代码实战环节。讲师新建文件“03-发送POST请求”，导入requests库，准备参数字典。首先演示表单参数的传法。他从抓包结果中得知百度翻译手机版POST请求携带的参数为kw（即搜索关键词），赋值为python3。请求头从浏览器开发者工具中复制相关字段。调用requests.post()方法，url参数为目标URL，data参数传入参数字典。发送请求后通过response.content获取二进制内容解码输出。运行后得到返回结果，其中包含一个session_id字段，每次请求的session_id值都不同，证明请求成功发送。

### 六、JSON参数传递：模拟登录案例

讲师新建文件“03-02-帧请求传递表单参数案例一”（此处“帧”应为“JSON”的口误），切换到另一个实际网站的登录页面。他提示学生：有些网站的登录接口传递的是JSON格式参数，需要使用json字段传递而不是data字段。

他从浏览器Network面板中复制该登录请求的请求参数。参数中包含username（账号）、password（密码）等字段，type字段表示登录类型（邮箱登录）。他将复制的参数粘贴到代码中，逐一加上引号构成Python字符串。账号和密码字段中，密码他故意填写了一个错误的值，以便观察请求返回结果而不触发页面跳转。他指出如果密码填正确，请求成功后页面会直接跳转，反而抓不到登录请求本身。

发送请求时，他使用requests.post()，url为目标登录接口地址，由于该接口参数为JSON格式，因此使用json=参数字典的方式传递，而不使用data字段。运行代码后返回了一段内容，包含数字码和一段中文提示。讲师解释这段返回内容是JSON数据而非HTML网页，因此他使用response.json()方法将响应内容直接解析为Python字典格式。结果显示“密码错误，你还可以尝试两次”，证明请求成功发送并得到了服务器的正确响应。

### 七、模拟登录的核心问题：如何保持登录状态

讲师回到理论层面，提出关键问题：登录成功之后，如何用代码访问需要登录才能查看的页面？他以中华网和课堂派两个实际网站为例说明。

登录成功后，服务器如何知道用户已经登录？这涉及鉴权机制。讲师指出目前主流的鉴权方式有两种：第一种是Cookie + Session，第二种是基于Token的鉴权。

### 八、Cookie + Session鉴权机制的完整流程

讲师以登录操作为例，详细图解了Cookie + Session鉴权的工作流程。

当用户第一次访问某个网站并填写账号密码提交时，浏览器发送一个POST请求到服务器。服务器收到请求后，验证账号密码正确，随即在服务器端生成一个Session（会话），Session中保存用户信息和登录时间等数据。服务器将这个Session对应的一个唯一标识符——Session ID——通过响应头中的Set-Cookie字段返回给浏览器。浏览器收到Set-Cookie后，自动将Session ID保存到本地的Cookie存储区中。此后，用户每次向该网站发送新的请求时，浏览器会自动携带Cookie中的所有数据，包括Session ID。服务器收到请求后，从Cookie中取出Session ID，去后台缓存或数据库中查找对应的Session记录，如果能找到且Session仍在有效期内，就认定该用户已登录，从而返回对应的数据。

讲师强调Cookie的特点是每次请求都会自动携带，且有过期时间，过期后自动清除。他随后切换到中华网登录页面演示实际操作：先打开浏览器的Application面板查看Cookies，初始只有四个字段。然后输入正确的账号密码登录，登录成功后刷新Application面板，发现Cookie中新增了大量字段，其中包括LINKNAME等用户身份信息。登录后访问该网站的其他页面（如修改密码页面），刷新Network面板，可以观察到请求信息自动携带了全部Cookie数据传给后台服务器。

他还演示了另一个场景：复制一个需要登录才能访问的页面URL，粘贴到另一个未登录的浏览器中打开，结果页面直接跳转到登录提示页面，证明服务器成功拒绝了未携带有效Cookie的请求。这说明Cookie + Session鉴权机制在实际网站中广泛使用。

### 九、Token鉴权机制详解

讲师随后介绍第二种鉴权方式——基于Token的鉴权机制。他以课堂派网站为例说明。

当用户在客户端输入账号密码并提交登录后，服务器校验成功，会根据账号密码加上登录时间和有效期等信息，通过加密算法生成一串密文（即Token），将这串密文放在响应体中返回给客户端。Token不像Session ID那样放在Cookie的Set-Cookie响应头中返回，而是直接放在响应体里。

客户端收到Token后，通常保存在浏览器的LocalStorage或SessionStorage中。下次用户访问其他页面时，浏览器不会像携带Cookie那样自动携带Token。前端开发人员在编写页面JavaScript代码时，会主动从LocalStorage中取出Token，将其放在请求头或请求体中发送给服务器。服务器收到请求后，提取Token并验证其有效性，从而判断用户是否有权访问。

讲师在课堂派网站上演示：登录后打开Application面板，切换到Local Storage选项卡，可以看到其中存储了用户的Token信息。而切换到Cookies选项卡时，该网站的Cookie是空的，说明这个网站完全采用Token鉴权，不依赖Cookie。

### 十、两种鉴权方式的识别特征

讲师最后总结如何从请求特征判断一个网站使用哪种鉴权方式。

如果登录请求发送时，Network面板中Payload下的参数类型显示为Form Data（表单格式），那么这个网站通常没有做前后端分离，绝大多数（90%以上）使用Cookie + Session鉴权。如果登录请求发送时，Payload中显示的不是Form Data，而是请求载荷（Request Payload），那么这个网站大概率采用了前后端分离架构，使用的是Token鉴权。

另一个辅助判断方法是查看登录接口的域名地址。如果登录请求发往的域名与当前网站的域名不同（例如课堂派的登录接口域名为account.ktpedu.com，而主站域名为www.ktpedu.com），这强烈暗示该网站使用了Token鉴权方式，因为前后端分离项目中登录认证服务通常独立部署。

## 本P结论

本P建立了爬虫中处理请求参数和保持会话状态的核心知识体系。URL中的查询字符串参数可通过直接拼接URL或使用requests.get()的params参数两种方式传递。POST请求携带的参数分为表单参数（Form Data，使用data字段传递）和JSON参数（使用json字段传递），具体使用哪种取决于目标接口的要求。在模拟登录场景中，维持登录状态的核心在于理解鉴权机制：Cookie + Session方式下，登录成功后服务器通过Set-Cookie响应头返回Session ID，浏览器自动保存并在后续请求中携带，服务器据此判断用户身份；Token方式下，登录成功后服务器在响应体中返回Token，客户端需通过前端代码将Token取出并附加到后续请求中。对于需要登录才能抓取的网站，必须先确认其鉴权方式，再通过代码模拟对应的登录流程并携带相应的认证信息，才能成功获取目标数据。
