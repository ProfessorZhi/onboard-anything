# P70 讲解过程 - 【网络爬虫】http协议
## 本P定位

本P是网络爬虫章节的前置理论课，目标是让学习者理解 HTTP 和 HTTPS 协议的工作原理，为后续编写爬虫代码打下基础。老师强调“搞清楚了请求里包含什么，才能在代码中模拟浏览器发请求”。本P不涉及代码实操，属于纯概念讲解。

## 讲解过程

### 一、引入：从浏览网页到 HTTP 协议

老师先从日常操作切入——在浏览器地址栏输入网址回车，这个过程背后发生了什么。指出爬虫的本质就是“用代码模拟人去发请求拿数据”，而发请求依赖的技术就是 HTTP/HTTPS。然后逐一解释：

- **HTTP**：HyperText Transfer Protocol，超文本传输协议，是发布和接收 HTML 网页的方法。
- **HTTPS**：HTTP + SSL，安全套接层，对数据进行加密传输，防止网络传输过程中被截取明文。

两者本质区别在于 HTTPS 多了一层加密层，老师的结论是“写爬虫时不用纠结用 HTTP 还是 HTTPS，它们的请求报文格式是一样的”。

### 二、浏览器请求网页的完整流程（结合图示）

老师用图示讲解了一个请求从发出到响应返回的完整链路：

1. 用户在浏览器地址栏输入 URL
2. 浏览器先进行**域名解析**（DNS），将域名转换为 IP 和端口
3. 浏览器向目标服务器**发送 HTTP 请求**
4. 服务器接收请求，按照 HTTP 响应格式**返回数据**
5. 浏览器接收响应数据，**解析渲染**成网页呈现给用户

老师特别强调：**打开一个页面，发出的不是一个请求，而是多个请求**。第一次请求返回 HTML 之后，浏览器会解析 HTML 中引用的 CSS、JS、图片、字体等资源地址，然后自动再次发请求把这些资源也拿过来，最终组合成完整页面。

### 三、URL 的结构解析

老师详细拆解了 URL 的各部分组成，以 `https://www.baidu.com/s?wd=美食` 为例：

| 部分 | 示例 | 含义 |
|------|------|------|
| 协议 | `https://` | 决定用什么协议通信（还有 ftp、smtp 等） |
| 域名 | `www.baidu.com` | 服务器的域名 |
| 路径 | `/s` | 请求的资源路径 |
| 参数 | `?wd=美食` | 问号后面是查询字符串参数 |

老师预告：后续写爬虫时会在 URL 后面加参数，这个叫 Query String Parameters（查询字符串参数）。

### 四、HTTP 请求报文结构（重点）

老师指出这是爬虫工程师必须掌握的核心内容，因为“模拟浏览器发请求，就需要知道请求里要包含哪些东西”。

**请求报文由四部分构成**：

1. **请求首行**：包含请求方法、请求地址、协议版本（如 `GET / HTTP/1.1`）
2. **请求头**（Key-Value 形式）：从第二行到空白行之间的所有内容
3. **空白行**：分隔请求头和请求体
4. **请求体**：要传输给服务器的数据

老师现场演示：在浏览器按 F12 打开调试控制台，切换到 Network（网络）标签，刷新页面后点击任意请求，选择“Raw Headers”（原始Headers）格式，可以直接看到请求报文原始内容。演示了两个例子——一个是 GET 请求（只有请求首行和请求头，没有请求体），一个是 POST 请求（登录场景，有请求体数据）。

### 五、HTTP 请求方法

老师按协议版本演进顺序讲解：

- **HTTP/0.9**：只有 GET
- **HTTP/1.0**：新增 POST、HEAD
- **HTTP/1.1**（目前主流）：在 1.0 基础上新增 OPTIONS、PUT、DELETE、TRACE、CONNECT 五种

**爬虫最常用的是 GET 和 POST 两种**：

| 方法 | 用途 | 参数位置 | 特点 |
|------|------|----------|------|
| GET | 获取页面/数据 | URL 路径后面（查询字符串） | 参数暴露在地址栏，有长度限制 |
| POST | 提交数据（如登录） | 请求体里面 | 参数不可见，相对更安全，无长度限制 |

老师强调：登录场景必须用 POST，因为账号密码不能明文暴露在 URL 里。GET 请求参数过多时因为首行长度限制会传不了。

### 六、常见 HTTP 请求头讲解

老师逐一介绍浏览器开发者工具中常见的请求头字段：

| 请求头 | 含义 |
|--------|------|
| Accept | 期望返回的内容类型（如 `*/*` 表示接受所有类型） |
| Accept-Encoding | 支持的压缩方式（gzip、br 等），用于解压缩响应 |
| Accept-Language | 期望的语言（如 zh-CN 表示中文） |
| Content-Length | 请求体的字符长度 |
| Content-Type | 请求体的数据类型（如 `application/json`） |
| Cookie | 存储用户凭证信息，后续会重点讲 |
| Host | 请求的目标域名 |
| Referer | 当前请求是从哪个页面跳转来的（防盗链用） |
| X-Requested-With | 标识 AJAX 异步请求 |
| User-Agent | 用户身份信息，包含浏览器内核、版本、操作系统等 |
| Connection | 连接类型（长连接/短连接） |

**重点强调三个字段**：

- **User-Agent**：浏览器的身份标识，如果不设置，服务器可能识别为爬虫（默认 User-Agent 是爬虫程序名如 "Baiduspider"）。
- **Cookie**：存储用户登录状态和会话信息，后续写爬虫经常用到。
- **Referer**：表示“我从哪个页面跳转过来的”，很多网站用它做防盗链校验。

### 七、HTTP 响应报文结构

老师同样用四块结构讲解响应报文：

1. **状态行**（也叫开始行）：包含协议版本、状态码、状态描述（如 `HTTP/1.1 200 OK`）
2. **响应头**：Key-Value 形式的元信息
3. **空白行**
4. **响应体**：服务器返回的实际内容（HTML、JSON 等）

### 八、响应状态码详解

老师按区间讲解状态码含义及常见值：

| 区间 | 含义 | 典型状态码 |
|------|------|------------|
| **1xx** | 请求收到，继续处理（很少见） | 100 |
| **2xx** | 请求成功 | **200 OK** |
| **3xx** | 重定向，资源转移到其他地址 | 302（临时重定向）、304（使用缓存）、307 |
| **4xx** | 客户端错误，请求有问题 | **401**（未授权/登录失败）、**403**（服务器拒绝/权限不足，爬虫常见）、**404**（资源不存在） |
| **5xx** | 服务器端错误，后台处理报错 | **500**（内部错误）、**503**（服务不可用，常用于反爬） |

老师提醒：爬虫最常碰到的就是 200、302、403、404、500 这几个。

### 九、Cookie 机制深度讲解

老师用百度搜索场景举例，详细解释 Cookie 的工作原理：

**为什么淘宝搜索某商品后，关掉浏览器再打开还会推荐相关广告？**

流程是：

1. 第一次访问百度，搜索“美食”
2. 百度服务器在响应头中通过 **Set-Cookie** 字段返回一些 Cookie 数据
3. 浏览器收到后自动把这些 Cookie 保存到本地（可通过开发者工具 Application → Cookies 查看）
4. 之后无论打开百度多少次、搜索什么内容，发出的请求都会自动携带这些 Cookie 一起发给服务器
5. 服务器从 Cookie 中读取之前的搜索记录，据此推送个性化内容

**Cookie 的特点**：

- 存储在浏览器本地，有有效期，过期才自动清除
- 浏览器发请求时自动携带，无需手动处理
- 可存储用户登录状态、搜索记录、偏好设置等
- 对于需要登录才能访问的数据，爬虫必须携带正确的 Cookie 才能获取

老师演示：在浏览器中查看 Cookie 存储位置（Application 面板 → Cookies），以及在 Network 中查看请求头里自动携带的 Cookie。

## 本P结论

1. **HTTP 是超文本传输协议，HTTPS 是加了 SSL 加密的安全版本**，两者请求报文格式一致，爬虫无需纠结协议类型。
2. **一次页面访问背后会发出多个请求**，浏览器自动解析 HTML 中的资源引用并再次请求。
3. **HTTP 请求报文由四部分构成**：请求首行、请求头、空白行、请求体。爬虫写代码时要模拟的正是这四部分。
4. **GET 和 POST 是爬虫最常用的两种请求方法**，GET 参数在 URL 中，有长度限制；POST 参数在请求体中，无长度限制，适合登录等场景。
5. **User-Agent、Cookie、Referer 是爬虫最需要关注的三个请求头**，直接影响请求是否能成功。
6. **响应状态码反映请求结果**，2xx 成功、3xx 重定向、4xx 客户端错误（如 403/404）、5xx 服务器错误。
7. **Cookie 是维持用户会话的核心机制**，爬虫访问需要登录的网站时必须正确处理 Cookie。
