# P84 讲解过程 - 【网络爬虫】多线程并发爬虫案例
## 本P定位

本P是“Python网络爬虫”系列的第84集，承接前面单线程爬虫的基础知识，引入**多线程并发爬取**的概念与实现思路。课程从一个具体的音乐网站案例出发，先演示了**串行下载歌曲（封面图、MP3、歌词）**的完整流程，随后将该流程拆解为三个阶段（请求→解析→存储），讲解了如何利用**队列（Queue）**作为数据管道，将三个阶段解耦为独立线程并行执行，从而提升爬虫的整体效率。

---

## 讲解过程

### 一、为什么要用异步并发爬虫

- **背景**：当需要抓取的数据量很大时，串行爬取耗时长，效率低。
- **解决方案**：使用异步并发爬虫，同时发送多个请求。
- **注意事项**：并发请求会导致访问目标服务器的频率大幅提升，容易被对方检测并封禁IP。老师建议异步爬虫**最好配合代理IP使用**，每次请求使用不同IP，降低被反爬机制识别和封禁的概率（本P不展开代理IP的使用细节，预告后续会讲）。

### 二、案例需求说明

老师展示了一个**音乐网站**，目标是：

- 搜索某个歌手（如周杰伦），获取该歌手的所有歌曲列表。
- 对每首歌，下载以下三类资源到本地：
  - **MP3文件**：歌曲的音频数据。
  - **封面图片**：歌曲的展示图（通过URL参数可调节尺寸）。
  - **TXT文件**：歌曲的歌词内容（接口直接返回LRC格式纯文本）。
- 文件组织结构为：每个歌手对应一个顶层文件夹，其下每首歌各占一个子文件夹，子文件夹内包含 `歌曲名.jpg`、`歌曲名.mp3`、`歌曲名.txt` 三个文件。

### 三、网页结构分析

#### 1. 搜索与翻页机制

- 在网站搜索框输入歌手名（如“周杰伦”），页面通过**AJAX请求**获取歌曲列表。
- 列表支持翻页，每翻一页对应一个新的请求。

#### 2. 请求参数分析

老师打开浏览器开发者工具（F12），在 Network 面板中捕获到 AJAX 请求，确认：

- **请求方法**：`POST`
- **请求地址（URL）**：`https://xxx`（具体地址见字幕中复制的地址）
- **请求头（Headers）**必须包含：
  - `User-Agent`：模拟浏览器标识。
  - `X-Requested-With`：值为 `XMLHttpRequest`（标识这是一个 AJAX 请求，否则服务端可能返回HTML页面而非JSON数据）。
- **请求体参数**共四个（均为 `input` 字段）：
  - `input`：歌手名称，用于按名称过滤。
  - `filter`：过滤类型，值为 `2`（含义未详细解释）。
  - `page`：页码，从 `1` 开始。
  - `types`：类型标识，值为 `1`（含义未详细解释）。

#### 3. 响应数据结构分析

接口返回的是 **JSON 数据**，结构如下（关键字段）：

- `author`（或 `ARTIST`）：歌手名字。
- `title`（或 `NAME`）：歌曲名称。
- `url`：MP3文件的下载地址。
- `pic`：歌曲封面图片的地址。
- `lrc`：歌词文本内容。

老师演示了验证数据正确性的方法：用浏览器直接打开 `url` 确认是歌曲音频，打开 `pic` 地址确认是封面图。

### 四、代码实现：串行版本

#### 第一步：发送请求获取歌曲列表

```python
import requests

url = "接口地址"
data = {
    "input": "周杰伦",
    "filter": "2",
    "page": "1",
    "types": "1"
}
headers = {
    "User-Agent": "浏览器UA字符串",
    "X-Requested-With": "XMLHttpRequest"
}

response = requests.post(url, data=data, headers=headers)
print(response.json())
```

老师在此处踩了两个坑：

- **坑1**：一开始访问的是返回HTML页面的地址，而非AJAX接口地址，导致响应是整个网页而非JSON数据，排查后发现是URL复制错误。
- **坑2**：请求头中缺少 `X-Requested-With` 字段，导致服务端没有返回JSON数据。加上该字段后，请求成功返回JSON。

#### 第二步：解析JSON提取歌曲信息

从响应中提取 `data["list"]`，遍历列表，逐一获取：

```python
import json

result = response.json()
songs = result["data"]["list"]
for item in songs:
    title = item.get("title")           # 歌曲名
    author = item.get("author")          # 歌手名
    mp3_url = item.get("url")            # MP3下载地址
    pic_url = item.get("pic")            # 封面图地址
    lrc = item.get("lrc")                # 歌词文本
```

老师指出：虽然演示中使用了 `json.loads()` 和 `json.dumps()` 来处理，但本案例数据结构简单，嵌套层级不深，直接用 `item.get("字段名")` 从字典取值同样可行，且更简洁。

#### 第三步：创建文件夹结构

使用 `os` 模块创建两级目录结构：

```python
import os

# 创建歌手文件夹（如果不存在）
singer_dir = "周杰伦"
if not os.path.exists(singer_dir):
    os.makedirs(singer_dir)

# 创建歌曲子文件夹（拼接歌手名和歌曲名）
song_dir = os.path.join(singer_dir, f"{author}-{title}")
if not os.path.exists(song_dir):
    os.makedirs(song_dir)
```

老师强调：

- 使用 `os.path.exists()` 判断文件夹是否已存在，避免重复创建时报错。
- 文件夹命名采用“歌手-歌曲”格式，便于管理。

#### 第四步：下载并保存文件

**保存歌词（TXT）**：

```python
with open(f"{song_dir}/{title}.txt", "w", encoding="utf-8") as f:
    f.write(lrc)
```

**保存封面图片（ JPG）**：

```python
img_response = requests.get(pic_url)
# 从URL中截取文件名或自定义
img_name = pic_url.split("/")[-1].split(".")[0] + ".jpg"
with open(f"{song_dir}/{title}.jpg", "wb") as f:
    f.write(img_response.content)
```

**保存MP3文件**：

```python
mp3_response = requests.get(mp3_url)
with open(f"{song_dir}/{title}.mp3", "wb") as f:
    f.write(mp3_response.content)
```

老师演示了下载完成后在本地打开MP3文件，确认音频内容正确。

| 阶段 | 职责 | 效率特点 |

### 六、多线程并发原理讲解

#### 1. 为什么需要并发

串行模式下，一个线程从头到尾执行完整流程：发请求→解析→存储→发请求→解析→存储……所有环节必须依次等待。老师用图示说明：请求是最耗时的环节，串行模式下大量时间浪费在等待网络响应上。

#### 2. 并发模型：生产者-消费者模式

老师介绍了基于**队列（Queue）**的并发模型，核心思路是**三个阶段解耦并行**：

```
[请求线程] → 发送请求得到数据 → 放入队列A
                                    ↓
[解析线程] ← 从队列A取数据 → 解析 → 放入队列B
                                          ↓
[存储线程] ← 从队列B取数据 → 下载文件 → 写入磁盘
```

- **队列A**：作为请求线程与解析线程之间的数据管道（生产者-消费者关系）。
- **队列B**：作为解析线程与存储线程之间的数据管道。
- 三个线程各自独立运行，互不阻塞。例如：请求线程在等待网络响应时，解析线程可以处理已收到的数据，存储线程可以写入已完成解析的数据。

#### 3. 三个线程的职责划分

| 线程 | 职责 | 关键操作 |
|------|------|----------|
| 请求线程 | 不断发请求，将原始数据放入队列A | 循环请求不同页/不同歌手 |
| 解析线程 | 从队列A取原始数据，解析后放入队列B | 提取字段，构造歌曲信息对象 |
| 存储线程 | 从队列B取歌曲信息，执行下载和写文件 | 下载MP3、图片、歌词 |

- 每个线程只专注自己的任务，不调用其他线程的方法。
- 通过队列实现线程间的数据传递，实现了解耦。

#### 4. 与单线程的对比

- **单线程**：一个人干全部流程，期间其他人（CPU、网卡、磁盘）空闲等待。
- **多线程**：三个人各司其职，同时工作，整体吞吐量大增。

老师总结：并发爬虫的本质是**I/O等待时间的多线程复用**——当请求线程等待服务器响应时，CPU可以切换到解析线程或存储线程工作，从而充分利用系统资源。

## 本P结论

1. **异步并发爬虫的核心价值**：在网络I/O密集型任务中，通过多线程并发实现请求、解析、存储三个阶段并行执行，可以显著缩短整体爬取时间。

2. **并发爬虫的副作用与对策**：并发请求会增加对目标服务器的访问频率，存在被反爬机制检测和封禁的风险。建议在实际使用中配合**代理IP池**，每次请求使用不同出口IP。

3. **多线程爬虫的实现框架**：将爬虫拆分为三个独立线程（请求线程、解析线程、存储线程），通过**队列（Queue）**作为线程间数据传递的管道，实现生产者-消费者模式，三个阶段互不阻塞、并行运作。

4. **代码封装思路**：爬虫代码应按职责分层封装为请求、解析、存储三个方法，便于后续扩展为多线程版本。

5. **文件组织规范**：使用歌手名作为顶层目录、歌曲名作为子目录，保持本地文件的结构化管理。
