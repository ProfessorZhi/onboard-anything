# P47 知识框架 - 【进阶篇】装饰器
## 本P位置

Python入门课程 → 进阶篇 → 函数高级特性

前置依赖：P46 闭包（装饰器是闭包的直接应用）
后续延伸：类装饰器、带参数的装饰器、装饰器实战（日志/计时/权限）

## 核心问题

**如何在不修改原函数代码、不改变调用方式的前提下，给函数批量添加额外功能？**

场景：登录验证、日志记录、性能计时等横切关注点

## 知识骨架

```
装饰器 = 闭包 + 约定接口
    │
    ├── 本质：接收函数 → 返回函数 的闭包
    │
    ├── 结构（三要素）
    │   ├── 外层函数：接收被装饰函数
    │   ├── 内层函数：包装逻辑 + 调用原函数
    │   └── 返回值：内层函数引用
    │
    ├── 调用方式
    │   ├── 手动包装：func = decorator(func); func()
    │   └── 语法糖：@decorator（推荐）
    │
    ├── 参数传递
    │   └── wrapper(*args, **kwargs) → func(*args, **kwargs)
    │
    └── 多装饰器叠加
        └── 执行顺序：由内到外（离函数近的先执行）
```

## 关系与依赖

```
    ↓ 实际应用

```

依赖：闭包的三层结构（外层局部变量、外层返回内层引用、内层使用外层变量）

## 输入 / 输出 / 前置 / 支撑

| 类型 | 内容 |
|------|------|
| **输入** | 被装饰的原函数 func |
| **输出** | 包装后的新函数 wrapper（行为等价，但多了额外功能） |
| **前置** | 闭包的三要素：外层局部变量、内层引用外层变量、外层返回内层引用 |
| **支撑** | *args, **kwargs（参数透传）、函数一等公民（可作为参数和返回值） |

## 易错点

1. **多装饰器顺序**：离函数近的先执行，后写的装饰器先执行。示例：`@a @b def f()` → 先执行 b，再执行 a
2. **丢失返回值**：内层函数必须 return func(*args, **kwargs)，否则调用方收不到原函数返回值
3. **参数写死**：wrapper 不使用 *args, **kwargs，则只能装饰单一参数签名的函数，失去通用性
4. **语法糖等效误解**：@decorator 等价于 f = decorator(f)，不是 decorator(f())
