# P48 资源清单 - 【进阶篇】单例模式

> 本P涉及的外部资源均为 Python 内置机制与语言特性，无第三方库或外部平台依赖。资源数量虽少，但均为理解单例模式实现的核心支撑。

---

## 一、Python 内置类

### `object` 类

| 字段 | 内容 |
|---|---|
| **名称** | `object` |
| **类型** | Python 内置顶级父类（Built-in Base Class） |
| **在本P中的作用** | 在重写 `__new__` 时，必须调用 `object.__new__(cls)` 来完成真正的内存分配并返回对象引用；若不调用父类 `__new__`，则无法创建对象，`__init__` 也不会被触发 |
| **来源** | 字幕直接提到 |
| **建议方向** | 查阅 Python 官方文档中 `object.__new__` 的签名与行为；尝试不同参数组合（`*args`、`**kwargs`）下调用父类 `__new__` 的写法；探索多继承场景下 `object.__new__` 的 MRO 调用链 |

---

## 二、Python 内置函数

### `hasattr()`

| 字段 | 内容 |
|---|---|
| **名称** | `hasattr()` |
| **类型** | Python 内置函数（Built-in Function） |
| **在本P中的作用** | 在 `__init__` 中检查对象是否已存在指定属性，用于防止单例对象在多次实例化时被重复初始化、后赋值覆盖先赋值 |
| **来源** | 字幕直接提到 |
| **建议方向** | 与 `getattr()`、`setattr()`、`delattr()` 对比使用，理解四者构成的属性反射（Reflection）体系；尝试用 `hasattr` 替代 `try/except AttributeError` 写法，对比两种防御性编程风格的适用场景 |

### `id()`

| 字段 | 内容 |
|---|---|
| **名称** | `id()` |
| **类型** | Python 内置函数（Built-in Function） |
| **在本P中的作用** | 返回对象的内存地址（CPython 实现中为十进制整数），本P用于验证两次实例化的结果是否为同一对象——`id(obj1) == id(obj2)` 为 `True` 即证明单例成立 |
| **来源** | 字幕直接提到 |
| **建议方向** | 理解 `id()` 与 `is` 运算符的等价关系（`a is b` 等价于 `id(a) == id(b)`）；在单例验证中优先使用 `obj1 is obj2` 替代 `id` 比较，养成更 Pythonic 的写法习惯 |

## 三、Python 语言机制

### `__new__` 魔术方法

| 字段 | 内容 |
|---|---|
| **名称** | `__new__` |
| **类型** | Python 特殊方法 / 魔术方法（Dunder Method） |
| **在本P中的作用** | 对象实例化的第一步，负责在内存中为对象分配空间并返回对象引用；本P通过重写此方法实现单例模式的核心逻辑——用类属性 `_instance` 记录唯一实例，首次创建后直接复用 |
| **来源** | 字幕直接提到 |
| **建议方向** | 阅读 Python 数据模型文档（Data Model）中关于 `__new__` 的完整说明；尝试在元类（`metaclass`）中重写 `__new__` 实现单例，与本P的类内重写方式对比；探索 `__new__` 在不可变类型（如 `int`、`str`、`tuple`）子类化时的必要性 |

### `__init__` 魔术方法

| 字段 | 内容 |
|---|---|
| **名称** | `__init__` |
| **类型** | Python 特殊方法 / 魔术方法（Dunder Method） |
| **在本P中的作用** | 对象实例化的第二步，接收 `__new__` 返回的对象引用作为 `self`，负责属性赋值与初始化；本P重点演示其与 `__new__` 的执行顺序，并说明在单例场景下需配合 `hasattr` 防止重复初始化 |
| **来源** | 字幕直接提到 |
| **建议方向** | 与 `__new__` 配对练习，手写执行顺序追踪代码；尝试 `__init__` 无 `return` 但故意写 `return non-None` 时 Python 的报错行为，加深对返回值约束的理解 |

### `.pyc` 编译文件 / Python 模块导入机制

| 字段 | 内容 |
|---|---|
| **名称** | `.pyc` 文件 / 模块导入缓存机制 |
| **类型** | Python 运行时机制（Runtime Mechanism） |
| **在本P中的作用** | 说明 Python 模块天然具备单例特性：首次 `import` 时执行模块代码并生成 `.pyc` 缓存，后续 `import` 直接加载缓存而不重新执行；利用此特性可在模块文件中预实例化对象，作为无需重写 `__new__` 的简易单例替代方案 |
| **来源** | 字幕直接提到 |
| **建议方向** | 在文件系统中定位 `__pycache__` 目录，观察 `.pyc` 文件的命名规则（含 Python 版本与哈希信息）；用 `importlib.reload()` 验证强制重载模块时对象 `id` 的变化，理解模块缓存的边界；对比模块单例与 `__new__` 单例在多线程场景下的安全性差异（可作为进阶探索方向） |

## 四、设计模式

### 单例模式（Singleton Pattern）

| 字段 | 内容 |
|---|---|
| **名称** | 单例模式 |
| **类型** | 软件设计模式（Creational Design Pattern） |
| **在本P中的作用** | 本P的核心主题，属于创建型设计模式；通过限制类的实例数量为一，解决资源复用与数据一致性两类工程问题；本P给出两种 Python 实现：重写 `__new__` + 类属性 `_instance`，以及利用模块导入缓存 |
| **来源** | 字幕直接提到 |
| **建议方向** | 查阅《设计模式：可复用面向对象软件的基础》（GoF）中单例模式的原始定义，与本P的 Python 实现对照；尝试用装饰器（`decorator`）和元类（`metaclass`）分别实现单例，整理四种实现方式的优缺点对比表；在数据库连接、日志管理器等真实场景中练习单例的落地写法 |
