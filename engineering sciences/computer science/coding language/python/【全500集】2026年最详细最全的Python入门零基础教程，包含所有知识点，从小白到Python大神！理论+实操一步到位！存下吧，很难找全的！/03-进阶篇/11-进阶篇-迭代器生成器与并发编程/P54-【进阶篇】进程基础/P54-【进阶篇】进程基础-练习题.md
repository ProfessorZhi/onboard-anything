## 结构识别

**Q1｜进程五态排序**
将以下五种状态按典型生命周期顺序排列，并说明哪两个状态之间存在双向转换：

> 运行态、终止态、等待态、就绪态、新建态

<details>
<summary>参考答案</summary>

新建态 → 就绪态 → 运行态 → 终止态；其中运行态与等待态（阻塞态）之间可双向转换：运行中遇到 IO 等事件进入等待态，事件完成后重新回到就绪态（而非直接回到运行态）。

</details>

下列场景各适合用「进程」还是「线程」？写出选择并说明理由（一句话即可）。

| 场景 | 选择 | 理由 |
|------|------|------|
| 批量下载 1000 个网络图片 | | |
| 对 10GB 视频文件做帧级转码 | | |
| 爬虫等待服务器响应的同时写日志 | | |
| 利用 8 核 CPU 并行执行矩阵运算 | | |

<details>
<summary>参考答案</summary>

| 场景 | 选择 | 理由 |
|------|------|------|
| 批量下载图片 | 线程 | IO 密集型，GIL 在等待期间会释放，线程切换成本低 |

| 矩阵并行运算 | 进程 | CPU 密集型，进程可真正并行于多个核心 |

</details>

## 机制理解

**Q3｜全局变量隔离验证**
阅读以下代码，**不运行**，预测输出结果并解释原因：

```python
import multiprocessing

count = 0

def add():
    global count
    count += 10
    print(f"子进程 count = {count}")

if __name__ == '__main__':
    p = multiprocessing.Process(target=add)
    p.start()
    p.join()
    print(f"主进程 count = {count}")
```

预测输出（填空）：
- 子进程打印：`__________`
- 主进程打印：`__________`
- 原因（一句话）：`__________`

<details>
<summary>参考答案</summary>

- 子进程打印：`子进程 count = 10`
- 主进程打印：`主进程 count = 0`
- 原因：每个进程拥有全局变量的**独立副本**，子进程修改 `count` 不影响主进程的 `count`。

</details>

**Q4｜`if __name__ == '__main__'` 的必要性**
在 Windows 上，若去掉 `if __name__ == '__main__'` 保护，直接在模块顶层调用 `p.start()`，会发生什么？请从「脚本重新导入」这一机制出发解释。

<details>
<summary>参考答案</summary>

Windows 使用 `spawn` 方式创建子进程，子进程启动时会**重新导入（import）当前脚本**。若顶层有 `p.start()`，子进程导入脚本后又会执行 `p.start()`，再次生成孙进程……如此循环，形成无限递归式的进程爆炸，最终耗尽系统资源并崩溃。加上 `if __name__ == '__main__'` 后，子进程导入脚本时 `__name__` 为模块名而非 `'__main__'`，保护块不会被执行，递归得以终止。

</details>

## 最小实践

**Q5｜双进程并发演示**
编写一个脚本，要求：

1. 定义两个函数 `sing()` 和 `dance()`，各自打印 3 次（含进程 PID），每次间隔 0.5 秒。
2. 在主进程中分别创建两个子进程执行上述函数。
3. 主进程等待所有子进程结束后，打印 `"主进程结束，PID: {主进程PID}"`。
4. 必须加 `if __name__ == '__main__'` 保护。

验收标准：运行后两个函数的输出**交替出现**（而非一个全部完成再执行另一个），且 PID 各不相同。

<details>
<summary>参考实现</summary>

```python
import multiprocessing
import os
import time

def sing():
    for i in range(3):
        print(f"唱歌 第{i+1}次 PID={os.getpid()}")
        time.sleep(0.5)

def dance():
    for i in range(3):
        print(f"跳舞 第{i+1}次 PID={os.getpid()}")
        time.sleep(0.5)

if __name__ == '__main__':
    p1 = multiprocessing.Process(target=sing, name="singer")
    p2 = multiprocessing.Process(target=dance, name="dancer")
    p1.start()
    p2.start()
    p1.join()
    p2.join()
    print(f"主进程结束，PID: {os.getpid()}")
```

</details>

## 扩展思考

**Q6｜进程稳定性的代价**
「单进程崩溃不影响其他进程」被列为进程相对于线程的优点，但这一特性在实际工程中会带来什么新问题？结合「进程不共享全局资源」这一特点，说明为什么引入 IPC（如 Queue）是必然的，而不是可选的。

<details>
<summary>参考思路</summary>

稳定性来自隔离，但隔离本身制造了两个新问题：

1. **状态同步问题**：主进程无法直接感知子进程的运行结果（如计算产出的数据），因为子进程修改的全局变量是副本，主进程中的原始变量不受影响。要获取结果，必须有专门的通信信道。
2. **协调成本问题**：多个独立进程若需要协作（如生产者-消费者模型），必须通过 IPC 传递信号和数据，否则各自为政，无法形成工作流水线。

因此 Queue 等 IPC 机制不是锦上添花，而是在进程隔离设计下**弥补通信能力缺失的必要补丁**。这也是下一节课 Queue 紧跟本节的原因。

</details>
