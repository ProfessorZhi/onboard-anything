# P54 知识框架 - 【进阶篇】进程基础
## 本P位置

进阶篇 → 并发编程 → 进程基础

- 前置：P53 线程基础（理解线程短板才能理解进程动机）
- 后续：Queue 进程间通信（进程隔离导致必须引入 IPC 机制）
- 定位：承上启下，建立「独立资源空间」这一核心认知

---

## 核心问题

**进程如何突破线程局限，如何正确创建与管理进程？**

---

## 知识骨架

### 节点一：为什么需要进程
线程两大局限 → 资源竞争 + GIL 锁死多核 → 进程作为重量级替代方案

### 节点二：进程是什么
**进程 = 正在运行的程序**。每个进程拥有独立资源空间（内存、CPU），彼此隔离。

### 节点三：进程 vs 线程（六维对比）

| 维度 | 进程 | 线程 |
|------|------|------|
| 资源 | 独立空间，占用高 | 共享资源，占用低 |
| 通信 | 必须借助 IPC，难度高 | 直接共享，难度低 |
| 效率 | 创建/切换成本高，较慢 | 成本低，较快 |
| 多核 | 真正并行，无 GIL | GIL 限制，假并行 |
| 稳定 | 单进程崩溃不影响其他 | 单线程崩溃可能导致全崩 |
| 适用 | CPU 密集型 | IO 密集型 |

**选型法则**：CPU 密集型 → 进程；IO 密集型 → 线程。

### 节点四：进程五种状态
新建态 → 就绪态 → 运行态 → 等待态/阻塞态 → 终止态

### 节点五：进程创建与管理
**核心 API**：`multiprocessing.Process`

- `target`：指定任务函数（**不加括号**，加括号表示调用）
- `args` / `kwargs`：传参
- `p.start()`：启动进程
- `p.join()`：主进程阻塞等待子进程

**必须加 `if __name__ == '__main__'`**：Windows 下 fork 机制会重新导入脚本，无保护会导致无限递归创建进程。

### 节点六：进程属性

| 属性/方法 | 含义 |
|-----------|------|
| `p.name` | 进程名称 |
| `p.pid` | 进程 ID |
| `p.is_alive()` | 存活状态 |
| `os.getpid()` | 当前进程 ID |
| `os.getppid()` | 父进程 ID |

### 节点七：进程不共享全局资源（核心区别）
- 进程拥有全局变量的**独立副本**
- 修改全局变量只影响自身，不影响其他进程和主进程
- 优点：无资源竞争，无需加锁
- 缺点：无法通过全局变量通信

### 节点八：进程间通信需求
进程隔离 → 必须引入 IPC 机制 → `multiprocessing.Queue`（先进先出，安全）

## 输入 / 输出 / 前置 / 支撑

**输入**：任务函数、`if __name__ == '__main__'` 保护块、参数（args 或 kwargs）

**输出**：子进程 PID、进程名称、存活状态、主进程最终全局变量值

**前置知识**：线程基础（GIL 概念、线程创建语法可类比）

**支撑工具**：
- `multiprocessing.Process`：创建进程的核心类
- `os.getpid()` / `os.getppid()`：进程 ID 查询
- `time.sleep()`：模拟任务耗时
- `p.is_alive()`：存活状态查询

---

## 易错点

1. **Windows 必须加 `if __name__ == '__main__'`**，否则递归创建进程导致崩溃
2. **`target` 参数不加括号**：`target=sing` 不是 `target=sing()`
3. **启动用 `.start()`**，不是调用函数，否则顺序执行失去并发效果
4. **全局变量在多进程中不共享**：子进程修改全局变量不会影响主进程，每个进程持独立副本
5. **进程创建成本远高于线程**：独立分配内存和资源，不要误以为和线程一样轻量
