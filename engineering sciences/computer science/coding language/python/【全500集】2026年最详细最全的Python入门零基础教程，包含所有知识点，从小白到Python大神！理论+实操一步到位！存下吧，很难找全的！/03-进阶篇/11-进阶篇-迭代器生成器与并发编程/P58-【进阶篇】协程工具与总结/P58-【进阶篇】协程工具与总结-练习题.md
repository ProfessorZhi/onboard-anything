## 结构识别

### 题目 1.1：协程工具链层级关系

以下四个工具按依赖层级从底层到上层排列，正确的是：

A. 生成器 → GEVENT → GREENLET → IO操作

B. 生成器 → GREENLET → GEVENT → IO操作

C. GREENLET → 生成器 → GEVENT → monkey.patch_all

D. monkey.patch_all → GEVENT → GREENLET → 生成器

**答案**：B

**解析**：生成器是GREENLET的底层实现，GREENLET是GEVENT的底层实现，三者构成依赖链。monkey.patch_all是补丁层，用于扩展GEVENT的IO识别能力。

---

### 题目 1.2：切换机制分类

| 工具 | 切换触发方式 | IO阻塞自动处理 |
|------|--------------|----------------|
| ① | next()手动 | 否 |
| ② | switch()手动 | 否 |
| ③ | 遇到IO自动 | 是 |

表格中①②③分别对应哪个工具？

**答案**：①生成器 ②GREENLET ③GEVENT

**解析**：生成器需要手动next()驱动，GREENLET需要手动switch()切换，两者遇到IO阻塞都不会自动切。GEVENT注册IO事件后，遇到IO自动触发切换。

## 机制理解

### 题目 2.1：monkey.patch_all()位置问题

以下代码执行后，打印结果是什么？

```python
import gevent

def task(name):
    gevent.sleep(1)
    print(f"{name} done")

g1 = gevent.spawn(task, "A")
g1.join()
print("main end")
```

```python
# 位置1：文件顶部
from gevent import monkey
monkey.patch_all()

# 位置2：其他import之后
import time
import requests
```

如果把 `monkey.patch_all()` 从位置1移到位置2，代码行为是否改变？

**答案**：此代码行为不变，因为使用的是 gevent.sleep() 而非 time.sleep()

**解析**：monkey.patch_all() 的位置要求仅在代码使用 time.sleep()、requests 等标准库/第三方库IO时才有影响。直接使用 gevent.sleep() 不需要补丁。

### 题目 2.2：greenlet 参数传递

以下代码运行后会输出什么？

```python
from greenlet import greenlet

def eat():
    print("吃")
    g2.switch()
    print("吃完")

def drink():
    print("喝")
    g1.switch()

g1 = greenlet(eat)
g2 = greenlet(drink)

g1.switch()
print("结束")
```

A. 吃 → 喝 → 吃完 → 结束

B. 吃 → 喝 → 结束 → 吃完

C. 吃 → 吃完 → 喝 → 结束

**答案**：A

**解析**：`greenlet(函数名)` 传入的是函数引用，而非调用函数。执行 `g1.switch()` 后 eat() 开始运行，打印"吃"，然后手动 switch 到 g2，打印"喝"，再 switch 回来继续执行，打印"吃完"。

### 题目 2.3：自动切换原理

GEVENT 实现自动 IO 切换的核心机制是什么？

**答案**：GEVENT 通过 monkey.patch_all() 将系统标准库的 IO 函数（如 time.sleep、socket、requests 等）替换为 GEVENT 可识别的版本。这些被替换的函数内部会在阻塞前自动触发协程切换，让出执行权给其他任务。

**关键词**：事件注册、绿线调度、协作式多任务

## 最小实践

### 题目 3.1：修复 greenlet 创建错误

以下代码存在一个问题，运行会报错。请指出并改正：

```python
from greenlet import greenlet

def task1():
    print("task1 start")
    g2.switch()
    print("task1 end")

def task2():
    print("task2 start")
    g1.switch()
    print("task2 end")

g1 = greenlet(task1())
g2 = greenlet(task2())
g1.switch()
```

**答案**：

错误：传入的是函数调用结果 `task1()`，而非函数引用 `task1`

修改：
```python
g1 = greenlet(task1)  # 不带括号
g2 = greenlet(task2)  # 不带括号
```

**解析**：`greenlet(函数)` 接收函数对象，调用时由 greenlet 内部执行。带括号会立即执行函数并传入返回值。

### 题目 3.2：实现并发任务计时

使用 gevent 实现：同时启动 3 个任务，每个任务 sleep 2 秒，打印开始和完成消息，最后打印总耗时。

```python
# 补全代码
import gevent

def task(name, delay):
    # TODO: 实现任务
    pass

# 期望输出格式：
# task1 开始
# task2 开始
# task3 开始
# task1 完成 (delay=Xms)
# task2 完成
# task3 完成
# 总耗时: X.XX秒
```

**参考答案**：

```python
import gevent
import time

def task(name, delay):
    start = time.time()
    print(f"{name} 开始")
    gevent.sleep(delay)
    print(f"{name} 完成 (delay={(time.time()-start)*1000:.0f}ms)")

start_time = time.time()
g1 = gevent.spawn(task, "task1", 2)
g2 = gevent.spawn(task, "task2", 2)
g3 = gevent.spawn(task, "task3", 2)

g1.join()
g2.join()
g3.join()

print(f"总耗时: {time.time() - start_time:.2f}秒")
```

**关键点**：`gevent.sleep()` 触发自动切换，`spawn()` 创建并启动，`join()` 等待完成。

## 扩展思考

### 题目 4.1：场景选型

某爬虫项目需要并发请求 100 个 API，响应时间约 200ms。以下两种方案各有何优劣？

**方案A**：使用 GEVENT + monkey.patch_all()

**方案B**：使用 Python 3.5+ 的 asyncio + aiohttp

**参考答案**：

| 维度 | GEVENT | asyncio |
|------|--------|---------|
| 语法 | 接近同步代码 | 原生 async/await |
| 兼容性 | 需打补丁，部分库可能冲突 | 标准语法，生态统一 |
| 性能 | 高（底层C实现） | 高（Python异步生态） |
| 学习曲线 | 较低（同步写法） | 较高（需理解异步概念） |
| 现代项目推荐度 | 低（维护放缓） | 高（官方推荐） |

**结论**：新项目推荐 asyncio，老项目或快速迁移可选 GEVENT。

### 题目 4.2：补丁失效分析

某项目在使用 gevent 时发现 `time.sleep()` 没有被自动切换，整个程序仍然串行执行。排查后发现是项目结构导致的：

```
project/
├── config.py
├── models.py
├── utils.py
└── main.py
```

其中 `main.py` 开头有 `monkey.patch_all()`，但 `models.py` 中先 import 了一个第三方库 `from external_lib import Client`。

请分析为什么会失效，以及如何解决。

**参考答案**：

**原因**：`external_lib` 在 `monkey.patch_all()` 之前被导入，它内部使用的 IO 函数（如 socket）已经被缓存为原始版本，后续打补丁无法影响已导入的模块。

**解决方案**：

1. **方案一**：确保补丁在最开头
   ```python
   # main.py
   from gevent import monkey
   monkey.patch_all()  # 第一行！
   
   import config  # 然后再导入其他模块
   from models import Model
   ```

2. **方案二**：在 external_lib 导入前打补丁
   ```python
   from gevent import monkey
   monkey.patch_all()
   
   from external_lib import Client  # 必须在补丁之后
   ```

3. **方案三**：使用 gevent 启动器（推荐生产环境）
   ```bash
   geventrun main.py
   # 或
   python -m gevent monkey
   ```

### 题目 5.1：asyncio 等价实现

不安装 gevent，使用 Python 标准库 asyncio 重写以下 gevent 代码：

```python
# gevent 版本
import gevent

def fetch(url):
    print(f"请求: {url}")
    gevent.sleep(1)  # 模拟IO
    print(f"完成: {url}")

gevent.joinall([
    gevent.spawn(fetch, "http://api1.com"),
    gevent.spawn(fetch, "http://api2.com"),
    gevent.spawn(fetch, "http://api3.com"),
])
```

**参考答案**：

```python
import asyncio

async def fetch(url):
    print(f"请求: {url}")
    await asyncio.sleep(1)  # 模拟IO
    print(f"完成: {url}")

async def main():
    await asyncio.gather(
        fetch("http://api1.com"),
        fetch("http://api2.com"),
        fetch("http://api3.com"),
    )

asyncio.run(main())
```

**核心映射**：

| gevent | asyncio |
|--------|---------|
| `greenlet` | `协程` |
| `gevent.sleep()` | `await asyncio.sleep()` |
| `gevent.spawn()` | `asyncio.create_task()` |
| `gevent.joinall()` | `asyncio.gather()` |
| `monkey.patch_all()` | 不需要（原生语法） |

### 题目 5.2：工具选择决策

根据以下场景，选择使用 GREENLET、GEVENT 还是 asyncio：

| 场景 | 推荐工具 | 理由 |
|------|----------|------|
| 遗留项目快速添加并发 | ① | ② |
| 新项目需要高并发IO | ③ | ④ |
| 需要精细控制切换时机 | ⑤ | ⑥ |
| 对接异步数据库驱动 | ⑦ | ⑧ |

**答案**：

| 场景 | 推荐工具 | 理由 |
|------|----------|------|
| 遗留项目快速添加并发 | GEVENT | 一行补丁，改动最小 |
| 新项目需要高并发IO | asyncio | 官方标准，生态完善 |
| 需要精细控制切换时机 | GREENLET | 纯手动，想何时切就何时切 |
| 对接异步数据库驱动 | asyncio | 驱动原生支持async/await |
