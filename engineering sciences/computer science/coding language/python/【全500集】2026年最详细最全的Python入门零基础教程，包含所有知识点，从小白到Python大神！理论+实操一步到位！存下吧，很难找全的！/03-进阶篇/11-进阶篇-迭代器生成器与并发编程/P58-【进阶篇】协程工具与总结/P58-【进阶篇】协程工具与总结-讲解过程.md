# P58 讲解过程 - 【进阶篇】协程工具与总结
## 本P定位

本P是协程工具的进阶讲解与章节总结。老师依次介绍了两个协程工具库——GREENLET（手动切换）和GEVENT（自动切换），通过代码演示了两者的使用方式、切换机制、优缺点，并总结了协程的应用场景与选型建议。

---

## 讲解过程

### 一、GREENLET 工具介绍（手动切换协程）

**1. GREENLET 库概述**
- GREENLET 是一个第三方协程库
- 基于生成器封装
- 提供了 switch 方法，能够更直观地手动切换协程
- 相比生成器的 next 函数更加简洁

**2. 安装 GREENLET**
- 命令：`pip install greenlet`
- 老师演示：在 CMD 中通过 Windows+R 输入 CMD 回车，输入 pip install greenlet
- 安装成功后显示 "successfully installed greenlet"
- 第一次安装可能较慢，因为之前已安装过所以演示时速度较快

**3. GREENLET 基本使用**

导入方式：
```python
from greenlet import greenlet
```

创建两个任务函数：

```python
# 任务一：吃水果
def eat_fruit():
    print("准备吃苹果 🍎")
    g2.switch()  # 手动切换到喝水的任务
    time.sleep(1)
    print("苹果吃完了")

# 任务二：喝水
def drink_water():
    print("准备喝水 💧")
    time.sleep(1)
    print("喝完水了")
```

创建协程对象并运行：
```python
if __name__ == "__main__":
    g1 = greenlet(eat_fruit)  # 创建协程对象，指定任务函数，不要加小括号
    g2 = greenlet(drink_water)
    
    g1.switch()  # 启动第一个协程（吃水果）
    g2.switch()  # 手动切换到喝水
    g1.switch()  # 切换回吃水果
    g2.switch()  # 再次切换到喝水
```

**4. 运行效果分析**
- 程序按照手动切换的顺序执行
- 准备吃水果 → 切换到喝水 → 切换回吃水果（从上次位置继续）→ 打印"苹果吃完了" → 切换回喝水 → 打印"喝完水了"

**5. GREENLET 优缺点总结**
- 优点：切换逻辑比生成器更清晰，switch 方法更直观，不用写 yield
- 缺点：需要手动调用 switch 方法切换任务；遇到 IO 阻塞（如 time.sleep）不会自动切换；如果一个任务卡在 IO 等待，整个线程就会卡住
- 适用场景：更适合学习协程的切换逻辑，实际开发中更推荐使用 GEVENT

### 二、GEVENT 工具介绍（自动切换协程）

**1. GEVENT 库概述**
- GEVENT 是能够自动切换协程的工具，是实际项目中最常用的协程库
- 基于 greenlet 实现，更强大的协程库
- 核心优势：遇到 IO 操作时能够自动切换协程，不用手动调用 switch 方法
- 遇到 gevent.sleep、网络请求、文件读写等 IO 操作时能够自动切换到其他任务，完成后再切换回来

**2. 安装 GEVENT**
- 命令：`pip install gevent`
- 出现 "successfully installed" 或 "requirement already satisfied" 表示安装成功

**3. GEVENT 核心方法**

| 方法 | 作用 |
|------|------|
| `gevent.spawn(函数, 参数)` | 创建协程对象并启动任务，参数以元组形式传递 |
| `gevent.sleep()` | 模拟 IO 阻塞操作，GEVENT 能够识别，会触发自动切换 |
| `协程对象.join()` | 阻塞主程序，等待该协程执行完毕 |
| `gevent.joinall(列表)` | 等待多个协程全部执行完毕，参数为包含多个协程对象的列表 |
| `gevent.monkey.patch_all()` | 打补丁，把 time.sleep、requests.get 等系统 IO 函数替换为 GEVENT 可识别的版本 |

**4. GEVENT 基本使用示例一（使用 gevent.sleep）**

导入模块：
```python
import gevent
```

定义任务函数：
```python
def download_image(image_name):
    print(f"开始下载图片 {image_name}")
    gevent.sleep(2)  # 模拟 IO 阻塞（等待网络响应），这里会触发自动切换
    print("图片下载完成 ✅")

def process_data(data_name):
    print(f"开始处理数据 {data_name}")
    gevent.sleep(2)  # 同样是模拟 IO 阻塞
    print("数据处理完成 ✅")
```

创建并运行协程：
```python
if __name__ == "__main__":
    g1 = gevent.spawn(download_image, "风景.png")
    g2 = gevent.spawn(process_data, "用户数据.csv")
    
    gevent.joinall([g1, g2])  # 等待所有协程执行完毕
    print("所有任务都完成了 🎉")
```

**运行效果分析**：
- 先执行 download_image，遇到 gevent.sleep 自动切换到 process_data
- process_data 遇到 gevent.sleep 又切换回 download_image
- 继续执行直到完成，总耗时约 2 秒（等于最长 IO 等待时间）
- 相比串行执行（4 秒）效率大幅提升

**5. GEVENT 基本使用示例二（使用系统 IO 函数 + monkey.patch_all）**

场景说明：
- 使用 time.sleep 这种系统的 IO 函数，GEVENT 默认无法识别，不会触发自动切换
- 这时需要使用 monkey.patch_all() 打补丁

定义任务（使用 time.sleep）：
```python
import gevent

def task(name):
    for i in range(1, 4):
        print(f"任务 {name} 开始执行第 {i} 次")
        time.sleep(2)  # 系统的 IO 函数，默认 GEVENT 无法识别
```

运行（不打补丁）：
```python
if __name__ == "__main__":
    g1 = gevent.spawn(task, "A")
    g2 = gevent.spawn(task, "B")
    g3 = gevent.spawn(task, "C")
    
    gevent.joinall([g1, g2, g3])
    print("所有任务执行完毕")
```

**问题演示**：不打补丁时，任务顺序执行（A 全部完成 → B 全部完成 → C 全部完成），没有实现协程切换

**解决方案：打补丁**
```python
from gevent import monkey
monkey.patch_all()  # 放在文件最开头，导入其他模块之前
```

打补丁后的效果：
- 任务 A、B、C 一起执行第一次
- 遇到 IO 阻塞时自动切换到其他任务
- 最终实现真正的自动切换并发执行

**6. monkey.patch_all() 关键注意事项**

- 建议放在文件最开头、导入其他模块之前，否则可能替换无效
- 打补丁后原来的系统 IO 函数用法不变，但底层已变成 GEVENT 实现
- 打补丁后不仅 time.sleep 可识别，常用第三方库如 requests、urllib 的 IO 操作也能被 GEVENT 识别
- 实际爬虫场景中的网络请求 IO 操作也会被识别

### 三、协程应用场景

协程的优势在 IO 密集型高并发场景中体现得淋漓尽致，主要应用于以下四个场景：

| 场景 | 说明 |
|------|------|
| **网络爬虫** | 爬取大量网页时，每个网页请求都要等待服务器响应（IO 阻塞）。用协程可同时发起上千个请求，不用逐个等待，爬取效率提升十倍以上 |
| **异步接口服务** | 处理大量 HTTP 请求时，API 接口等待数据库查询、第三方服务响应可自动切换到其他请求，提高并发处理能力，支持更多用户同时访问 |
| **文件批量处理** | 批量读取或写入多个文件时，IO 等待时间占比较高。协程可以在一个文件等待 IO 时切换到其他文件操作，减少整体等待时间 |
| **消息队列消费** | 同时消费多个消息队列的消息，某个队列没有消息（IO 阻塞）时切换到其他队列消费，提高消息处理效率，避免队列消息堆积 |

### 四、本小节核心内容总结

**1. 协程的本质**
- 单个线程内的状态保存 + 主动切换
- 轻量级的多任务方案，不依赖操作系统调度，没有系统调度的开销
- 核心优势：资源占用极低、切换高效、无锁竞争、支持百万级高并发
- 仅限 IO 密集型场景

**2. 三种实现方式对比**

| 方式 | 切换方式 | 适用场景 |
|------|----------|----------|
| **生成器** | 手动（yield + next） | 理解原理，基础学习 |
| **GREENLET** | 手动（switch） | 学习协程切换逻辑，逻辑清晰但不够智能 |
| **GEVENT** | 自动（遇到 IO 自动切换） | 实际开发，推荐使用 |

**3. GEVENT 关键方法速记**
- `gevent.spawn()` - 创建协程
- `gevent.sleep()` - 触发自动切换的 IO 阻塞
- `gevent.joinall()` - 等待多个协程完成
- `gevent.monkey.patch_all()` - 替换系统 IO 函数为 GEVENT 可识别版本

**4. 选型建议**
- IO 密集型高并发场景（如爬虫、接口服务）→ 用协程，最好用 GEVENT
- CPU 密集型场景（如大规模计算、数据分析）→ 用进程或进程池
- 普通并发简单场景（如少量文件读写）→ 用线程

**5. 课后实践建议**
- 尝试用 GEVENT 写一个简单的多网页爬虫
- 用协程批量处理文件
- 感受协程的高并发优势，多练几次即可熟练掌握

## 本P结论

1. **GREENLET** 是手动切换协程的第三方库，基于生成器封装，提供 switch 方法。使用时需手动调用 switch 进行协程切换，逻辑清晰但遇到 IO 阻塞不会自动切换，适合学习协程切换原理。

2. **GEVENT** 是自动切换协程的工具，基于 greenlet 实现，是实际项目中最推荐使用的协程库。核心优势在于遇到 IO 操作时能够自动切换协程，gevent.sleep() 可触发自动切换，monkey.patch_all() 可让系统 IO 函数（如 time.sleep、requests.get）也被 GEVENT 识别，从而实现真正的自动切换并发。

3. **协程本质**是单个线程内的状态保存加主动切换，是轻量级的多任务方案，不依赖操作系统调度，无系统调度开销，支持百万级高并发，但仅适用于 IO 密集型场景。

4. **选型原则**：IO 密集型高并发用协程（GEVENT）、CPU 密集型用进程、简单并发场景用线程。
