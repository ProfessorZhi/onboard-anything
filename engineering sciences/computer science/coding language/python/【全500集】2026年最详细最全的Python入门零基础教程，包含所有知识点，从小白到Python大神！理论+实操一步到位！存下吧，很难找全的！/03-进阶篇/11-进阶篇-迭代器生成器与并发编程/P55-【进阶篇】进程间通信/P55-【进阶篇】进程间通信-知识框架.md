# P55 知识框架 - 【进阶篇】进程间通信
## 本P位置

Python入门教程 → 进阶篇 → 多进程编程 → 进程间通信

本P承接P54进程创建与启动，引入进程间通信问题，是多进程编程从"能创建进程"到"能用进程协作"的关键跨越。

## 核心问题

多个独立进程如何安全传递数据？

## 知识骨架

```
multiprocessing.Queue
├── put()    → 阻塞式存入，队列满时等待
├── get()    → 阻塞式取出，队列空时等待
├── empty()  → 非阻塞判空
├── full()   → 非阻塞判满
└── qsize()  → 非阻塞获取长度
```

**核心特性**：阻塞同步机制（队列满/空时自动等待）、FIFO顺序

**典型模式**：生产者-消费者模型

```
生产进程 ──put()──→ [Queue] ──get()──→ 消费进程
```

**执行控制**：join()控制进程执行顺序，实现串行化

## 输入 / 输出 / 前置 / 支撑

| 类型 | 内容 |
|------|------|
| 前置知识 | 进程创建（multiprocessing.Process）、if __name__ == "__main__" |
| 输入 | 生产者向队列put()的数据（任意Python对象） |
| 输出 | 消费者从队列get()的数据（与存入顺序一致） |
| 支撑工具 | time.sleep()模拟生产/消费耗时 |
| 后续扩展 | pipe管道、共享内存、信号量等进程同步机制 |

## 易错点

**阻塞等待 ≠ 死循环**

队列满时put()、队列空时get()的等待是操作系统层面的进程挂起，不消耗CPU，不是代码错误。区分标志：程序在等待IO，而非CPU空转。

**empty()/full()返回False不保证下一刻状态**

判断与使用之间存在时间窗口，结果不保证准确性。生产-消费场景中用empty()判断退出时需格外注意。

**队列必须通过参数传递给子进程**

子进程函数接收队列对象作为参数，而非在子进程中重新创建，否则各进程持有不同队列实例，无法通信。

**join()控制的是主进程，而非子进程间顺序**

join()阻塞调用进程直到指定子进程结束，若在两个start()后分别join()，则顺序等待。

## 外部资源与对象

| 对象 | 类型 | 关系 |
|------|------|------|
| multiprocessing.Queue | 标准库类 | 本P核心对象，提供进程安全的队列数据结构 |
| multiprocessing.Process | 标准库类 | 用于创建子进程执行生产/消费任务（前置依赖） |
| time | 标准库模块 | 提供sleep()，模拟生产/消费的时间间隔 |

**注**：Queue相比pipe和共享内存，是进程间通信最推荐的方式，因其更安全（自动同步）且接口简单。
