# P53 资源清单 - 【进阶篇】线程核心与线程同步

> 本P资源均属 Python 标准库范畴，无需额外安装。来源标注说明：
> - 📌 字幕直接提到
> - 🔍 基于上下文推断

---

## 一、模块

### threading

| 字段 | 内容 |
|------|------|
| 名称 | threading |
| 类型 | Python 标准库模块 |
| 在本P中的作用 | 多线程编程的核心基础设施，提供 Thread 类与 Lock 类；本P通过 `import threading` 引入，贯穿所有演示代码 |
| 来源 | 📌 字幕直接提到 |
| 官方文档 | https://docs.python.org/zh-cn/3/library/threading.html |

**建议方向**
- 阅读官方文档中 threading 模块的完整 API 列表，了解除 Thread / Lock 之外还提供哪些同步原语（如 RLock、Semaphore、Event、Condition）
- 对比 threading 与 multiprocessing 模块的接口设计，理解为何两者 API 风格相近

---

## 二、类

### threading.Thread

| 字段 | 内容 |
|------|------|
| 名称 | threading.Thread |
| 类型 | Python 标准库类 |
| 在本P中的作用 | 创建子线程对象；本P用 `threading.Thread(target=函数名)` 实例化并通过 `start()` 启动，演示两个线程并发访问共享变量的场景 |
| 来源 | 📌 字幕直接提到 |
| 所属模块 | threading |

**建议方向**
- 练习通过继承 Thread 类并重写 `run()` 方法的另一种线程创建方式，对比与 target 参数方式的适用场景
- 验证 `daemon` 参数的效果：将子线程设为守护线程后，主线程退出时子线程是否随之结束

### threading.Lock

| 字段 | 内容 |
|------|------|
| 名称 | threading.Lock |
| 类型 | Python 标准库类（互斥锁） |
| 在本P中的作用 | 本P的核心解决方案；通过 `lock = threading.Lock()` 创建全局锁对象，在修改共享变量的代码块前后分别调用 `acquire()` / `release()`，确保同一时刻只有一个线程进入临界区 |
| 来源 | 📌 字幕直接提到 |
| 所属模块 | threading |

**建议方向**
- 将 `lock.acquire() / lock.release()` 模式替换为 `with lock:` 上下文管理器写法，对比两者在异常安全性上的等价关系
- 对比 Lock 与 RLock（可重入锁）的区别，验证同一线程对 Lock 二次 acquire 时是否会死锁

---

## 三、方法 / 函数

### lock.acquire()

| 字段 | 内容 |
|------|------|
| 名称 | lock.acquire() |
| 类型 | Lock 实例方法 |
| 在本P中的作用 | 上锁；若锁已被其他线程持有则阻塞等待，直到锁被释放后再获取。本P演示了 `acquire(timeout=5)` 的超时参数，用于规避无限等待引发的死锁 |
| 来源 | 📌 字幕直接提到 |
| 所属类 | threading.Lock |

**建议方向**
- 实验 `timeout` 参数：设置较短超时后检查返回值（True/False），基于返回值编写获取锁失败时的降级处理逻辑
- 对比阻塞模式（默认）与非阻塞模式 `acquire(blocking=False)` 在业务场景中的适用差异

### lock.release()

| 字段 | 内容 |
|------|------|
| 名称 | lock.release() |
| 类型 | Lock 实例方法 |
| 在本P中的作用 | 解锁；释放当前线程持有的锁，使其他等待线程得以继续执行。本P强调必须与 acquire() 成对使用，并建议包裹在 `try...finally` 块中以防代码抛出异常时锁未被释放 |
| 来源 | 📌 字幕直接提到 |
| 所属类 | threading.Lock |

**建议方向**
- 构造一个故意在临界区内抛出异常的场景，分别对比使用 `try...finally` 与不使用时程序的行为差异（是否发生死锁）
- 改写为 `with lock:` 语法，验证上下文管理器是否在异常情况下自动释放锁

### Thread.join()

| 字段 | 内容 |
|------|------|
| 名称 | Thread.join() |
| 类型 | Thread 实例方法 |
| 在本P中的作用 | 线程等待；调用方会阻塞，直到目标线程执行结束后才继续。本P将其作为解决资源竞争的第一种方案（串行化），并分析其弊端：牺牲并发效率，退化为顺序执行 |
| 来源 | 📌 字幕直接提到 |
| 所属类 | threading.Thread |

**建议方向**
- 测量"双线程均使用 join 串行执行"与"双线程使用 Lock 并发执行"在相同任务量下的耗时差距，用数据直观感受并发优势
- 探索 `join(timeout=N)` 参数，验证超时后主线程是否继续执行以及子线程状态

### Thread.start()

| 字段 | 内容 |
|------|------|
| 名称 | Thread.start() |
| 类型 | Thread 实例方法 |
| 在本P中的作用 | 启动线程；使线程进入就绪状态并由操作系统调度执行。本P所有演示代码均以此方法触发并发场景 |
| 来源 | 📌 字幕直接提到（承接P52，本P延续使用） |
| 所属类 | threading.Thread |

**建议方向**
- 验证 `start()` 与直接调用 `run()` 的区别：后者不开新线程，仍在主线程中同步执行

## 四、语言机制 / 运行时概念

### GIL（全局解释器锁）

| 字段 | 内容 |
|------|------|
| 名称 | GIL（Global Interpreter Lock，全局解释器锁） |
| 类型 | CPython 运行时机制 |
| 在本P中的作用 | 本P结尾处提及，用于解释为何 Python 线程在 CPU 密集型任务上无法真正并行：GIL 限制同一时刻只有一个线程执行 Python 字节码，是线程应用边界的硬性约束 |
| 来源 | 📌 字幕直接提到 |
| 参考资料 | CPython 官方 Wiki：https://wiki.python.org/moin/GlobalInterpreterLock |

**建议方向**
- 设计一个 CPU 密集型任务（如大数分解），分别用单线程、多线程、多进程执行，用 `time` 模块测量耗时，直观验证 GIL 对多线程的限制
- 查阅 Python 3.13 关于 GIL 可选禁用（free-threaded 模式）的进展，了解未来版本的方向

### try...finally 语句

| 字段 | 内容 |
|------|------|
| 名称 | try...finally |
| 类型 | Python 语言内置语法结构 |
| 在本P中的作用 | 用于包裹 Lock 的临界区代码；确保即使临界区内发生异常，`finally` 块中的 `lock.release()` 也一定会执行，从而避免死锁 |
| 来源 | 📌 字幕直接提到 |

**建议方向**
- 将 `try...finally` + Lock 的手动模式重构为 `with` 语句（上下文管理器），理解两者的等价关系与 Python 的资源管理惯用法
