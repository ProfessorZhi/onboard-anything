# P53 知识框架 - 【进阶篇】线程核心与线程同步
## 本P位置

Python多线程进阶篇。承接P52线程创建与启动，进入线程间资源共享的问题域，为后续进程、线程池、异步编程奠定基础。

## 核心问题

**如何安全地在多线程间操作共享资源？**

## 知识骨架

```
共享资源
    ↓ 暴露问题
资源竞争（race condition）
    ↓ 根因分析
read-modify-write 非原子性
    ↓ 解决方案
├── join 线程等待（串行化，牺牲并发）
└── 互斥锁 Lock（并发保留，问题转移）
        ↓ 引入新问题
    死锁（Deadlock）
        ↓ 规避原则
    锁管理规范
```

## 关系与依赖

- **共享资源**是线程区别于进程的核心特征之一
- **资源竞争**是共享资源在并发场景下的必然产物
- **join**与**Lock**是解决资源竞争的两种路线，前者放弃并发换安全，后者保留并发但引入锁管理复杂度
- **死锁**是Lock方案的新增风险，属于锁管理范畴
- **GIL限制**是线程在CPU密集型场景的硬性约束，决定了应用边界

## 输入 / 输出 / 前置 / 支撑

| 类型 | 内容 |
|------|------|
| 前置知识 | P52 线程创建与启动（threading.Thread） |
| 输入 | 多个线程并发访问同一全局变量 |
| 核心矛盾 | 并发写入 → 数据不一致 |
| 输出 | 线程安全的共享资源操作方案 |
| 支撑工具 | threading.Lock（acquire/release） |
| 应用边界 | IO密集型任务适用；CPU密集型受GIL限制 |

## 易错点

1. **误判原子性**：`count += 1`不是一步操作，底层分读取→计算→写回三步
2. **忘记释放锁**：acquire后缺少release会导致死锁
3. **锁范围过大**：锁住整个函数等同于串行，失去多线程意义
4. **多锁顺序不一致**：多线程按不同顺序获取多把锁必然死锁
5. **读操作加锁**：只读不写不需要锁，加锁徒增开销
6. **混淆并发目标**：线程解决IO等待场景的并发，不解决计算密集场景的并行

| 对象 | 所属 | 作用 |
|------|------|------|
| threading | Python标准库 | 多线程基础设施 |
| threading.Thread | threading模块 | 线程创建与启动 |
| threading.Lock | threading模块 | 互斥锁实现 |
