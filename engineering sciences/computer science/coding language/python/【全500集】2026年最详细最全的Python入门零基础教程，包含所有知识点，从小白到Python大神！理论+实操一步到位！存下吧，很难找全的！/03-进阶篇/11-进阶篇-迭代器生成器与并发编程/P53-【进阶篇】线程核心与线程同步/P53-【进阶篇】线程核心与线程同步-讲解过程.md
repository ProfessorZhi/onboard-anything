# P53 讲解过程 - 【进阶篇】线程核心与线程同步
## 本P定位

本P是Python多线程进阶篇的核心内容，承接上一P的线程创建与启动，进一步讲解线程间资源共享带来的问题（资源竞争）以及两种线程同步解决方案（join 线程等待、互斥锁 Lock），是理解多线程编程的关键章节。

## 讲解过程

### 一、引出问题：线程共享资源

老师首先从"县城"类比入手，指出线程相对于进程的优势之一是"共享资源"。同一个进程内的线程会共享全局变量、文件等资源，通信非常方便。但随后话锋一转，指出这也会带来一个麻烦的问题——**资源竞争**：当多个线程同时修改同一个共享资源时，会导致数据错误。

### 二、演示线程共享全局变量

老师演示了一个简单代码：

```python
count = 0  # 全局变量，共享资源

def add_count():  # 任务一：加法计算
    global count
    count += 1
    time.sleep(1)  # 模拟耗时操作
    print(f"线程1 count加1后 = {count}")

def sub_count():  # 任务二：减法计算
    global count
    count -= 1
    time.sleep(1)
    print(f"线程2 count减1后 = {count}")
```

创建两个子线程 T1 执行 add_count，T2 执行 sub_count。运行结果显示：线程1后 count=1，线程2后 count=0。老师解读：说明两个线程都能访问到同一个 count 变量，这就是共享资源的体现。

### 三、放大问题：资源竞争现象

老师将问题规模放大，使用 1000 万次循环来制造明显的资源竞争：

```python
count = 0

def add_count():
    global count
    for _ in range(10000000):  # 1000万次
        count += 1
    print(f"执行后 count = {count}")
```

创建两个子线程同时执行。运行后发现：
- 小数值（如100万）时，每次结果都是 100万、200万，看起来正常
- 大数值（1000万）时，结果变得不规律，不再是每次都是 1000万、2000万

老师指出：这说明资源竞争确实存在。

### 四、分析资源竞争的根本原因

老师解释道：`count += 1` 看似是一步操作，实际在底层分为三步：

1. **读取**：从内存读取 count 的值（如 100）
2. **计算**：执行加一操作（变成 101）
3. **写回**：将结果写回 count 变量

当两个线程同时执行时会出现互相覆盖。例如：
- 线程1 读取 count=100，还没来得及加一
- 线程2 此时也读取到 count=100，加一后写回 101
- 线程1 再把 100+1 写回 101
- 两次加一只生效了一次

### 五、解决方案一：join 线程等待

老师介绍第一种方案：**join 线程等待**。核心思想是让线程排队执行，避免同时修改共享资源。

```python
t1 = threading.Thread(target=add_count)
t2 = threading.Thread(target=add_count)

t1.start()
t1.join()  # 等待T1执行完再继续

t2.start()
t2.join()  # 等待T2执行完再继续
```

运行结果稳定为 1000万、2000万。老师随后分析了 join 的**弊端**：失去了多任务的并发优势，总耗时变成两个线程时间之和（相当于单线程串行执行）。适用场景是"先写后读"或"先烧水后泡面"这种有明确先后顺序的情况。

### 六、解决方案二：互斥锁 Lock（推荐方案）

老师指出互斥锁是更常用、推荐使用的方案。它能保证同一时间只有一个线程操作共享资源，像公共厕所的门锁一样——有人用时锁上，用完解锁，其他人才能用。

**核心方法有两个：**
- `acquire()`：上锁，如果锁已被占用则等待
- `release()`：解锁，释放锁供其他线程使用

老师演示代码：

```python
import threading

lock = threading.Lock()  # 创建互斥锁对象

def add_count():
    global count
    for _ in range(10000000):
        lock.acquire()  # 上锁
        count += 1
        lock.release()  # 解锁
    print(f"执行后 count = {count}")
```

运行结果稳定为 1000万、2000万，且两个线程并发执行，总耗时比单线程快很多。

### 七、互斥锁的关键注意点

老师提出两个重要注意点：

**1. 上锁后必须解锁，建议使用 try finally 语句：**

```python
lock.acquire()
try:
    # 操作共享资源的核心代码
    count += 1
finally:
    lock.release()  # 确保即使代码报错也能释放锁，避免死锁
```

如果不这样处理，当核心代码报错时锁不会释放，导致死锁（里面的人一直关在里面，外面的人一直等待）。

**2. 减少锁的范围，只锁核心代码块：**

不要给整个函数上锁，只给修改共享资源的核心代码上锁。否则会变成串行执行，失去多线程的效率优势，和单线程没区别。

### 八、死锁问题及避免方法

老师指出互斥锁有一个致命问题：**死锁**——多个线程互相等待对方释放锁，导致程序卡住不动。

避免死锁的核心原则：
- 避免多把锁嵌套，尽量用一把锁解决问题
- 必须用多把锁时，保证所有线程获取锁的顺序一致
- 给锁设置超时时间：

```python
lock.acquire(timeout=5)  # 5秒后放弃获取锁，避免无限等待
```

- 减少锁的使用：优先用无状态设计，每个线程处理独立数据（不共享就不需要锁）
- 只读操作不需要锁（读取不会产生冲突）

### 九、本节总结

老师最后回顾了本节核心内容：

1. **线程核心**：实现多任务并发，轻量级共享进程资源，执行顺序无序
2. **线程创建启动**：导入 threading 模块，通过 Thread 类实例化创建，start() 方法启动
3. **关键问题**：线程共享全局变量带来资源竞争
4. **解决方案**：
   - join 线程等待：简单但牺牲并发效率，适合有明确先后顺序的场景
   - 互斥锁（推荐）：acquire/release 成对使用，确保同一时间只有一个线程操作共享资源
5. **避坑要点**：线程执行无序、锁必须成对使用（建议 try finally）、避免多锁嵌套、只读不需锁

老师最后提到线程适合 IO 密集型任务（如下载文件、处理网络请求），大部分时间在等待时可以让出 CPU 执行其他任务；如果是 CPU 密集型任务（大量计算），Python 的 GIL 会限制线程并发效率，此时可考虑使用进程（后续知识点）。

## 提到的外部资源与对象

- **threading 模块**：Python 标准库，用于多线程编程，提供 Thread 类创建线程，提供 Lock 类实现互斥锁
- **threading.Thread 类**：用于创建线程对象，构造函数接受 target（目标函数）、args（参数元组）、kwargs（参数字典）
- **threading.Lock 类**：互斥锁实现类，提供 acquire() 和 release() 两个核心方法

## 本P结论

1. **线程共享资源**：同一进程内的线程共享全局变量，通信方便但带来资源竞争风险
2. **资源竞争原因**：`count += 1` 底层分为读取→计算→写回三步，多线程同时执行会导致互相覆盖
3. **join 线程等待**：让线程排队执行，适合有明确先后顺序的场景，但会失去并发优势
4. **互斥锁 Lock（推荐方案）**：通过 acquire/release 确保同一时间只有一个线程操作共享资源，保留并发优势
5. **使用锁的注意事项**：必须成对使用（建议 try finally）、只锁核心代码块、避免死锁（避免多锁嵌套、设置超时时间）
6. **应用场景**：线程适合 IO 密集型任务，不适合 CPU 密集型任务（受 GIL 限制）
