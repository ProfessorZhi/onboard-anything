### 2. `class` 语句（继承语法）

| 字段 | 内容 |
|------|------|
| **名称** | `class` 语句 / 继承语法 `class 子类(父类):` |
| **类型** | Python 语言关键字 / 语法构件 |
| **在本P中的作用** | 核心语法载体；通过括号内指定父类名实现单继承与多继承；`class 子类(父类1, 父类2):` 实现多继承 |
| **来源** | 字幕直接提到 |
| **建议方向** | 手写"电子产品 → 电脑 → 笔记本电脑"三层继承链，验证属性与方法的传递性；尝试调换多继承父类顺序，观察同名方法的解析结果变化 |

### 3. `super()` 内置类

| 字段 | 内容 |
|------|------|
| **名称** | `super()` |
| **类型** | Python 内置类（built-in class） |
| **在本P中的作用** | 在方法重写的"扩展"场景下调用父类同名方法；老师特别强调其优于 `父类名.method(self)` 的原因：不依赖父类名，改父类名时子类代码无需修改，可维护性更好 |
| **来源** | 字幕直接提到 |
| **建议方向** | 在多层继承链（A→B→C）中分别在 B 和 C 的重写方法里调用 `super()`，打印 MRO（`类名.__mro__`）观察 `super()` 实际解析的对象；对比 `super().method()` 与 `父类名.method(self)` 在改父类名时的代码改动量 |

---

### 4. `__init__` 方法

| 字段 | 内容 |
|------|------|
| **名称** | `__init__` 方法 |
| **类型** | Python 特殊方法（dunder method） / 语言机制 |
| **在本P中的作用** | 本P通过"子类未定义 `__init__` 时自动继承父类 `__init__`"的报错演示，说明继承对初始化方法的传递规则；是理解"子类实例化必须传参"易错点的关键 |
| **来源** | 字幕直接提到 |
| **建议方向** | 分别测试：①子类不定义 `__init__` 直接实例化；②子类定义自己的 `__init__` 并在其中调用 `super().__init__()` 传参；③子类定义 `__init__` 但不调用 `super().__init__()`，对比三种情况下父类属性的可访问性 |

---

### 5. MRO（方法解析顺序）机制

| 字段 | 内容 |
|------|------|
| **名称** | MRO（Method Resolution Order，方法解析顺序） |
| **类型** | Python 语言内置机制 / 算法（C3 线性化） |
| **在本P中的作用** | 决定多继承时同名方法的调用优先级；本P结论：单继承为"子类→父类→更上层父类"，多继承为"子类→左侧父类→右侧父类→object"；老师以此解释同名冲突的根本原因 |
| **来源** | 知识框架直接提到（"方法解析顺序（MRO）"），讲解过程以继承顺序规则形式呈现 |
| **建议方向** | 对任意继承结构调用 `类名.__mro__` 或 `help(类名)` 查看完整 MRO 列表；构造菱形继承（D 同时继承 B 和 C，B、C 均继承 A），观察 MRO 如何避免重复调用 |

---

### 6. `object`（顶级父类）

| 字段 | 内容 |
|------|------|
| **名称** | `object` |
| **类型** | Python 内置类 / 隐式基类 |
| **在本P中的作用** | 所有类（无论是否显式继承）的最终父类；本P在继承链末端提及，说明单继承和多继承最终都指向 `object` |
| **来源** | 知识框架直接提到（"继承链"部分） |
| **建议方向** | 查看 `object.__dict__` 了解所有类默认拥有哪些方法（如 `__str__`、`__repr__`、`__eq__` 等）；尝试在子类中重写 `__str__`，体会 `object` 提供的默认实现与自定义实现的区别 |

---

### 7. 组合模式（Composition）

| 字段 | 内容 |
|------|------|
| **名称** | 组合（Composition）/ 单继承 + 组合模式 |
| **类型** | 设计模式 / 编程范式 |
| **在本P中的作用** | 作为多继承的替代方案被推荐；通过在 `__init__` 中创建其他类的实例（`self.hd = HealthDevice()`）实现功能复用，规避多继承的同名冲突与高耦合问题 |
| **来源** | 字幕直接提到 |
| **建议方向** | 将本P的 `SmartWatch` 多继承版本改写为单继承 + 组合版本，对比两种方式在新增父类方法时各需要修改哪些地方；查阅"组合优于继承"（Composition over Inheritance）原则的相关资料，了解其在大型项目中的实际应用场景 |
